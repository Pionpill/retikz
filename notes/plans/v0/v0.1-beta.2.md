## TODO-1 — Scene `ViewBox` → `Layout` 抽象

### 问题陈述

`packages/core/src/primitive/view-box.ts` 导出 type `ViewBox = { x, y, width, height }`，是 Scene primitive 的场景级布局边界——`viewBox` 这个词汇来自 SVG，让 type 系统看起来"为 SVG 准备"，与 alpha.5 ADR-01 把 PathPrim / GroupPrim 结构化、去 SVG 词汇的方向一致——type / field / file 层名字也应当中性化。

`ViewBox` 描述的是"场景的整体边界"，本质是几何布局；改名 `Layout` 表意中性、与 canvas / Skia / PDF 等 adapter 概念兼容。SVG adapter 内部把 `Layout` 翻译成 SVG `<svg viewBox="...">` 字符串（react adapter 内部已有 `formatViewBox` helper，保留），但 retikz 公开类型层不应叫 `ViewBox`。

### 建议方案

```ts
// 旧（beta.1）
import { type ViewBox, type Scene } from '@retikz/core';
type Scene = {
  primitives: Array<ScenePrimitive>;
  viewBox: ViewBox;
};

// 新（beta.2）
import { type Layout, type Scene } from '@retikz/core';
type Scene = {
  primitives: Array<ScenePrimitive>;
  layout: Layout;
};
```

文件 rename：`packages/core/src/primitive/view-box.ts` → `primitive/layout.ts`；`packages/core/src/compile/view-box.ts` → `compile/layout.ts`；函数 `computeViewBox` → `computeLayout`。`react/src/render/viewBox.ts` 保留——SVG adapter 内部 helper，`formatViewBox` 函数 / 文件名是产出 SVG viewBox 字符串的事实陈述，合理（其参数类型 `ViewBox` → `Layout`）。

### 影响范围

- `packages/core/src/primitive/view-box.ts` → `layout.ts`（rename + type 改名）
- `packages/core/src/primitive/scene.ts`（`viewBox: ViewBox` → `layout: Layout`）
- `packages/core/src/primitive/index.ts` barrel
- `packages/core/src/compile/view-box.ts` → `layout.ts`（rename + function 改名）
- `packages/core/src/compile/compile.ts`（调用方）
- `packages/core/src/compile/index.ts` barrel
- `packages/core/src/index.ts`（公开 export rename）
- `packages/react/src/kernel/TikZ.tsx`（`scene.viewBox` → `scene.layout`）
- `packages/react/src/render/viewBox.ts`（参数类型 import 名变化）
- 全部测试文件（`scene.viewBox` 断言 / `computeViewBox` 引用）
- mdx 文档 + AGENTS.md 注释同步
- `notes/` 内提到 `ViewBox` 类型的位置（不动 SVG attribute `viewBox` 字面）

> 排除：SVG icon 文件 / `<svg viewBox="...">` 字面 / `<marker viewBox="...">` 字面 / `arrowMarkers.tsx` 注释中提到 SVG marker viewBox——这些是 SVG attribute 写法，本就 SVG-specific。

### 风险

**BREAKING**：`ViewBox` / `Scene.viewBox` / `computeViewBox` 是公开 API surface 变化。beta 阶段不考虑兼容性，rc 才冻结——本版做合规。changelog 必写 BREAKING + 迁移路径（`ViewBox → Layout`、`scene.viewBox → scene.layout`、`computeViewBox → computeLayout`）。

### ADR 拆分

已拆为 [ADR-01：Scene `ViewBox` → `Layout` 抽象](../../adr/v0/v0.1-beta.2/01-scene-viewbox-to-layout.md)。虽然是命名抽象，但它改动 core 公开 type / Scene 字段 / helper export，属于 beta breaking，需要独立记录实现契约、测试象限与迁移路径。

## TODO-2 — `<TikZ>` → `<TikZ>` 组件改名

### 问题陈述

`packages/react/src/kernel/TikZ.tsx` 导出的 React 组件叫 `TikZ`，但 TikZ 原项目（LaTeX 包名）品牌大小写是 `TikZ`——尾部 Z 大写。retikz 作为"浏览器原生的 TikZ 子集"，组件名与原品牌一致更尊重历史 / 让 LLM 训练数据上的 TikZ 词汇直接命中 / 让用户从 LaTeX TikZ 切过来 import 时心智无落差。

当前 `<TikZ>` 写法对所有 retikz JSX 用户都是 top-level 组件——影响所有 demo / mdx / 用户代码。改名是 BREAKING。

### 建议方案

```tsx
// 旧（beta.1）
import { Tikz, Node, Path } from '@retikz/react';
<Tikz width={300} height={200}>
  <Node id="A" position={[0, 0]} />
</Tikz>

// 新（beta.2）
import { TikZ, Node, Path } from '@retikz/react';
<TikZ width={300} height={200}>
  <Node id="A" position={[0, 0]} />
</TikZ>
```

File rename: `packages/react/src/kernel/Tikz.tsx` -> `TikZ.tsx`; `TikzProps` -> `TikZProps`; `Tikz` component -> `TikZ`.

### 影响范围

- `packages/react/src/kernel/Tikz.tsx` -> `TikZ.tsx`
- `packages/react/src/index.ts` (`Tikz` / `TikzProps` export -> `TikZ` / `TikZProps`)
- `packages/react/src/kernel/_displayNames.ts`（组件 displayName 同步）
- `packages/react/tests/`（全部用 `<TikZ>` 的测试 → `<TikZ>`）
- `apps/docs/src/contents/**/*.demo.tsx` 全部 demo（grep ≈ 80+ 文件）
- `apps/docs/src/contents/**/*.mdx`（API 表 / 章节描述 `<TikZ>` 提及）
- `apps/docs/src/components/**/*.tsx` 内提到的 `<TikZ>` 描述
- `notes/` 文档 + AGENTS.md 同步

### 风险

**BREAKING**: `Tikz` component rename impacts every user import and top-level JSX tag. Changelog must include migration: `import { Tikz }` -> `import { TikZ }`, JSX `<Tikz>` -> `<TikZ>`.

Migration helper: boundary replace `'Tikz'` -> `'TikZ'`.

### TODO-3 — core endpoint arrow split 命名中性化

`packages/core/src/compile/path/split.ts` 当前的 `splitSubPathsForMarkers` 以 SVG marker 行为解释多 sub-path + arrow 的拆分策略。实际需求是 renderer-neutral 的“端点箭头只作用于整条 path 的首末端”，SVG marker 只是 React/SVG adapter 的一种实现方式。beta.2 后续调整：

- `splitSubPathsForMarkers` 改名为 `splitSubPathsForEndpointArrows`。
- 注释从 “SVG marker” 改为“endpoint arrows / path endpoints”。
- 测试标题中仅保留“箭头挂首末段”的业务语义，不再把 SVG marker 作为 core 语义。
- React renderer 仍可继续把 `arrowStart` / `arrowEnd` 翻译成 SVG marker。

Level: internal。目标是收紧 core/renderer 边界，不改 IR / Scene 结构与运行时行为。

### TODO-4 — arrow shrink 几何从 SVG viewBox/refX 语义中抽离

`packages/core/src/compile/path/shrink.ts` 当前用 SVG marker `viewBox` / `refX` 描述箭头 shrink 公式，并在注释中要求和 `react/render/arrowMarkers.tsx` 的内部几何一致。这会让 core 的箭头几何隐式依赖 React/SVG renderer。beta.2 后续调整：

- 抽出 renderer-neutral 的 arrow shape geometry 常量或 helper，例如 `arrowTipGeometry` / `arrowShapeGeometry`。
- core shrink 只依赖“尖端位置 / 线端接触位置 / 默认尺寸 / 空心描边宽度”等中性几何概念。
- React/SVG renderer 消费同一份几何定义后再映射到 SVG marker 的 `viewBox` / `refX`。
- 保留现有 shrink 数值测试，新增至少一条 core 与 renderer 箭头几何一致性的守门测试。

Level: internal。目标是消除 core 对 SVG marker 坐标命名的隐式依赖，不改变箭头视觉结果。

### TODO-5 — Path 线型命名对齐 TikZ 术语

核对 TikZ / PGF manual 后结论：

- TikZ 使用 `line cap` / `line join`，当前公开 `lineCap` / `lineJoin` 与 TikZ 术语一致，不需要调整。
- TikZ 使用 `dash pattern`，当前公开 `strokeDasharray` 来自 SVG/CSS。beta.2 若继续收敛公开命名，应评估 `strokeDasharray` → `dashPattern`，并同步 `<Path>` / `<Draw>` / docs / changelog。
- TikZ 填充规则使用 nonzero winding number rule / even odd rule，当前 `fillRule: 'nonzero' | 'evenodd'` 语义可接受，不需要调整。

Level: breaking（仅当执行 `strokeDasharray` → `dashPattern` 时）。如果本轮不做公开字段 rename，需至少把 docs 中“SVG 原生语法逃生口”的表述调整为“dash pattern string”，避免继续强化 SVG 绑定。
