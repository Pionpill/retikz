## TODO-1 — `Node.position` / `Coordinate.position` 补"任意 offset 相对定位"

### 问题陈述

当前 `Node.position` / `Coordinate.position` 的 union 三种形态，按"是否相对 + 表达力"列开：

| schema | 相对性 | 任意 `(dx, dy)` |
|---|---|---|
| `PositionSchema` (`[x, y]`) | 绝对 | — |
| `PolarPositionSchema` (`{ origin, angle, radius }`) | 相对 origin（极坐标） | 否——要手算 `atan2/hypot` |
| `AtPositionSchema` (`{ direction, of, distance? }`) | 相对 of | **否，仅 8 方向 + 单标量距离** |

缺口：**"相对节点 A 偏移 `(dx, dy)`"** 这种最直白的相对定位现在表达不了。

`IRTarget` 上有 `RelTargetSchema` (`{ rel: [dx, dy] }`)，但基准点是 path 前一步的终点，不是命名节点；且只挂 `step.to`，不入 position union。

**对应 TikZ**：`calc` library 的 `($(A) + (dx, dy)$)` 语法。

### 建议方案

新增第 4 种 position schema，命名暂定 `OffsetPositionSchema`：

```ts
// packages/core/src/ir/position/offset-position.ts
OffsetPositionSchema = {
  of: NodeId,              // 必填，被引用节点 id
  offset: [number, number] // 必填，[dx, dy] user units
}
```

加入 `Node.position` 与 `Coordinate.position` 的 union；解析在 `compile/position.ts` 的 `resolvePosition` 加分支。

**不扩 `AtPosition` 让 `distance` 兼接 `[dx,dy]`** 的理由：

- direction + 二维 offset 同字段角色冲突，语义混淆
- `.describe(...)` 给 LLM 看难写清"二选一"
- union 内字段不重叠优于单 schema 内字段重叠

### 改动清单

| 改动 | 文件 |
|---|---|
| 新建 schema + `IROffsetPosition` 派生类型 | `packages/core/src/ir/position/offset-position.ts`（新文件） |
| 加入 `Node.position` union | `packages/core/src/ir/node.ts` |
| 加入 `Coordinate.position` union | `packages/core/src/ir/coordinate.ts` |
| barrel 导出 | `packages/core/src/ir/position/index.ts` |
| 公开 API 导出 schema + 类型 | `packages/core/src/index.ts` |
| 解析分支 | `packages/core/src/compile/position.ts` 的 `resolvePosition` |
| 测试 | `packages/core/tests/compile/` 加 schema validation + 解析用例（绝对 / 嵌套 / 前向引用拒绝） |
| React adapter | builder 透传即可，无需特殊改造；确认 `_unbuilder` 反推也覆盖 |
| 文档（双语并行） | `apps/docs/doc/zh\|en/components/node.mdx`、`coordinate.mdx` API 表 + demo |

### 需写 ADR

- **命名**：`OffsetPosition` 字段名 `offset` vs `delta` vs `rel`——是否与 path `RelTarget.rel` 复用同名？两者基准点不同（命名节点 vs 前步终点），**建议不复用**避免歧义。
- **是否同步进 `IRTarget`**：让 step.to 也能写 `{ of: 'A', offset: [dx, dy] }`？这扩大了 ADR 覆盖范围；本待办默认 **只动 position union，不动 IRTarget**——后续按需另开 ADR。
- **前向引用规则**：与 polar `origin` / at `of` 一致，要求被引用节点在 IR 中**先**定义（顺序敏感），不接受前向引用。

## TODO-2 — Path 箭头重设计：删 `arrowShape`，新增 `arrowDetail` 对象配置

### 问题陈述

当前 `PathSchema` 上箭头相关只有两个扁平字段：

- `arrow: 'none' | '->' | '<-' | '<->'` — 方向 / 出现位置
- `arrowShape: ArrowShape` — 形状（normal / open / stealth / diamond / openDiamond / circle / openCircle 共 7 种）

视觉属性**全部被锁死等于 path 自身**（`render/arrowMarkers.tsx`）：

- 颜色：硬编码 `fill="context-stroke"` / `stroke="context-stroke"` —— 强制等于 path stroke
- 大小：硬编码 `markerWidth=6` / `markerHeight=6` + `markerUnits="strokeWidth"` —— 只能按 path strokeWidth 等比缩放
- 透明度：跟随 path `opacity` / `drawOpacity` —— 无独立 override
- 起末两端：**共用一个 `arrowShape`** —— 没法 start 一种形状、end 另一种

实际诉求"线浅色 / 箭头深色"、"半透明箭头"、"箭头放大让密集图可读"、"UML 起末异形"全做不到。

### 与 TikZ `arrows.meta` 能力对比

| TikZ 选项 | 用途 | retikz 现状 | 本次纳入 |
|---|---|---|---|
| shape (Latex / Stealth / ...) | 形状选择 | `arrowShape` 字段，7 种内置 | ✅ 搬进 `arrowDetail.shape` |
| `scale` | 等比缩放 | ❌ | ✅ |
| `length` | 尖长（user units） | ❌（固定 6） | ✅ |
| `width` | 尖宽（user units） | ❌（固定 6） | ✅ |
| `color` | 描边颜色 override | ❌（强制 context-stroke） | ✅ |
| `fill` | 填充色 override（实心 shape 主用） | ❌ | ✅ |
| `opacity` | 透明度 override | ❌ | ✅ |
| `lineWidth` | 空心 shape 的描边粗细 | ❌（硬编码 1.5） | ✅ |
| 起末异形 / 异色（`>=A, <=B`） | start / end 分别配置 | ❌（共享 `arrowShape`） | ✅ 通过 `arrowDetail.{ start, end }` 实现 |
| `harpoon` / `harpoon swap` | 单边倒钩 | ❌ | ⚠️ 不纳入本次（留 v0.2+） |
| `reversed` | 反向 | ❌ | ⚠️ 不纳入本次 |
| `inset` | stealth 倒钩内移 | ❌ | ⚠️ 不纳入本次（直接调 length/width 已可绕） |
| `slant` | 斜切 | ❌ | ❌ 长期不做（niche） |
| `bend` | 尖端弯曲 | ❌ | ❌ 长期不做 |
| `sep`（多 tip 间距） | 多重箭头 | ❌ | ❌ 不做 |

### 建议方案

**删除字段**：

- `arrowShape: ArrowShape` —— 搬进 `arrowDetail.shape` 子字段

**保留字段**：

- `arrow: 'none' | '->' | '<-' | '<->'` —— 含义不变（控制起末两端是否出箭头 / 哪一端出）

**新增字段 `arrowDetail?`**——一个对象，**默认配置**写在顶层、**起末分别 override**写在 `start` / `end` 子对象里：

```ts
type ArrowDetailSchema = {
  // 默认值（起末两端共享，没填 start/end 时生效）
  shape?: ArrowShape      // 缺省 'normal'
  scale?: number          // 等比缩放，默认 1；乘到 length/width 上
  length?: number         // 尖端长度（user units），缺省按 shape 内置（≈6×strokeWidth）
  width?: number          // 尖端宽度（user units），同上
  color?: string          // 描边颜色（空心 shape 主用 + 实心 shape stroke）；缺省继承 path stroke
  fill?: string           // 填充色（实心 shape 主用）；缺省 = color；空心 shape 强制 'none'
  opacity?: number        // 0..1；缺省继承 path opacity
  lineWidth?: number      // 空心 shape 描边粗细（user units），缺省 1.5

  // 起末分别 override（与上面默认逐字段 merge——缺省的字段继承顶层默认）
  start?: ArrowEndDetailSchema    // 仅当 arrow 包含起端（`<-` / `<->`）时生效
  end?: ArrowEndDetailSchema      // 仅当 arrow 包含末端（`->` / `<->`）时生效
}

// ArrowEndDetailSchema 字段集 = ArrowDetailSchema 减去 start/end 本身（不递归）

PathSchema = {
  ...
  arrow?: 'none' | '->' | '<-' | '<->'
  arrowDetail?: ArrowDetailSchema    // 新增
  // arrowShape: 删
}
```

**示例**：

```ts
// 默认 normal 实心箭头，path 继承所有视觉
{ arrow: '->' }

// 起末同形换 shape
{ arrow: '->', arrowDetail: { shape: 'stealth' } }

// 起末同形 + 缩放 + 红箭头
{ arrow: '<->', arrowDetail: { shape: 'stealth', scale: 1.5, color: 'red' } }

// 起末异形
{ arrow: '<->', arrowDetail: { start: { shape: 'normal' }, end: { shape: 'stealth' } } }

// 起末异色（共享 shape，颜色各异）
{ arrow: '<->', arrowDetail: { shape: 'stealth', start: { color: 'red' }, end: { color: 'blue' } } }

// 半透明 + 显式 length/width 覆盖 scale
{ arrow: '->', arrowDetail: { length: 12, width: 10, opacity: 0.5 } }
```

### 需写 ADR

1. **字段名**：`arrowDetail` vs `arrowStyle` vs `arrowOptions` vs `arrows`（复数，匹配 TikZ `arrows={...}`）vs `tipStyle`。建议 `arrowDetail`——直白、不撞 TikZ "style" 的复杂语义、给 LLM 看含义清楚。
2. **start/end merge 语义**：start/end 与顶层默认是 **逐字段 merge**（缺项继承顶层），还是 **完全替换**（一旦给了 start，所有字段都得重填）？**建议 merge**——与 `NodeLabel.font` 继承块级 font 的语义一致，DX 更友好。
3. **`scale` 与 `length`/`width` 的关系**：
   - 方案 A：scale 乘到 length/width 之后（用户给 `length=10, scale=1.5` → 实际 15）
   - 方案 B：互斥——给了 `length`/`width` 就忽略 `scale`
   - **建议 A**——scale 是"再缩放一次"的纯乘子，跟 SVG transform 的 scale 直觉一致
4. **`color` / `fill` 的语义在空心 vs 实心 shape 间的歧义**：
   - 实心 shape（normal / stealth / diamond / circle）：fill 主导，color 为 stroke 备用
   - 空心 shape（open / openDiamond / openCircle）：color 主导（描边），fill 强制 'none'
   - 文档要把这条规则白纸黑字写清——避免用户给 `fill='red'` 给空心 shape 然后困惑

## TODO-3 — `StepLabel.position` 扩充：7 keyword + 任意 t 数值

### 问题陈述

当前 `StepLabelSchema.position` (`packages/core/src/ir/path/step.ts:16`) 只接受 3 个 keyword：

```ts
position: z.enum(['midway', 'near-start', 'near-end']).optional()
```

对照 TikZ 的完整集（7 个 keyword + `pos=<float>` 任意数值），缺 4 个 keyword + 完全没数值形态。

### 建议方案

`position` 升级为 `union(enum, number)`，数值用 0..1 比例（与 TikZ `pos=<float>` 同步，**非**百分比 0..100）：

```ts
position: z
  .union([
    z.enum([
      'at-start',          // t=0.0
      'very-near-start',   // t=0.125
      'near-start',        // t=0.25
      'midway',            // t=0.5 (默认)
      'near-end',          // t=0.75
      'very-near-end',     // t=0.875
      'at-end',            // t=1.0
    ]),
    z.number().min(0).max(1),
  ])
  .optional()
```

Keyword 与 t 一一映射（无论 step kind 是哪种），是 t 的语义糖；**不同 step kind 对 t 的几何解释不同**——见下节。

### Numeric t 在不同 step kind 上的解释规则

t 是**沿整个 step 的归一化参数**（t=0 起点 / t=1 终点），但不同 kind 的参数化方式**与 TikZ 对齐**：

| Step kind | 参数化方式 | t=0.25 | t=0.5 | t=0.75 |
|---|---|---|---|---|
| `line` | 直线段，t 即归一化弧长 | 直线 1/4 处 | 直线中点 | 直线 3/4 处 |
| `step`（fold，N 段） | **N 段等 t 拼接**（详见下方"Fold N 段通用规则"）；当前 schema 只支持 N=2（`-\|` / `\|-`），未来扩展 N>2 沿用同一规则 | N=2: 第 1 段中点；N=4: 第 1 个拐角 | N=2: 拐角；N=4: 第 2 个拐角 | N=2: 第 2 段中点；N=4: 第 3 个拐角 |
| `curve` | quadratic Bezier 参数 t（**非弧长**） | Bezier t=0.25 位置 | Bezier t=0.5 位置（**通常不是视觉中点**） | Bezier t=0.75 位置 |
| `cubic` | cubic Bezier 参数 t | 同 curve | 同 curve | 同 curve |
| `bend` | 内部 lower 成 cubic Bezier，再走 Bezier t | 同 cubic | 同 cubic | 同 cubic |
| `arc` | 角度参数化：t 线性映射 `startAngle..endAngle`（圆弧情况下等价于弧长） | 起末角度的 1/4 处 | 起末角度的中点 | 起末角度的 3/4 处 |
| `circlePath` | 沿整圆 t∈[0, 1]：t=0 起点为角度 0（+x 方向），CCW 增长 | 圆周 90°（屏幕下方 +y） | 圆周 180°（-x） | 圆周 270°（屏幕上方 -y） |
| `ellipsePath` | 同 circlePath：t=0 在 angle 0（+x），CCW 增长 | 椭圆 90° 处 | 椭圆 180° 处 | 椭圆 270° 处 |

#### Fold N 段通用规则

Fold step 由 N 段直线 + (N-1) 个拐角组成。t 参数化规则**与段实际长度无关**：

- 每段在 t 轴上各占 `1/N` 长度
- 第 i 段（0-indexed）覆盖 `t ∈ [i/N, (i+1)/N]`，段内 t 仍归一化弧长
- 第 j 个拐角（连接段 j-1 和段 j，1-indexed）落在 `t = j/N`

**举例**：

| N | 拐角位置 | 段中点位置 | t=0.25 落点 | t=0.5 落点 | t=0.75 落点 |
|---|---|---|---|---|---|
| 2（当前 `-\|` / `\|-`） | 拐角 1 @ 0.5 | 段 1 中点 @ 0.25，段 2 中点 @ 0.75 | 第 1 段中点 | **拐角 1** | 第 2 段中点 |
| 3（未来） | 拐角 1 @ 1/3，拐角 2 @ 2/3 | 段中点 @ 1/6, 3/6, 5/6 | 段 1 内 t=0.75 处 | 段 2 内 t=0.5 处（视觉中段中点） | 段 3 内 t=0.25 处 |
| 4（未来） | 拐角 1 @ 0.25，拐角 2 @ 0.5，拐角 3 @ 0.75 | 段中点 @ 0.125, 0.375, 0.625, 0.875 | **拐角 1** | **拐角 2** | **拐角 3** |

**当前 schema 只支持 N=2**（`via: '-\|' \| '\|-'`）。规则按通用 N 段写一次到位，未来若加 N>2 的折角（比如 `'-\|-'` / `'\|-\|'` / `corners: Array<Position>` 等结构）**复用同一参数化**，不再改 t 解释规则。

**关键设计选择**（每条都对齐 TikZ）：

1. **Bezier 用 Bezier 参数 t，不用弧长** —— 与 TikZ 一致；按弧长定位需数值积分，昂贵且与 TikZ 训练数据 / 用户预期偏离
2. **Fold N 段每段等占 `1/N` t 区间**（不按弧长加权） —— 第 j 个拐角恒在 `t = j/N`，与实际段长比例无关。当前 N=2 → 拐角在 t=0.5；未来 N=3/4 多角 fold 沿用同一规则（拐角在 1/N, 2/N, ...）。与 TikZ 行为一致
3. **Arc / circlePath / ellipsePath 用角度参数化** —— 简单 + 跟 TikZ 习惯接近；圆弧情况下与弧长等价，椭圆下不等价但跨平台一致

**Move / Cycle step 无 label**——schema 本来就没 label 字段，不在本规则覆盖范围。

### 需写 ADR

1. **`circlePath` / `ellipsePath` 的 t=0 起点定义** —— 建议 angle 0（+x 方向）、CCW 增长，与 TikZ 一致；备选 "cursor 进入点"——但 cursor 在圆心、无"进入点"概念，不可取。
2. **Fold N 段是否按弧长加权 t** —— 建议**不加权**（每段各占 `1/N` t 区间，拐角恒在 `t = j/N`），与 TikZ 一致；加权方案 DX 直觉更好但跟 TikZ 偏离 + 增加 compile 复杂度。**规则按通用 N 段一次写定**，避免未来 schema 扩展时既改结构又改 t 语义。文档要明确写"段长比不影响拐角的 t 位置"。
3. **异常值处理** —— `pos=-0.1` / `pos=1.2`：zod `.min(0).max(1)` 严格拒绝（schema 校验失败），还是 compile 阶段 clamp？**建议严格拒绝**，与 schema 一贯严格性一致。

### 归类理由

功能扩展（不是 bug / 性能 / 重构 / 文档）——按 v0 roadmap 拆分原则归 alpha 窗口，不归 beta.1。

## TODO-6 — Scene `PathPrim` + `GroupPrim` 结构化重构（canvas adapter 铺路）

### 问题陈述

`packages/core/src/primitive/path.ts` 的 `PathPrim`：

```ts
type PathPrim = {
  type: 'path'
  d: string   // ← SVG path mini-language 字符串，如 "M 10 20 L 30 40 A r r 0 1 0 ..."
  ...
}
```

`d: string` 是 **SVG path mini-language 字符串**——`compile/path.ts` 整段工作是把结构化 IRStep 编译成这个字符串。这违反 DESIGN.md "Scene primitive 是矢量图形最大公约子集"原则——它把 SVG mini-language 锁死成了"公约"。

**后果**：

- Canvas adapter 要么自己 parse SVG path string、要么用浏览器 `new Path2D(d)`（Node / 非浏览器 canvas 没有 `Path2D`）
- Skia / PDF / native adapter 同样要 parse SVG mini-language——core 内部本来就有结构化数据，吐成字符串再让 adapter parse 回来是反向操作
- SVG-only 细节（A 命令的 endpoint-parameterization、large-arc-flag / sweep-flag）渗到 core compile 层（这是 TODO-5 那个 `arcSvgFlags` 之所以出现在 core 的根因）

**架构原则**：core 不假定任何具体渲染端。"如何把抽象路径表达成 X-语言"是 adapter 的工作。

### 建议方案

把 `PathPrim.d: string` 替换为结构化 `commands: Array<PathCommand>`：

```ts
// 新 PathCommand union（packages/core/src/primitive/path.ts）
type PathCommand =
  | { kind: 'move'; to: [number, number] }
  | { kind: 'line'; to: [number, number] }
  | { kind: 'quad'; control: [number, number]; to: [number, number] }
  | { kind: 'cubic'; control1: [number, number]; control2: [number, number]; to: [number, number] }
  | { kind: 'arc'; center: [number, number]; radius: number; startAngle: number; endAngle: number; counterClockwise?: boolean }
  | { kind: 'ellipseArc'; center: [number, number]; radiusX: number; radiusY: number; rotation?: number; startAngle: number; endAngle: number; counterClockwise?: boolean }
  | { kind: 'close' }

type PathPrim = {
  type: 'path'
  commands: Array<PathCommand>
  // stroke / fill / opacity 等字段不变
}
```

**关键设计点**：

1. **Arc / ellipseArc 用 center-parameterization**——`ctx.arc(cx, cy, r, start, end, ccw)` / `ctx.ellipse(...)` 是 canvas / Skia / PDF 的原生形态；SVG adapter 把它转 endpoint-parameterization + flags
2. **不保留 `d` 字段**——避免双源同步；adapter 各自从 commands 渲染
3. **`PathPrim` 外层属性不变**——stroke / fill / opacity / dasharray / arrow 引用等照旧；只改 path 内部表达
4. **每个 adapter 实现 commands → 原生调用**：
   - SVG (react): commands → d 字符串（react/src/render/path-d-builder.ts，把 compile/path.ts 现有字符串构造逻辑搬过来）
   - Canvas (未来 `@retikz/canvas`): commands → `ctx.moveTo / lineTo / quadraticCurveTo / bezierCurveTo / arc / ellipse / closePath`
   - Skia / PDF: 同理

### 并发处理：`GroupPrim.transform: string` 同样结构化

`packages/core/src/primitive/group.ts` 的 `GroupPrim.transform?: string` 是 SVG transform mini-language（如 `"translate(10 20) rotate(30)"`）——与 `PathPrim.d` 是**完全同质的 SVG 字符串 leakage**。canvas / Skia / PDF 没这语法，要 parse 字符串。

同步重构：

```ts
// 新（packages/core/src/primitive/group.ts）
type Transform =
  | { kind: 'translate'; x: number; y: number }
  | { kind: 'rotate'; degrees: number; cx?: number; cy?: number }
  | { kind: 'scale'; x: number; y?: number }

type GroupPrim = {
  type: 'group'
  transforms?: Array<Transform>   // 按数组顺序应用（与 SVG `transform` 多操作列表语义一致）
  children: Array<ScenePrimitive>
}
```

- 字段名：`transform?: string` → `transforms?: Array<Transform>`（复数，强调可链式）
- SVG adapter 把 `Array<Transform>` 拼成 transform 字符串（与 path-d-builder 同级的 transform-builder helper）；canvas adapter 调 `ctx.translate / ctx.rotate / ctx.scale`；Skia / PDF 同理
- rotate 的 `cx`/`cy` 旋转中心点保留（与 SVG `rotate(deg cx cy)` 三参形对齐），缺省时绕原点旋转
- `scale.y` 缺省时与 `scale.x` 相等（等比缩放）

**关键设计点**（同 PathPrim）：所有 SVG-flavored 字符串构造从 core 挪到 react adapter；core 只持有结构化数据。

### 为什么 alpha.5 做、不留 v0.2+

1. 与 TODO-2（箭头重设计）/ TODO-3（label position）协同——alpha.5 已经动 path 相关 schema；同步把 PathPrim 重构进去，path 整条链路一次破完
2. beta.1 起 schema 冻结，再做大重构要等到 v0.2 破坏性窗口——延期半年到一年
3. canvas adapter 是 v1 的关键里程碑，越早把 PathPrim 跨平台基础打好越早能起步
4. arcSvgFlags（TODO-5）天然作为本 TODO 的一部分自然解决——避免两次破坏性提交

### 需写 ADR

1. **`PathCommand` 字段命名 / 形态**：
   - arc 命令的 angle 单位：度 vs 弧度？**建议度**——与 IR `ArcStep` 一致，跨 adapter 兼容性最好
   - `counterClockwise` vs `sweep` 字段：用 canvas 习惯的 `counterClockwise: boolean` 还是其他？**建议 canvas 风格**——大多数非 SVG adapter 都用这个语义
   - `ellipseArc` 是否合并进 `arc`（rx === ry 时退化为 circle）？**建议分开**——简化每个命令的解释，与 canvas `ctx.arc` vs `ctx.ellipse` 一一对应
2. **是否保留 SVG 快速路径 helper 在 core**：core 是否给一个 `pathCommandsToSvgD(commands): string`？**建议不放 core**——adapter 各自做，避免 core 被 SVG 偏向；SVG 转换逻辑全部住 react adapter
3. **circlePath / ellipsePath 的 IR → command 映射**：
   - 现状 SVG 渲染拆 2 个半弧（避 A 命令 360° 退化）。canvas 没这个问题
   - **建议 emit 1 个 `ellipseArc` 全 sweep**——把"如何渲染整圆/椭圆"留给 adapter（SVG renderer 在需要时拆半弧，canvas 直接调 `ctx.ellipse` 整圈）
4. **是否同时 emit `boundsHint`**：path bbox 现在由 viewBox 计算阶段算出，能否提前在 commands 编译时算好放进 `PathPrim`？**建议先不做**——超出 TODO-6 范围，超额工作量
