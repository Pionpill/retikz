`@retikz/core` 是 retikz 的 **绘图底座**：不是又一个图表库，而是面向 AI 原生工作流设计的底层图形契约，负责把“用图描述思想”这件事沉淀成稳定、可校验、可渲染的结构。

它把成熟的节点 / 路径 / anchor 词汇搬进浏览器，用 React 组件作为表达形式；底层的 JSON IR 则作为统一的数据契约，连接人类写的 JSX、LLM 生成的 JSON、当前已支持的 SVG / Canvas renderer，以及后续 PDF / 原生平台方向。

## 两个核心概念

`@retikz/core` 最重要的心智模型只有两层：先描述**图元是什么**，再描述**图元之间的连接与几何关系**。

| 核心能力 | 由谁承担 | 解决什么问题 |
| --- | --- | --- |
| **图元模型** | `Node` | 描述一个可见对象本身：形状、文本、填充、边框、尺寸、位置和可被引用的 anchor。 |
| **图元关系** | `Path` / `Step` | 描述对象之间的连接与几何关系：起点、终点、中间动作、箭头和边标注如何沿路径组织。 |

因此你不需要先选择“流程图”“UML”或“图表”模板。先把重要对象放成命名 `Node`，再用 `Path` 把对象之间的连接、依赖、流向或注释关系画出来；更高层的 domain 包只是把这两个底层能力批量组织起来。

## 当前版本

当前 v0.3 的能力集中在 **绘图底座 + 多 renderer runtime**：`@retikz/core` 提供命名节点、坐标点、路径动作、箭头、边标注、Scope 分组 / 局部变换 / 样式继承，以及渐变 / 图案 / 图片填充、可注册形状和时间轴动画；`@retikz/render` 承接 SVG / Canvas 后端；`@retikz/react` 与 `@retikz/vanilla` 则分别提供 React DSL 与无框架 runtime / SSR。

这个阶段已经进入 beta 收口：重点是文档 demo、SVG / Canvas 对照测试、服务端导出、包体与公开 API 清理。图表语义、自动布局、codec 与更多框架 adapter 仍属于独立 domain 包或后续集成方向，不是 `@retikz/core` 要直接承担的能力。

## AI 友好优先

**AI 友好不是 retikz 的功能之一，是 retikz 存在的理由。**

retikz 选择 IR JSON 而非 JSX 作为 AI 接入的主接口，借力 OpenAI Structured Outputs / Anthropic Tool Use 的 schema 强约束输出能力——LLM 输出 100% 合法 IR、runtime 强校验。

| 能力 | 落地 |
| --- | --- |
| **类型 / schema 同源** | zod 单源 → TS 类型 + JSON Schema |
| **增量编辑** | JSON Patch（RFC 6902），patch 比重写整份 IR 便宜 |
| **text DSL（远期）** | 复用成熟绘图语法，让 LLM 更容易零样本生成 |

## IR 居中

<ComponentPreview name="ir-centric" hideCode />

所有输入方式（JSX / DSL / AI）都翻成同一份 IR；所有输出方式（renderer）都从 IR 出发。
**持久化、跨平台和 AI 接入并不是彼此独立的能力，而是 IR 居中带来的同一份红利。**
这样做的好处很直接：
- 你只需要围绕同一份 IR 维护保存、编辑和渲染逻辑。
- 不必为 JSX、AI 输出和不同 renderer 分别设计一套格式。
- 系统边界会更清楚，后续扩展也更省力。

## 分层抽象

retikz DSL 表面有三类构造，混淆它们是事后迁移噩梦：

| 层         | 例子                                | 是否进 IR                   | 落在哪          |
| ---------- | ----------------------------------- | --------------------------- | --------------- |
| **Kernel** | `<Layout>` `<Node>` `<Path>` `<Step>` | ✅ 直接对应 IR 节点         | `@retikz/core`  |
| **Sugar**  | `<Draw way={[...]} />`              | ❌ 编译期展开为 Kernel      | `@retikz/react` |
| **Tier 2** | `<Axis>` `<BarPlot>` `<Tree>`       | ✅ 高层节点 + lowering 下沉 | 独立 domain 包  |

关键约束：**Sugar 不引入新能力**——任何 Sugar 能表达的，Kernel 都能表达。这样 Sugar 可以放心地服务书写体验，而不把核心语义藏进 adapter 里。

## 通用图元，任意组合

`@retikz/core` 的 Kernel 保持克制——`<Layout>`、`<Scope>`、`<Node>`、`<Coordinate>`、`<Path>`、`<Step>`——但它们足够通用，能拼出大多数二维关系图。`<Draw>` / `<Text>` 这类 Sugar 只改善书写体验，最终仍然落回同一批 Kernel IR。

| 类别 | 由 Kernel 拼出 | 在哪 |
| --- | --- | --- |
| **流程图 / DAG** | Node + Step + Anchor + 外部布局算法 | 独立 domain 包 |
| **图表 / chart** | Step + Path + Node + data binding | `@retikz/plot` 方向 |
| **UML / 网络 / 思维导图** | 同 Kernel，换 layout 与样式约定 | 独立 domain 包 |
| **任意自定义** | 直接写 Kernel JSX | 你的应用 |

**你在图元层组合**——形状、连接方式、anchor、标签，都按几何 / 路径词汇直接表达：柱子上加一条曲线连到旁边的注释、节点之间画一条带 label 的曲线，都跟多画两个原图元一样自然。

domain 包是 Kernel 之上 + 数据绑定 / 布局算法 / 默认样式的薄薄一层；底层结构、持久化、AI 接口在所有 domain 间完全一致。当前先把通用图元打稳，避免在 core 里过早塞入图表 / 布局算法。

## 包关系

retikz 拆成四个发布包，从框架无关的底层到面向用户的入口逐层叠加：

<ComponentPreview name="package-graph" size="md" hideCode />

实线是主构建链——右侧的包构建在左侧的包之上；虚线表示 `@retikz/react` / `@retikz/vanilla` 也直接消费 `@retikz/core` 的 IR。四个包同版本发布。

| 包 | 职责 | 依赖 |
| --- | --- | --- |
| `@retikz/core` | IR、`compileToScene` 编译器、Scene 契约；零 React / 零 DOM | `zod` |
| `@retikz/render` | 渲染后端：`./svg`（Scene → SVG 描述 / 字符串）、`./canvas`（Scene → Canvas 2D） | core |
| `@retikz/react` | React adapter：JSX 构建 IR，`<Layout renderer>` 选 svg / canvas 后端 | core · render |
| `@retikz/vanilla` | 无框架 runtime / SSR：挂载到 DOM 或产 SVG 字符串，附命令式 builder | core · render |

`@retikz/render` 把原本在 `@retikz/react` 里的 SVG 渲染核心抽出来，和新增的 Canvas 后端合并成独立包；`@retikz/react` 与 `@retikz/vanilla` 都不再各自维护渲染逻辑，只组合 render 的后端。

## 跨框架，共享 IR

retikz 的架构是 **core + runtime / adapter**：React DSL、无框架 runtime 和后续其它框架共享同一份 IR、同一份 `compileToScene` 编译器、同一份 Scene 渲染契约；上层入口只负责 authoring 体验、生命周期和渲染后端选择。

| 角色 | 作用 | 当前状态 |
| --- | --- | --- |
| `@retikz/core` / `@retikz/render` | 框架无关的 IR + 编译器 + 渲染后端 | ✅ 当前主线 |
| `@retikz/react` | React adapter（peerDep: react） | ✅ 当前主线 |
| `@retikz/vanilla` | 无框架 runtime / SSR | ✅ 当前主线 |
| `@retikz/vue` / `svelte` / `native` … | 其它框架 / 渲染目标 adapter | ⏳ 社区或后续 |

**架构层面 retikz 不绑任何框架——IR 是 universal 的，DSL 因框架而异是合理的。** 你今天用 `@retikz/react` 画的图，其 IR 可以被未来任意 adapter 直接渲染，不需要重写。

<Comparison target="tikz" title="与 TikZ 的关系">

retikz 的命名 / 路径动作 / anchor 体系直接致敬 LaTeX 的 TikZ 包。

| 维度     | LaTeX TikZ      | retikz                         |
| -------- | --------------- | ------------------------------ |
| 运行环境 | LaTeX 编译器    | 浏览器 / Node                  |
| 语法     | `\node` `\draw` | `<Node>` `<Draw>`              |
| 输出     | PDF / SVG       | SVG / Canvas；PDF / Native 计划中 |
| 交互     | 无              | React / runtime 事件           |
| 持久化   | `.tex` 源码     | JSON IR                        |

IR ↔ TikZ 源码转换属于后续 codec 方向：目标是在浏览器里编辑后能进入 LaTeX 工作流，或把已有 `.tex` 中的 TikZ 图迁移到 retikz 的 IR / React 表达里。

不需要先懂 TikZ 才能使用 retikz。retikz 借用的是命名词汇（node / path / anchor / step），API 仍然是纯 React 组件。

## 适合谁

**用图描述思想的人**——技术写作者、研究者、文档作者、教师。

具体场景：

1. 论文 / 博客里嵌入可交互示意图
2. 流程图、概念图、网络结构图、UML、思维导图
3. 由 LLM 自动生成或编辑的图形配图
4. 在浏览器里需要精细几何控制力的任何场景

## 技术对比

| 方案 | 适合什么 | 优点 | 局限 |
| --- | --- | --- | --- |
| `@retikz/core` | 关系图、概念图、AI 配图 | Node / Path / anchor + JSON IR | 不内置图表模板 |
| 传统图表库 | 统计图、仪表盘、报表 | 图表类型和数据映射成熟 | 自由几何组合较弱 |
| 裸 SVG / `<svg>` | 单张精修图、静态资产 | 平台原生、最直接 | 关系、anchor、后续编辑要手写 |

长期规划里，`@retikz/core` 会作为底座，向上封装更多领域组件与上层包；core 只保留通用图元、IR 契约和编译能力。
