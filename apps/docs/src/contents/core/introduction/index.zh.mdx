retikz **不是又一个图表库**，而是把"用图描述思想"这件事重新设计为 AI 原生工作流的绘图基座。

它把成熟的节点 / 路径 / anchor 词汇搬进浏览器，用 React 组件作为表达形式；底层的 JSON IR 则作为统一的数据契约，连接人类写的 JSX、LLM 生成的 JSON，以及 SVG / Canvas / PDF / 原生平台 renderer。

当前 v0.1 的稳定能力集中在 **Node / Coordinate / Path / Step / Draw / Text**：命名节点、坐标点、路径动作、箭头、边标注、多行文本和 IR 渲染。后文提到的 chart / flow / codec / 更多框架 adapter 属于架构预留或后续 domain 包方向，不是当前 `@retikz/react` 已发布 API。

## AI 友好优先

**AI 友好不是 retikz 的功能之一，是 retikz 存在的理由。**

retikz 选择 IR JSON 而非 JSX 作为 AI 接入的主接口，借力 OpenAI Structured Outputs / Anthropic Tool Use 的 schema 强约束输出能力——LLM 输出 100% 合法 IR、runtime 强校验。

| 能力 | 落地 |
| --- | --- |
| **类型 / schema 同源** | zod 单源 → TS 类型 + JSON Schema |
| **增量编辑** | JSON Patch（RFC 6902），patch 比重写整份 IR 便宜 |
| **text DSL（远期）** | 复用成熟绘图语法，让 LLM 更容易零样本生成 |

## IR 居中

<ComponentPreview name="ir-centric" hideCode />

所有输入方式（JSX / DSL / AI）都翻成同一份 IR；所有输出方式（renderer）都从 IR 出发。
**持久化、跨平台和 AI 接入并不是彼此独立的能力，而是 IR 居中带来的同一份红利。**

## 分层抽象

retikz DSL 表面有三类构造，混淆它们是事后迁移噩梦：

| 层         | 例子                                | 是否进 IR                   | 落在哪          |
| ---------- | ----------------------------------- | --------------------------- | --------------- |
| **Kernel** | `<TikZ>` `<Node>` `<Path>` `<Step>` | ✅ 直接对应 IR 节点         | `@retikz/core`  |
| **Sugar**  | `<Draw way={[...]} />`              | ❌ 编译期展开为 Kernel      | `@retikz/react` |
| **Tier 2** | `<Axis>` `<BarPlot>` `<Tree>`       | ✅ 高层节点 + lowering 下沉 | 独立 domain 包  |

关键约束：**Sugar 不引入新能力**——任何 Sugar 能表达的，Kernel 都能表达。这样 Sugar 可以放心地服务书写体验，而不把核心语义藏进 adapter 里。

## 通用图元，任意组合

core 的 Kernel 先保持很小——`<TikZ>`、`<Node>`、`<Coordinate>`、`<Path>`、`<Step>`——但它们足够通用，能拼出大多数二维关系图。`<Draw>` / `<Text>` 这类 Sugar 只改善书写体验，最终仍然落回同一批 Kernel IR。

| 类别 | 由 Kernel 拼出 | 在哪 |
| --- | --- | --- |
| **流程图 / DAG** | Node + Step + Anchor + dagre/elkjs 布局 | `@retikz/flow`（v1.x） |
| **图表 / chart** | Step + Path + Node + data binding | `@retikz/plot`（v2.x） |
| **UML / 拓扑 / 思维导图** | 同 Kernel，换 layout 与样式约定 | 独立 domain 包 |
| **任意自定义** | 直接写 Kernel JSX | 你的应用 |

**你在图元层组合**——形状、连接方式、anchor、标签，都按几何 / 路径词汇直接表达：柱子上加一条曲线连到旁边的注释、节点之间画一条带 label 的曲线，都跟多画两个原图元一样自然。

domain 包是 Kernel 之上 + 数据绑定 / 布局算法 / 默认样式的薄薄一层；底层结构、持久化、AI 接口在所有 domain 间完全一致。当前先把通用图元打稳，避免在 core 里过早塞入图表 / 布局算法。

## 跨框架，当前先发 React

retikz 的架构是 **core + 各框架薄 adapter**：所有框架共享同一份 IR、同一份 `compileToScene` 编译器、同一份 Scene 渲染契约；每个 adapter 只负责把 Scene primitives 翻译成本框架的 element，几十行级别。

| 角色 | 作用 | 当前状态 |
| --- | --- | --- |
| `@retikz/core` | IR + Scene 编译器；零 React / 零 DOM | ✅ v0.1 官方 |
| `@retikz/react` | React adapter（peerDep: react） | ✅ v0.1 官方 |
| `@retikz/vue` / `svelte` / `native` … | 其它框架 / 渲染目标 adapter | ⏳ 社区或后续 |

**架构层面 retikz 不绑任何框架——IR 是 universal 的，DSL 因框架而异是合理的。** 你今天用 `@retikz/react` 画的图，其 IR 可以被未来任意 adapter 直接渲染，不需要重写。

<Comparison target="tikz" title="与 TikZ 的关系">

retikz 的命名 / 路径动作 / anchor 体系直接致敬 LaTeX 的 TikZ 包。

| 维度     | LaTeX TikZ      | retikz                      |
| -------- | --------------- | --------------------------- |
| 运行环境 | LaTeX 编译器    | 浏览器 / Node               |
| 语法     | `\node` `\draw` | `<Node>` `<Draw>`           |
| 输出     | PDF / SVG       | SVG / Canvas / PDF / Native |
| 交互     | 无              | React 事件、状态            |
| 持久化   | `.tex` 源码     | JSON IR                     |

远期会通过 `@retikz/codec` 做 IR ↔ TikZ 源码双向转换——浏览器画完直接贴进 LaTeX 论文，反向把 `.tex` 中的 TikZ 搬进浏览器（v3.x+）。

不需要先懂 TikZ 才能使用 retikz。retikz 借用的是命名词汇（node / path / anchor / step），API 仍然是纯 React 组件。

## 适合谁

**用图描述思想的人**——技术写作者、研究者、文档作者、教师。

具体场景：

1. 论文 / 博客里嵌入可交互示意图
2. 流程图、概念图、网络拓扑、UML、思维导图
3. 由 LLM 自动生成或编辑的图形配图
4. 在浏览器里需要精细几何控制力的任何场景

## 技术对比

| 方案 | 适合什么 | 优点 | 局限 |
| --- | --- | --- | --- |
| `@retikz/core` | 关系式示意图、流程图、概念图、AI 生成 / 编辑的图形 | 有命名节点、路径动作、anchor、JSON IR；适合描述"元素之间的关系"，也方便持久化和局部修改 | 不是内置图表模板库；复杂布局、数据绑定和图表语义会放到后续 domain 包 |
| 传统图表库 | 数据集驱动的统计图、仪表盘、业务报表 | 图表类型成熟，坐标轴、图例、tooltip、数据映射等能力开箱即用 | 更擅长"把数据映射成图表"，不擅长自由组合节点、路径、注释和任意几何关系 |
| 裸 SVG / `<svg>` | 手工精确控制单个图形或导出静态资产 | 平台原生、最直接、没有额外抽象 | 没有节点命名、路径语义和 anchor 系统；关系线、边界贴合、后续编辑都要自己维护坐标 |

长期规划里，`@retikz/core` 会作为底座，向上封装更多领域组件与上层包；core 只保留通用图元、IR 契约和编译能力。
