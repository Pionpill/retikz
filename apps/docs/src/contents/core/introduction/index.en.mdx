retikz **is not yet another chart library**. It's a drawing foundation that re-designs "diagrams as a way to describe ideas" around an AI-native workflow.

It brings a mature vocabulary of nodes, paths, and anchors into the browser, expressed as React components; the underlying JSON IR acts as the shared data contract between human-authored JSX, LLM-generated JSON, and renderers for SVG / Canvas / PDF / native targets.

## Current Version

The current v0.2 surface focuses on **general primitives plus extensible drawing semantics**: named nodes, reference points, path actions, arrows, edge labels, multi-line text, Scope grouping / local transforms / style inheritance, plus gradient / pattern / image fills and registries for shapes, arrows, patterns, and path generators. Charting, flow layout, codecs, and additional framework adapters remain domain-package or future integration directions, not core `@retikz/react` APIs.

## AI-First

**AI-friendliness is not one of retikz's features — it's the reason retikz exists.**

retikz picks IR JSON over JSX as the primary AI interface, leveraging OpenAI Structured Outputs and Anthropic Tool Use to enforce strict schema compliance — the LLM emits 100% valid IR, validated at runtime.

| Capability | How |
| --- | --- |
| **Single-source types & schema** | zod single source → TS types + JSON Schema |
| **Incremental editing** | JSON Patch (RFC 6902) — patch beats rewriting the IR |
| **Text DSL (long-term)** | Mature drawing syntax that gives LLMs stronger zero-shot priors |

## IR at the Center

<ComponentPreview name="ir-centric" hideCode />

Every input form (JSX / DSL / AI) compiles to the same IR; every output form (renderer) reads from it.
**Persistence, cross-platform support, and AI integration aren't separate features — they're the same dividend paid by an IR-centric model.**
The practical benefit is straightforward:
- You maintain saving, editing, and rendering around one IR contract.
- You do not need separate formats for JSX, AI output, or each renderer.
- System boundaries stay clearer, and future extensions get cheaper.

## Layered Abstraction

retikz's DSL surface has three kinds of constructs. Mixing them up is a migration nightmare:

| Layer | Examples | In IR? | Where |
| --- | --- | --- | --- |
| **Kernel** | `<Layout>` `<Node>` `<Path>` `<Step>` | ✅ Maps 1:1 to IR nodes | `@retikz/core` |
| **Sugar** | `<Draw way={[...]} />` | ❌ Expands into Kernel at compile time | `@retikz/react` |
| **Tier 2** | `<Axis>` `<BarPlot>` `<Tree>` | ✅ High-level node + lowering hook | Domain packages |

Hard rule: **Sugar introduces no new capabilities** — anything Sugar can express, Kernel can. That lets Sugar optimize authoring ergonomics without hiding core semantics inside an adapter.

## General Primitives, Arbitrary Composition

The Kernel stays deliberately restrained: `<Layout>`, `<Scope>`, `<Node>`, `<Coordinate>`, `<Path>`, and `<Step>`. It is still general enough to compose most 2D relationship diagrams. Sugar such as `<Draw>` / `<Text>` improves authoring ergonomics, but still lowers back to the same Kernel IR.

v0.2 pushes composition further: `<Scope>` moves, rotates, scales, or styles a group of primitives; Paint and registries make fills, shapes, arrows, and path generators extensible without baking charting or layout algorithms into core.

| Category | Composed from Kernel | Where |
| --- | --- | --- |
| **Flow charts / DAG** | Node + Step + Anchor + external layout algorithms | Domain packages |
| **Charts** | Step + Path + Node + data binding | `@retikz/plot` direction |
| **UML / topology / mind maps** | Same Kernel, different layout & styling | Domain packages |
| **Anything custom** | Write Kernel JSX directly | Your app |

**You compose at the primitive layer.** Shape, connection, anchor, label — all expressed directly in geometry / path vocabulary: drawing a curve from a bar to a side annotation, or a labeled curve between two nodes, is as natural as drawing two more primitives.

A domain package is a thin layer of data binding, layout algorithms, and default styling on top of the Kernel; the underlying structure, persistence, and AI surface stay identical across domains. The current release keeps core focused on primitives first, instead of baking charting or layout algorithms into the base package too early.

## Package Relationships

retikz ships as four packages, layered from the framework-agnostic base up to the user-facing entry points:

<ComponentPreview name="package-graph" size="md" hideCode />

Solid lines are the main build chain — packages on the right build on those on the left; dashed lines mark that `@retikz/react` / `@retikz/vanilla` also consume `@retikz/core`'s IR directly. All four ship at the same version.

| Package | Role | Depends on |
| --- | --- | --- |
| `@retikz/core` | IR, the `compileToScene` compiler, the Scene contract; zero React / zero DOM | `zod` |
| `@retikz/render` | Render backends: `./svg` (Scene → SVG descriptor / string), `./canvas` (Scene → Canvas 2D) | core |
| `@retikz/react` | React adapter: build IR with JSX, pick the svg / canvas backend via `<Layout renderer>` | core · render |
| `@retikz/vanilla` | Framework-free runtime / SSR: mount to the DOM or emit an SVG string, plus an imperative builder | core · render |

`@retikz/render` lifts the SVG render core out of `@retikz/react` and merges it with the new Canvas backend into a standalone package; neither `@retikz/react` nor `@retikz/vanilla` maintains its own rendering logic — they just compose render's backends.

## Framework-Agnostic, React Today

retikz follows a **core + thin per-framework adapter** architecture: every framework shares the same IR, the same `compileToScene` compiler, and the same Scene contract; an adapter only translates Scene primitives into that framework's element type — typically tens of lines of code.

| Role | Purpose | Status |
| --- | --- | --- |
| `@retikz/core` / `@retikz/render` | Framework-agnostic IR + compiler + render backends | ✅ Current line |
| `@retikz/react` | React adapter (peerDep: react) | ✅ Current line |
| `@retikz/vanilla` | Framework-free runtime / SSR | ✅ Current line |
| `@retikz/vue` / `svelte` / `native` … | Other framework / target adapters | ⏳ Community / later |

**Architecturally retikz isn't tied to any framework — the IR is universal; DSLs may legitimately differ between frameworks.** A diagram you draw today with `@retikz/react` produces an IR that any future adapter can render unchanged.

<Comparison target="tikz" title="Relation to TikZ">

retikz's naming, path actions, and anchor model are direct tributes to LaTeX's TikZ package.

| Aspect | LaTeX TikZ | retikz |
| --- | --- | --- |
| Runtime | LaTeX compiler | Browser / Node |
| Syntax | `\node` `\draw` | `<Node>` `<Draw>` |
| Output | PDF / SVG | SVG / Canvas / PDF / Native |
| Interactivity | None | React events & state |
| Persistence | `.tex` source | JSON IR |

IR ↔ TikZ source conversion belongs to a future codec direction: the goal is to edit in the browser and still fit into a LaTeX workflow, or migrate existing TikZ figures from `.tex` files into retikz IR / React.

You do not need to know TikZ to use retikz. retikz borrows naming vocabulary (node / path / anchor / step), while the API remains plain React components.

## Who It's For

**People who use diagrams to describe ideas** — technical writers, researchers, documentation authors, teachers.

Concrete scenarios:

1. Interactive diagrams embedded in papers and blog posts
2. Flow charts, concept maps, network topologies, UML, mind maps
3. LLM-generated or LLM-edited illustrations
4. Anywhere in the browser that needs precise geometric control

## Technical Comparison

| Option | Best fit | Strengths | Limits |
| --- | --- | --- | --- |
| `@retikz/core` | Relational diagrams, flow charts, concept maps, AI-generated / AI-edited figures | Named nodes, path actions, anchors, and JSON IR; good at describing relationships between elements and at persistence / local edits | Not a built-in chart-template library; complex layout, data binding, and chart semantics belong in future domain packages |
| Traditional chart libraries | Dataset-driven statistical charts, dashboards, business reports | Mature chart types, axes, legends, tooltips, and data mapping out of the box | Better at mapping data to charts than at freely composing nodes, paths, annotations, and arbitrary geometric relationships |
| Raw SVG / `<svg>` | Hand-tuned individual graphics or static exported assets | Native platform primitive, direct, no extra abstraction | No node naming, path semantics, or anchor system; relationship lines, boundary attachment, and later edits require manual coordinate management |

Long term, `@retikz/core` is the foundation for higher-level domain components and packages. Core itself stays focused on general primitives, the IR contract, and compilation.
