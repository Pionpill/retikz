## Introduction

This is the final figure from the TikZ tutorial — "Karl teaching his daughter trigonometry": a unit circle plus the sin / cos / tan geometry around a 30° angle.

The page breaks it into 7 accumulative steps. From a single naked circle each step adds one piece and one retikz capability — by the end you have the full figure and have walked through how [Path](/core/components/draw/path) / [Draw](/core/components/draw/overview) / [Coordinate](/core/components/node/coordinate) / [Node](/core/components/node/overview) combine, plus polar positions, accumulative paths, multi-Draw coordination, and multi-line / multi-color Node text.

## Steps

### Step 1 — Unit circle

Minimal start — one circle centered at the origin. The [Path](/core/components/draw/path) container with [`<Step kind="circlePath">`](/core/components/draw/step#circlepath): `circlePath` takes the previous step's anchor as the center, so we `move (0, 0)` first to position the cursor at the origin. `lineCap="round"` softens the line endpoints.

<ComponentPreview name="karl-circle-01-circle" />

### Step 2 — Axes + endpoint labels + named anchors

x and y axes use [Draw](/core/components/draw/overview) with `arrow="->"` to get arrow-tipped segments. The `x` / `y` labels at the axis ends are [Node](/core/components/node/overview)s (with `stroke="none"` to hide the default border) placed just past each axis. [Coordinate](/core/components/node/coordinate) names the two endpoints (`x-axis` / `y-axis`) so later steps could reference them by id string — this example doesn't end up using them, but we keep them for fidelity to TikZ's `coordinate(x axis)` idiom.

<ComponentPreview name="karl-circle-02-axes" />

### Step 3 — Grid + ticks

The background grid is a `.map()` over 5 horizontal + 5 vertical lines (IR has no `grid` operator yet); this `.map()` substitute for TikZ `\foreach` is common throughout retikz. Ticks are also array-driven: a short segment plus a `fill="white"` [Node](/core/components/node/overview) per tick — white bg covers the grid behind each label.

The `cm(x, y)` helper translates math (y-up) to retikz screen coordinates (y-down): the top-level `<TikZ>` has no picture-level scale or y-flip, so tutorial-style examples often carry this kind of helper inline.

<ComponentPreview name="karl-circle-03-ticks" />

### Step 4 — 30° wedge + α label

The wedge is the trickiest step. [`<Step kind="arc">`](/core/components/draw/step#arc) treats the previous step's anchor as the **center** (note: TikZ instead treats it as a point on the arc). So `move (0, 0)` → `arc start=0 end=−30 radius=30` automatically starts drawing from (radius, 0) → `line` back to the origin lets [Path](/core/components/draw/path)'s fill implicitly close into a pie slice.

The α label is a [Node](/core/components/node/overview) at a polar position (`{ angle, radius }`) — `polar(15, 0.22)` places it visually centered in the wedge.

<ComponentPreview name="karl-circle-04-wedge" />

### Step 5 — sin α red line + cos α blue line

`sin` drops from (30°, 1cm) vertically to the x-axis; `cos` goes from that foot back to the origin. Both use [Draw](/core/components/draw/overview), with a `{ label: { text, side } }` operator threaded into the `way` array to attach an edge label. `thickness="veryThick"` is a semantic stroke preset — saves having to pick a numeric width.

Note: the original TikZ uses `(30:1cm |- x axis)` to express "horizontal projection of (30°, 1cm) onto the x-axis" directly. retikz has no projection target yet — geometrically that equals (cos30°, 0), so we hand-compute the coordinate.

<ComponentPreview name="karl-circle-05-sin-cos" />

### Step 6 — tan α orange line + auxiliary ray

The tan line goes from (1, 0) up to (1, tan30°). The original uses TikZ's `name path` + `name intersections` to compute "intersection of x=1 vertical and 30° ray from origin"; retikz has no such constructs, so we hand-compute (1, tan30°) and name it `t` via [Coordinate](/core/components/node/coordinate). The auxiliary ray from the origin through `t` then just references the id.

The pattern here — multiple [Draw](/core/components/draw/overview)s coordinating through shared named anchors — is retikz's standard way to wire up a multi-segment figure.

<ComponentPreview name="karl-circle-06-tan" />

### Step 7 — Right-hand multi-color info box

Finally, a colored info card on the right. The [Node](/core/components/node/overview) `text` prop takes an array of lines — **each entry can be a string or `{ text, fill }`**, the latter switching color for that line. Empty strings act as visual separators.

`shape="rectangle"` + `roundedCorners` + `align="left"` packs the lines into a left-aligned rounded card; `stroke="none"` removes the default border. Note: in-line span coloring (multiple colors within a single line) is not supported yet — only whole-line color switches. See Limitations.

## Limitations

- 🚧 **`grid` operator**: planned. The example uses `.map()` to draw 5 horizontal + 5 vertical lines instead
- 🚧 **Picture scale + y-up axis**: planned for `<TikZ>` (`scale` + `yAxis="up"`) to let users feed math coordinates directly. The example uses local `cm()` / `polar()` helpers
- 🚧 **Projection target (`(A |- B)`)**: planned. The example hand-computes `cm(cos30, 0)`
- 🚧 **Named path + intersection target**: planned. The example hand-computes `cm(1, tan30)`
- 🚧 **In-line spans inside LineSpec**: planned (add a `spans` field). The example switches color only at line granularity
- ❌ **Inline LaTeX math typography** (`$\frac{1}{2}$` / `$\sin \alpha$`): not planned. The example uses plain characters (`1/2` / `sin α`)
