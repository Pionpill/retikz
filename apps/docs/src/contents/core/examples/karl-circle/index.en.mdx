## Steps

### Step 1 â?Unit circle

Minimal start â?one circle centered at the origin. Use [Circle](/core/components/shapes/circle-ellipse) here: `center + radius` keeps the first step short, and it still expands to the same underlying Path / Step shape, but the code reads like âthis is a circleâ?at a glance. `lineCap="round"` softens the line endpoints.

<ComponentPreview name="karl-circle-01-circle" size="lg" />

### Step 2 â?Axes + endpoint labels + named anchors

x and y axes use [Draw](/core/components/draw/overview) with `arrow="->"` to get arrow-tipped segments. The `x` / `y` labels at the axis ends are [Node](/core/components/node/overview)s (with `stroke="none"` to hide the default border) placed just past each axis. [Coordinate](/core/components/node/coordinate) names the two endpoints (`x-axis` / `y-axis`) so later steps could reference them by id string â?this example doesn't end up using them, but we keep them for fidelity to TikZ's `coordinate(x axis)` idiom.

<ComponentPreview name="karl-circle-02-axes" diffFrom="karl-circle-01-circle" size="lg" />

### Step 3 â?Grid + ticks

The background grid now uses [Grid](/core/components/shapes/grid) sugar, which expands to many lines for you. Ticks are also array-driven: a short segment plus a transparent [Node](/core/components/node/overview) per tick; the label sits **slightly offset** from its tick (x-axis labels nudged 10px left, y-axis labels 10px down) so they don't clash with the circle or axis lines, and the step reads like âgive me a gridâ?instead of âassemble 10 lines.â?
retikz coordinates are SVG pixels with y pointing downward â?so "1 unit up" is written as `[0, -100]` (this example uses 100px = unit-circle radius as the length basis). The top-level `<Layout>` has no picture-level scale or y-flip, so math coordinates from the TikZ tutorial need their y manually negated.

<ComponentPreview name="karl-circle-03-ticks" diffFrom="karl-circle-02-axes" size="lg" />

### Step 4 â?30ÂḞ wedge + Îḟ label

The wedge is the trickiest step. Use [Sector](/core/components/shapes/circle-ellipse) here: it starts from the center, draws the arc, and closes back to the center, so the intent reads more directly than a hand-written `arc` + `line` path, and it is easier for a first-time reader to remember as one clear API.

The Îḟ label is a [Node](/core/components/node/overview) at a polar position (`{ angle, radius }`) â?`polar(15, 0.22)` places it visually centered in the wedge.

### Step 5 â?sin Îḟ red line + cos Îḟ dodgerblue line

`sin` drops from (30ÂḞ, 100px) vertically to the x-axis; `cos` goes from that foot back to the origin. Both use [Draw](/core/components/draw/overview), with a `{ label: { text, side } }` operator threaded into the `way` array to attach an edge label. `thickness="veryThick"` is a semantic stroke preset â?saves having to pick a numeric width.

Note: the original TikZ uses `(30:1cm |- x axis)` to express "horizontal projection of (30ÂḞ, 1cm) onto the x-axis" directly. retikz has no projection target yet â?geometrically that equals (cos30ÂḞ, 0), so we hand-compute the coordinate.

### Step 6 â?tan Îḟ darkorange line + auxiliary ray

The tan line goes from (1, 0) up to (1, tan30ÂḞ). The original uses TikZ's `name path` + `name intersections` to compute "intersection of x=1 vertical and 30ÂḞ ray from origin"; retikz has no such constructs, so we hand-compute (1, tan30ÂḞ) and name it `t` via [Coordinate](/core/components/node/coordinate). The pattern here â?multiple [Draw](/core/components/draw/overview)s coordinating through shared named anchors â?is retikz's standard way to wire up a multi-segment figure.

### Step 7 â?Right-hand multi-color info box

Finally, a colored info card on the right. The [Node](/core/components/node/overview) `text` prop takes an array of lines â?**each entry can be a string or `{ text, fill }`**, the latter switching color for that line. Visual grouping comes from alternating "colored formula line + default-color caption line"; we don't insert empty `''` separators because they inflate `textHeight` (counted as a full lineHeight) while empty `tspan` `dy` advance is inconsistent across browsers, leaving a visible empty strip below the text.

`shape="rectangle"` + `roundedCorners` + `align="left"` packs the lines into a left-aligned rounded card; `stroke="#d4d4d4"` + `dashed` add a light-gray dashed border (no fill, visible in both light and dark themes). Note: in-line span coloring (multiple colors within a single line) is not supported yet â?only whole-line color switches. See Limitations.
