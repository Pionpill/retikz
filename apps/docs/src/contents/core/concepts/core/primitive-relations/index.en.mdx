The [primitive model](/core/concepts/core/primitive-model) covers the geometry of a *single* primitive: how big it is and which anchors it exposes. Primitive relations cover what happens *between two* primitives — when a path references another primitive, where its endpoint finally lands on the other's geometry.

retikz resolves such a connection in two modes:

- **Dynamic attach (auto)**: give only the target id, and the endpoint picks a border point by the relative direction of the two.
- **Locked position (explicit)**: name a specific anchor, and the endpoint is fixed, independent of relative direction.

## Dynamic attach vs locked position

Start with the difference between the two modes. The two groups below have identical geometry — only the reference syntax differs:

<ComponentPreview name="relation-dynamic-vs-locked" size="md" hideCode />

The left group uses plain ids (`'T'`), so endpoints land on the target's NW / SE border by source direction; the right group locks to `'T.north'`, so endpoints pin to the top-edge midpoint no matter where the source is. **Prefer auto by default** — it lets connections adapt to the layout; lock only when a line must leave a specific side, align with a layout, or replicate an existing diagram.

| Mode | Syntax | Endpoint behavior | Best for |
| --- | --- | --- | --- |
| Dynamic attach | `'T'` / `{ id: 'T' }` | Picks a border by relative direction | Flowcharts, DAGs, ordinary relation lines |
| Locked position | `'T.north'` / `{ id, anchor }` | Fixed, ignores direction | Fixed exit side, layout alignment, replicas |

## Auto border attachment

With a plain id reference (`<Step to="a">` or `<Draw way={['a', ...]}>`), retikz calls `boundaryPoint` at compile time: it casts a ray from the target center toward the **neighbor step's direction** and takes the intersection with the connection surface (boundary) as the endpoint. So the endpoint depends on the relative position of the two primitives — which is exactly what "relation" means.

Which surface the endpoint lands on is decided by the target's **boundary** (following shape by default) — for how boundary, shape, and bounding box divide their roles, see the [primitive model](/core/concepts/core/primitive-model). To keep lines off the border, use `margin` (below) rather than replacing the target with hand-computed coordinates.

## Locked connection: explicit anchor

To lock onto a specific position, a path target uses the **object form** `{ id, anchor?, offset? }` — this is the core IR contract (serialization / LLM generation / JSON patch all use objects):

| anchor form | Meaning |
| --- | --- |
| anchor omitted | `{ id: 'A' }`: dynamic attach (same as plain id) |
| named anchor | `{ id: 'A', anchor: 'north' }`: a standard compass anchor or Web alias |
| angle anchor | `{ id: 'A', anchor: 30 }`: visual border point at the given degree |
| edge proportional | `{ id: 'A', anchor: { side: 'north', t: 0.25 } }` |
| anchor + offset | `{ id: 'A', anchor: 'west', offset: [-4, 0] }`: resolve the point, then add a world-space `[dx, dy]` (not rotated by the node's `rotate`) |

The anchor naming rules and geometric meanings all live in the [primitive model · anchors](/core/concepts/core/primitive-model); here only one thing matters: named / angle / edge-proportional anchors are **fixed in place**, unaffected by neighbor step direction (only auto mode adjusts dynamically). ### String shorthand

In JSX / Draw way you may use a string shorthand; the React layer eager-parses it into the object form above (the core IR stays object-form):

| shorthand | equivalent object |
| --- | --- |
| `'A'` | `{ id: 'A' }` |
| `'A.north'` | `{ id: 'A', anchor: 'north' }` |
| `'A.top-left'` | `{ id: 'A', anchor: 'north-west' }` |
| `'A.30'` | `{ id: 'A', anchor: 30 }` |

> **dotted-id limitation**: the string splits on the **first `.`**, so an **id containing `.` cannot use the shorthand** — write the object form `{ id: 'a.b', anchor: 'north' }`. The `{ side, t }` edge proportional point is **object-form only**.

A path target can also be a Cartesian, polar, relative / offset / proportional position — see [positioning](/core/concepts/basic/position) for the full catalog. This page only covers "referencing another primitive".

## Group connection: Scope boundary

`<Scope id="cluster">` registers the visual bounding box of its subtree as a referenceable boundary, and outer paths reference it via `'cluster'` / `'cluster.north'` / `'cluster.30'` — both auto and locked modes apply, only the connection target changes from a single node to a whole group of primitives:

<ComponentPreview name="relation-scope" size="md" />

Above, the external `ext` uses plain `'cluster'` to auto-attach to the group boundary on its side, and `'cluster.north'` to lock onto the group's top-edge midpoint. This is useful for drawing outer connections to a whole group; namespace and bbox details live in [Scope](/core/components/layout/scope).

## `margin` gives connections breathing room

`<Node margin={n}>` leaves the visible border untouched but pushes the node's whole connection boundary outward by `n` user units — **auto endpoints and explicit border anchors (`'A.east'` / `'A.30'`) move out alike**, so lines don't touch the border directly (matching TikZ `outer sep`). In multi-node flowcharts `margin={4}` is usually enough. For where margin sits in the node box model (and how it divides from padding), see the [primitive model](/core/concepts/core/primitive-model).

<ComponentPreview name="anchors-margin" size="xs" />

Note: only border anchors (auto / compass / angle) move out with `margin`; `center`, shape-specific anchors, edge-proportional points `{side,t}`, and label attachment still hit the **visual shape** and ignore `margin`. To keep a connection on the visible edge at some point, use those or an explicit coordinate offset.
