retikz's IR children come in two tiers: **Tier 1** (`node` / `path` / `coordinate` / `scope` — built into core, **no namespace**) and **Tier 2 composites** (a domain package's high-level nodes, which **always carry a `namespace`**, e.g. `@retikz/plot`'s `plot` — a whole grammar-of-graphics spec). Discrimination is purely by **presence of `namespace`**: present → tier2, absent → tier1.

A Tier 2 node has its own **complete zod schema** (fields first-class, each described) just like `Node`. At the first compile step it is lowered to Tier 1 by a registered expand function, after which layout / anchor / zIndex / renderer all apply automatically. core ships no composite and understands no chart / domain semantics.

## Why composites

A composite is a **high-level IR node carrying domain semantics**. Compared with hand-writing a pile of Tier 1 `node` / `path`, its value is:

- **Self-describing, AI-friendly**: a composite expresses domain meaning through its zod schema — a flowchart node, a bar, a tree — rather than "20 rectangles + 19 lines". When an LLM authors or a human reads the IR, they face the **intent** `{ type: 'pipeline', stages: [...] }`, not the post-expansion geometry noise.
- **High-level abstraction, hiding the low level**: users and LLMs care only about domain fields (data, steps, ticks), not how coordinates / connectors / clipping are computed; the underlying `expand` implementation can be optimized or refactored anytime, and as long as the schema contract holds, caller IR stays untouched.
- **Small footprint, expanded at runtime**: a composite node stores only domain parameters (the pipeline below is just `{ stages, arrows }`, two short arrays), so it serializes tiny; the real geometry (boxes / connectors / arrows / labels / boundary clipping) is generated on the fly by `expand` at compile time. And what this diagram draws is the very expansion path a composite itself travels — `Tier 2 DSL → composite IR → core IR → graph` — where the `lower` step (composite IR → core IR) is exactly this page's subject:

A few more equally important advantages:

- **Serializable / persistable**: a composite is 100% JSON — it can be stored in a database, sent over the wire, generated by AI, version-controlled, diffed. And the high-level **intent stays in the IR** (unlike Sugar, which is flattened away at framework build time), so "swap this chart's data" edits one field on the composite node, not 100 rectangles.
- **Expand once, shared by every backend**: lowering runs in the core compile step, so svg / canvas / vanilla / SSR / animation / hit-test all reuse the same expansion — a domain package implements `expand` once, with no per-renderer rewrite.
- **Strongly typed + validated + defaulted**: the schema is the single source of truth and `expand` receives a strongly-typed node; invalid fields fail loud with a clear error before expansion, and omitted fields are auto-filled with defaults.

## Registering a composite

`defineComposite({ schema, expand })`: `schema` extends `CompositeBaseSchema` (which declares the required `namespace` / `type`) plus domain fields; `expand` receives the schema-validated, strongly-typed node and produces Tier 1. Register via `CompileOptions.composites` (in React, `<Layout composites={[...]}>`); the node enters via raw IR (`<Layout ir>`). The `barChart` below expands a `data` array into N bars plus a baseline: **the node count is driven by `data.length`, the bar heights scaled by `value/max`** — not 1:1 reversible and algorithmic, exactly the case for a composite rather than Sugar (see the table below):

<ComponentPreview name="composite-basic" size="sm" />

| Field | Type | Description |
| --- | --- | --- |
| `schema` | `ZodType` | Full node schema: `CompositeBaseSchema.extend({ namespace: z.literal(...), type: z.literal(...), ...fields })`; core extracts `namespace.type` from the two literals as the registry key |
| `expand` | `(node) => IRChild \| IRChild[]` | Lowers the node to Tier 1 (pure, props-driven; may produce many, may nest composites) |

> `namespace` / `type` are **not repeated at the definition top level** — core extracts them from the `z.literal`s in `schema`. The schema must be a ZodObject extending `CompositeBaseSchema` with both as `z.literal`, otherwise registration throws.

## Lowering

This is exactly the `lower` step (`composite IR → core IR`) from the pipeline figure above: `compileToScene` walks the IR first, and for a tier2 node (`namespace` present) looks up the `${namespace}.${type}` registry → `schema.parse(node)` (precise validation + strong typing + default filling) → `expand` → replace; expansion output may contain composites, recursing to all Tier 1 (fixpoint).

- **Unregistered** `namespace/type` → warns (`COMPOSITE_NOT_REGISTERED`) and **skips that node**, rendering the rest — the whole figure is not aborted (graceful degradation beats crashing the figure when a high-level node's package is missing).
- **Duplicate registration** of the same `namespace.type` → **throws** at registration time (`composites` is injected as an array with no natural dedup, so a key clash is a caller error).
- **Nesting depth** defaults to 32, adjustable via `CompileOptions.maxCompositeDepth`; cycles / over-depth throw (runaway guard).
- When the expansion output carries an `id`, other elements can anchor-reference it as usual (lowering happens before anchor resolution).

## vs. Sugar

| | Sugar (`<Draw>` etc.) | Tier 2 composite |
| --- | --- | --- |
| Enters IR | ❌ flattened to Kernel at the adapter | ✅ enters IR as a high-level node, persistable / AI-generatable |
| Expanded | At framework build time | At core compile time (shared by all renderers / runtimes) |
| For | Shorthand that round-trips 1:1 | High-level structures involving algorithms (auto-tick / layout / scale) |

Lowering is **always in core**, never in the framework layer — svg / canvas / vanilla / SSR / AI all get it for free, with no per-framework implementation.

## Embeddable Tier 2: writing Tier 2 components inside `<Layout>`

Above we covered how a Tier 2 *node* lowers to Tier 1. This section covers an adjacent problem: when you want to write several Tier 2 *React components* (such as a future `<Plot>`) inside one `<Layout>`, collected into a single figure, how does each one's high-level node and expand logic get into that figure's compile.

```tsx
<Layout width={800} height={600}>
  <Panel id="a" x={40} y={40} data={salesA} />
  <Panel id="b" x={420} y={60} data={salesB} />
  <Path from={{ id: 'a', anchor: 'east' }} to={{ id: 'b', anchor: 'west' }} />
</Layout>
```

The catch: `<Panel>` is an ordinary React component, and rendering it would run its own hooks. But `<Layout>` builds its IR by **statically walking the component tree and reading props** — it does not render child components. Calling `<Panel>` during that static walk would fire its `useId` / `useMemo` outside React's render pass, corrupting the host's hook order and crashing on re-render / language switch. So `<Layout>` must obtain the node and expand logic a component contributes **without rendering it — by statically reading its props**.

### The adapter: domain provides, core defines the interface

core defines only one TS interface, `EmbeddableTier2Adapter`, and knows no specific domain (plot / flow / table each implement their own adapter; core never imports them):

```ts
type EmbeddableTier2Adapter<TProps> = {
  displayName: string; // matches the component's displayName
  namespace: string;   // = composite namespace; core groups contributions by it
  contribute: (props: TProps) => {
    node: IRChild;                       // the composite node placed into the scene (adapter may self-wrap a translate <Scope> for placement)
    datasets: Record<string, unknown>;   // external data by reference key — **does not enter IR**
    makeComposites: (mergedDatasets: Record<string, unknown>) => Array<CompositeDefinition>;
  };
};
```

`contribute` must be a **pure function**: no hooks, no rendering. It statically turns element props into one contribution — a composite `node` (enters IR and lowers via the mechanism above), a `datasets` map (external data, kept out of IR), and a `makeComposites` (which, given the merged datasets, produces that domain's composite expand logic).

### A static marker makes it zero-config

The domain attaches two static properties to the component so that `<Layout><Panel/></Layout>` works with no config:

```ts
Panel.isTier2Embeddable = true;
Panel.embeddableAdapter = {
  /* ...EmbeddableTier2Adapter... */
};
```

`<Layout>`'s two static traversal chains (the IR builder and the hydration-handler collector) detect the `isTier2Embeddable` marker, call `adapter.contribute(props)` once, push `node` into the IR, and record `{ namespace, datasets, makeComposites }` — **all without rendering the component**. The component's render function runs only when it is used **standalone** (not inside `<Layout>`).

### Layout aggregates contributions

After collecting every contribution, `<Layout>` groups them by `namespace`:

| Step | Behavior |
| --- | --- |
| Merge datasets | `datasets` of the same namespace merge into one; the same `reference` appearing twice **must be the same object identity**, otherwise it fails loud |
| Call makeComposites | called once per namespace with the merged datasets, producing that domain's composites |
| Assemble | results across namespaces concat, then merge with the user's explicit `composites` prop, all fed to compile |

In other words, embeddable Tier 2 wires the lowering path above into `<Layout>`'s declarative surface: composites solve "how a Tier 2 *node* lowers", while the embeddable adapter solves "how a Tier 2 *component* delivers that node and its expand logic into a shared `<Layout>`". Data always stays out of IR and is injected at compile through composites — exactly like the `composites` prop, and like how `<Plot>` works standalone.

<ComponentPreview name="embeddable-flow" size="sm" hideCode />

In the figure, `<Panel/>` props are **read statically** (the component is not rendered) into `contribute`, which yields `node` (enters IR), `datasets` (dashed, kept out of IR), and `makeComposites`; `<Layout>` aggregates by namespace and feeds compile.

### Escape hatch: the `embeddables` prop

The static marker is the primary path. `<Layout embeddables={[adapter]}>` is an optional escape hatch — pass an explicit `Array<EmbeddableTier2Adapter>` that matches child components by `displayName` and **overrides** the static marker. Use it for test injection, explicit control, or domains that did not attach the static marker.

### Boundaries

- **core hard-codes no domain**: core provides only this generic mechanism; plot / flow / table each provide their own adapter, and core never imports them.
- **Data stays out of IR**: `datasets` is kept out of IR and injected at compile through composites — exactly like the `composites` prop.
- **No IR schema change**: this is a core react-layer mechanism; the composite node and expand pipeline already exist.
