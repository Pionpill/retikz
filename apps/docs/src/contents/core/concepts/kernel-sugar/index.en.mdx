title: Kernel and Sugar
description: The two layers of the retikz DSL: Kernel components enter IR directly, Sugar expands to equivalent Kernel in the adapter.
---

The retikz React DSL has Kernel and Sugar layers. The distinction tells you whether a form is part of the lower-level contract or just a shorter writing convenience.

## Definitions

| Layer | Examples | Enters IR directly | Notes |
| --- | --- | --- | --- |
| Kernel | `<TikZ>` / `<Node>` / `<Coordinate>` / `<Path>` / `<Step>` | Yes | One-to-one with IR nodes |
| Sugar | `<Draw way={...}>`, `DrawWay.Cycle`, relative-coordinate strings | No | Synchronously expands to Kernel in the adapter |

Sugar adds no new capability. Anything drawn with `<Draw>` can also be written manually with `<Path>` + `<Step>`.

## Choosing a Layer

| Scenario | Prefer |
| --- | --- |
| Regular connections, flowchart edges, arrowed paths | `<Draw>` |
| Per-segment control over move / line / cycle / curve | `<Path>` + `<Step>` |
| Tests or converter output closest to IR | Kernel |
| Everyday documentation examples | Prefer Sugar, explain Kernel only when needed |

## Constraints

Sugar components are called synchronously by the builder to collect JSX; they do not run in the React render call stack. Their expansion logic must not depend on hooks or async state.

A form should be Sugar only when its expanded Kernel output can be mapped back to the original intent one-to-one. Data layouts, automatic ticks, graph algorithms, and similar higher-level structures belong in later standalone packages, not in core.
