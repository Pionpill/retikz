Every `<Node>` has 9 named anchors. Paths can either let retikz auto-pick the border point (the common case), or lock onto a specific position with the object form `{ id, anchor?, offset? }` (in JSX you may use the `'A.north'` / `'A.30'` string shorthand). Since v0.2, a `<Scope>` with an `id` also registers a rectangular boundary, so the same anchor syntax can target a whole group.

## The 9 anchors

| Name | Position |
| --- | --- |
| `center` | Geometric center |
| `north` / `south` / `east` / `west` | The 4 cardinal border points |
| `north-east` / `north-west` / `south-east` / `south-west` | The 4 diagonal border points |

The four built-in [`shape`](/core/components/node) types share this naming (each picks the geometric equivalent):

| | rectangle | circle | ellipse | diamond |
| --- | --- | --- | --- | --- |
| `north` & 4 cardinals | Edge midpoints | Circle perimeter | Ellipse axis ends | The 4 vertices |
| `north-east` & 4 diagonals | The 4 corners | Perimeter at 45° | Ellipse parametric t=π/4 | Side midpoints |

When a node has `rotate`, all 9 anchors rotate with it.

Custom shapes define their anchor behavior through `ShapeDefinition`. Supporting these 9 geometric anchors is recommended so the shape can substitute for built-ins; if a registered shape does not support a requested anchor, compilation reports the corresponding error. See [shape registry](/core/reference/extending/shape-registry).

## Auto border attachment

The most common case: when a path references a node by plain id (`<Step to="a">` or `<Draw way={['a', ...]}>`), retikz calls `boundaryPoint` at compile time — casts a ray from the node center toward the neighbor step's direction and takes the intersection with the border.

The 8 Draws below are all `[outerNode, 'Node']` — endpoints land near N / S / E / W / NE / NW / SE / SW respectively:

<ComponentPreview name="anchors-auto-attach" size="lg" hideCode />

Rule of thumb: for flowcharts, DAG edges, and ordinary relationship lines, write plain node ids and let retikz choose the border point. Use explicit anchors / angles only when a line must leave a specific side, align with a layout, or replicate an existing diagram. To create breathing room around a border, use `margin` instead of replacing the target with hand-computed coordinates.

`<Scope id="cluster">` registers the visual bounding box of its subtree as a referenceable boundary. Outer paths may use `'cluster'` / `'cluster.north'` / `'cluster.30'`, which is useful when connecting to a whole group of nodes. Namespace and bbox details live in [Scope](/core/components/layout/scope).

## Explicit anchor: object form (the contract)

Don't want auto-selection, need a specific position? A path target uses the **object form** `{ id, anchor?, offset? }` — this is the core IR contract (serialization / LLM generation / JSON patch all use objects):

| anchor form | Meaning |
| --- | --- |
| anchor omitted | `{ id: 'A' }`: auto-attach to border (same as plain id, **most common**) |
| named anchor | `{ id: 'A', anchor: 'north' }`: one of the 9 named anchors |
| angle anchor | `{ id: 'A', anchor: 30 }`: visual border point at the given degree |
| edge proportional | `{ id: 'A', anchor: { side: 'north', t: 0.25 } }`: see "Edge proportional points" below |
| anchor + offset | `{ id: 'A', anchor: 'west', offset: [-4, 0] }`: resolve the anchor / edge point, then add a world-space `[dx, dy]` (not rotated by the node's `rotate`) |

Named / angle / edge-proportional anchors are **fixed in place** — they don't move with neighbor step direction (only auto mode does). The 8 Draws below use 8 different named anchors as endpoints:

### String shorthand (React DSL convenience)

In JSX / Draw way you may use a string shorthand; the React layer eager-parses it into the object form above (the core IR stays object-form):

| shorthand | equivalent object |
| --- | --- |
| `'A'` | `{ id: 'A' }` |
| `'A.north'` | `{ id: 'A', anchor: 'north' }` |
| `'A.30'` | `{ id: 'A', anchor: 30 }` |

> **dotted-id limitation**: the string shorthand splits on the **first `.`**, so an **id containing `.` cannot use the shorthand** — write the object form `{ id: 'a.b', anchor: 'north' }`. The `{ side, t }` edge proportional point is **object-form only**.
>
> **Supported anchor names**: the 9 geometric anchors above. Text-baseline anchors such as `'text'` / `'base'` / `'mid'` are not part of the public API yet.

## Edge proportional points `{ side, t }`

`{ side, t }` takes the point at proportion `t ∈ [0, 1]` along an edge of the shape's **real boundary** — direction north/south west→east, east/west north→south: rectangle along the straight edge, circle / ellipse along that side's visible perimeter arc, diamond along the visible slanted edges. `{ side: 'west', t: 1/3 }` is the left edge's one-third point; `{ side: 'north', t: 0.5 }` is the top-edge midpoint (equals the named `'north'`).

## `margin` gives lines breathing room

With `<Node margin={n}>`: the border stays put, but path endpoints in **auto mode** back off `n` user units. In multi-node flow charts, `margin={4}` keeps lines from touching the border directly.

<ComponentPreview name="anchors-margin" size="sm" />

Note: explicit anchors / angles (`'A.east'` / `'A.30'`) hit the **visual border** — `margin` does not apply. If you need the endpoint pushed further out, use extra padding or explicit coordinate offset.
