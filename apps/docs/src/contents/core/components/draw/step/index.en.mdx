---
title: Step
description: Path's child node — one step = one action (move / line / step fold / cycle close / curve / cubic / bend / arc / circlePath / ellipsePath); `to` accepts node id / Cartesian / polar / relative offsets (rel / relAccumulate); only valid as a direct child of `<Path>`.
---

`<Step>` describes a single path action: `move` lifts the pen, `line` draws a straight line, `step` folds through a single right-angle corner, `cycle` closes the path, `curve` / `cubic` / `bend` draw curves of various shapes, and `arc` / `circlePath` / `ellipsePath` use the cursor as the center to draw an arc / full circle / full ellipse. It is **only** valid as a direct child of `<Path>`; placed elsewhere it is silently ignored.

## Usage

```tsx
import { Path, Step } from '@retikz/react';
```

```tsx
<Path>
  <Step kind="move" to="a" />
  <Step kind="line" to="b" />
</Path>
```

### Folded (step)

`kind="step"` inserts a single right-angle corner between cursor and target — the TikZ `(A) -| (B)` / `(A) |- (B)` form. A flow-chart staple:

### Closing (cycle)

`kind="cycle"` closes the current sub-path back to the most recent `move` target — the TikZ `cycle` keyword / SVG path `Z`. `cycle` carries **no** `to` field (the destination is implicit). Saves writing one explicit endpoint vs. drawing a `line` back to the start:

<ComponentPreview name="step-cycle" />

### Quadratic curve

`kind="curve"` draws a quadratic Bezier with one control point — TikZ's `.. controls (B) ..` / SVG path `Q`. The control point governs the bend direction and amplitude (**alpha.3 supports Cartesian `[x, y]` control points only**):

<ComponentPreview name="step-curve" />

### Cubic curve

`kind="cubic"` draws a cubic Bezier with two control points — TikZ's `.. controls (B) and (C) ..` / SVG path `C`. `control1` shapes the start tangent; `control2` shapes the end tangent. Common for S-shaped curves and complex connectors:

<ComponentPreview name="step-cubic" />

### Bend (arc shorthand)

`kind="bend"` is shorthand for TikZ's `to[bend left=N]` / `to[bend right=N]` — give a direction and angle, control points are computed at compile time and lowered to a cubic. `bendDirection` is the visual left/right relative to the from→to path; `bendAngle` defaults to 30:

<ComponentPreview name="step-bend" />

### Relative coordinates (rel / relAccumulate)

`to` also accepts relative-offset objects: `{ rel: [dx, dy] }` and `{ relAccumulate: [dx, dy] }` both resolve against the **previous step's endpoint** (`prevEnd`, falling back to `[0, 0]` when there is none). The only difference is **whether prevEnd advances**:

- `{ rel }` — equivalent to TikZ `(+x, +y)`: chained offsets all anchor on the **same** point; prevEnd does **not** update
- `{ relAccumulate }` — equivalent to TikZ `(++x, ++y)`: each segment accumulates and prevEnd moves with it

You can also write the sugar strings `'+dx,dy'` / `'++dx,dy'` (e.g. `<Step to="+1,0" />`); the React adapter rewrites them in place to the matching IR objects. In the demo below, the two paths have identical step sequences and only the target form differs — the solid line (rel) traces an L, while the dashed line (relAccumulate) climbs a staircase:

<ComponentPreview name="step-rel" />

### Arc

`kind="arc"` uses the **cursor as center** and draws a circular arc by start / end angle and radius — matches TikZ's `arc[start angle=…, end angle=…, radius=…]`. Angles follow the retikz polar convention: 0° points to +x, 90° is visually downward. The pen ends at the arc's endpoint:

<ComponentPreview name="step-arc" />

### Circle

`kind="circlePath"` uses the **cursor as center** and draws a fully closed circle by `radius` — the pen returns to the center after closing. Useful for endpoint dots, circular node borders, etc.:

<ComponentPreview name="step-circle" />

### Ellipse

`kind="ellipsePath"` uses the **cursor as center** and draws a fully closed ellipse by `radiusX` / `radiusY` — the pen returns to the center. When `radiusX === radiusY` it degenerates into a circle:

<ComponentPreview name="step-ellipse" />

### Styling

`<Step>` itself has no stroke-related visual props — styling lives on the parent `<Path>`, and all child Steps share the same `stroke` / `strokeWidth` / `strokeDasharray`:

## API Reference

| Prop | Type | Default | Description |
| --- | --- | --- | --- |
| `kind` | `'move' \| 'line' \| 'step' \| 'cycle' \| 'curve' \| 'cubic' \| 'bend' \| 'arc' \| 'circlePath' \| 'ellipsePath'` | `'line'` | See the "Ten kinds" table below |
| `via` | `'-\|' \| '\|-'` | — | Required for `kind="step"`: `-\|` horizontal-then-vertical; `\|-` vertical-then-horizontal |
| `to` | `IRTarget` | — | Not used for `kind="cycle"` / `'arc'` / `'circlePath'` / `'ellipsePath'`; required for the rest. Target: node id, Cartesian `[x, y]`, polar object, relative-offset object `{ rel }` / `{ relAccumulate }`, or sugar string `'+dx,dy'` / `'++dx,dy'` |
| `control` | `[number, number]` | — | Required for `kind="curve"`: single control point of the quadratic Bezier |
| `control1` | `[number, number]` | — | Required for `kind="cubic"`: first control point (shapes start tangent) |
| `control2` | `[number, number]` | — | Required for `kind="cubic"`: second control point (shapes end tangent) |
| `bendDirection` | `'left' \| 'right'` | — | Required for `kind="bend"`: visual left / right relative to from→to direction |
| `bendAngle` | `number` | `30` | Optional for `kind="bend"`: bend angle in degrees |
| `startAngle` | `number` | — | Required for `kind="arc"`: start angle in degrees (0° → +x, 90° → visually down) |
| `endAngle` | `number` | — | Required for `kind="arc"`: end angle in degrees |
| `radius` | `number` | — | Required for `kind="arc"` / `'circlePath'`: radius of the circle / arc |
| `radiusX` | `number` | — | Required for `kind="ellipsePath"`: x-axis radius |
| `radiusY` | `number` | — | Required for `kind="ellipsePath"`: y-axis radius |

## Ten kinds

| kind | Behavior | SVG / TikZ analogue |
| --- | --- | --- |
| `'move'` | Move cursor, **no draw** | path "M" |
| `'line'` | Straight line from cursor to target | path "L" / TikZ `--` |
| `'step'` | Right-angle fold through one corner | TikZ `-\|` / `\|-` |
| `'cycle'` | Close back to most recent `move` (no `to`) | path "Z" / TikZ `cycle` |
| `'curve'` | Quadratic Bezier (one control point) | path "Q" / TikZ `.. controls (B) ..` |
| `'cubic'` | Cubic Bezier (two control points) | path "C" / TikZ `.. controls (B) and (C) ..` |
| `'bend'` | Arc shorthand (direction + angle), lowered to cubic | TikZ `to[bend left=N]` |
| `'arc'` | Cursor as center; arc by start / end angle + radius | path "A" / TikZ `arc[start angle…]` |
| `'circlePath'` | Cursor as center; full circle, pen returns to center | path "A"×2 / TikZ `circle[radius]` |
| `'ellipsePath'` | Cursor as center; full ellipse, pen returns to center | path "A"×2 / TikZ `ellipse[…]` |

The first step of a `<Path>` must be `move` — if you write `'line'` / `'step'` / `'cycle'`, the compiler coerces it to `move` (cycle has no `to`, so it falls back to origin `[0, 0]`).

## Two via values

Only meaningful when `kind="step"`. Given cursor `(px, py)` and target `(tx, ty)`:

| via | Corner | Path |
| --- | --- | --- |
| `'-\|'` | `(tx, py)` | Horizontal to `tx`, then vertical to `ty` |
| `'\|-'` | `(px, ty)` | Vertical to `ty`, then horizontal to `tx` |

## Five forms of `to`

`to` matches the first five entries of [Draw's way item](/core/components/draw/overview#thirteen-forms-of-way-item):

| Form | Example | Behavior |
| --- | --- | --- |
| Node id | `'a'` | Auto-attaches to Node border (see [Anchors](/core/concepts/anchors)) |
| Cartesian | `[60, 0]` | Literal `[x, y]` |
| Polar | `{ origin: 'a', angle: 30, radius: 50 }` | Polar (see [Positioning](/core/concepts/positioning)) |
| Relative (rel) | `{ rel: [10, 0] }` or `'+10,0'` | Previous step's endpoint + `[dx, dy]`; does **not** update prevEnd (TikZ `(+x, +y)`) |
| Relative (relAccumulate) | `{ relAccumulate: [10, 0] }` or `'++10,0'` | Previous step's endpoint + `[dx, dy]`; **updates** prevEnd (TikZ `(++x, ++y)`) |
