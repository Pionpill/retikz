---
title: Step
description: Path's child node ‚Äî one step = one action (move / line / step fold / cycle close / curve / cubic / bend / arc / circlePath / ellipsePath / rectangle); `to` accepts node id / Cartesian / polar / offset / proportional placement / relative offsets; only valid as a direct child of `<Path>`.
---

`<Step>` describes a single path action: `move` lifts the pen, `line` draws a straight line, `step` folds through a single right-angle corner, `cycle` closes the path, `curve` / `cubic` / `bend` draw curves of various shapes, `arc` / `circlePath` / `ellipsePath` use the cursor as the center to draw an arc / full circle / full ellipse, and `rectangle` draws an axis-aligned rectangle from two opposite corners. It is **only** valid as a direct child of `<Path>`; placed elsewhere it is silently ignored.

Use this table to pick a `kind` first:

| kind | Needs `to` | Draws | Advances cursor | Supports label |
| --- | --- | --- | --- | --- |
| `move` | Yes | No | Yes | No |
| `line` / `step` / `curve` / `cubic` / `bend` | Yes | Yes | Yes | Yes |
| `cycle` | No | Yes | Back to subpath start | No |
| `arc` | No | Yes | To arc endpoint | Yes |
| `circlePath` / `ellipsePath` | No | Yes | Back to circle / ellipse center | Yes |
| `rectangle` | Yes | Yes | To opposite corner | No |

## Usage

```tsx
import { Path, Step } from '@retikz/react';
```

```tsx
<Path>
  <Step kind="move" to="a" />
  <Step kind="line" to="b" />
</Path>
```

Use `<Step>` when you need exact control over Kernel path actions: multiple `move`s, mixed straight / folded / curved / arc segments, or debugging the structure produced by `<Draw>`. For everyday connections, prefer [Draw](/core/components/draw/overview).

### Closing (cycle)

`kind="cycle"` closes the current sub-path back to the most recent `move` target, matching SVG path `Z`. `cycle` carries **no** `to` field (the destination is implicit). Saves writing one explicit endpoint vs. drawing a `line` back to the start:

<ComponentPreview name="step-cycle" size="sm" />

### Quadratic curve

`kind="curve"` draws a quadratic Bezier with one control point, matching SVG path `Q`. A Bezier curve does not have to pass through the control point; the control point acts more like a handle that pulls the segment from start to end. The farther it is from the straight line, the stronger the bend. Control points currently use Cartesian `[x, y]`:

<ComponentPreview name="step-curve" size="xs" />

### Cubic curve

`kind="cubic"` draws a cubic Bezier with two control points, matching SVG path `C`. `control1` shapes the direction as the curve leaves the start; `control2` shapes the direction as it approaches the end. Because both ends can be controlled separately, this is useful for S-shaped curves and complex connectors:

<ComponentPreview name="step-cubic" size="xs" />

### Bend (arc shorthand)

`kind="bend"` is shorthand for an arced connection: say whether the segment bends left or right while looking from start to end, plus an angle. The compiler computes control points and lowers it to a cubic Bezier. `bendDirection` is the visual left/right relative to the from‚Üíto path; `bendAngle` defaults to 30:

<ComponentPreview name="step-bend" size="md" />

### Offset positioning

`to` also accepts the **`{ of, offset }`** form ‚Äî a base point plus an arbitrary `(dx, dy)`. `of` has three shapes ‚Äî node id, Cartesian literal, or PolarPosition (recursive polar) ‚Äî symmetric with `<Node position>`'s offset form:

<ComponentPreview name="step-offset-target" size="sm" />

### Relative coordinates (relative / relativeAccumulate)

`to` also accepts relative-offset objects: `{ relative: [dx, dy] }` and `{ relativeAccumulate: [dx, dy] }` both resolve against the **previous step's endpoint** (`prevEnd`, falling back to `[0, 0]` when there is none). The only difference is **whether prevEnd advances**:

- `{ relative }` ‚Äî chained offsets all anchor on the **same** point; prevEnd does **not** update
- `{ relativeAccumulate }` ‚Äî each segment accumulates and prevEnd moves with it

You can also write the sugar strings `'+dx,dy'` / `'++dx,dy'` (e.g. `<Step to="+1,0" />`); the React adapter rewrites them in place to the matching IR objects. In the demo below, the two paths have identical step sequences and only the target form differs ‚Äî the solid line (relative) traces an L, while the dashed line (relativeAccumulate) climbs a staircase:

<ComponentPreview name="step-relative" size="sm" />

### Arc

`kind="arc"` uses the **cursor as center** and draws a circular arc by start / end angle and radius. It describes "sweep from this angle to that angle", needs no `to`, and is useful for angle marks, circle fragments, or geometric guides. Angles follow the retikz polar convention: 0¬∞ points to +x, 90¬∞ is visually downward. The pen ends at the arc's endpoint:

<ComponentPreview name="step-arc" size="sm" />

### Circle

`kind="circlePath"` uses the **cursor as center** and draws a fully closed circle by `radius` ‚Äî the pen returns to the center after closing. Useful for endpoint dots, circular node borders, etc.:

<ComponentPreview name="step-circle" size="xs" />

### Ellipse

`kind="ellipsePath"` uses the **cursor as center** and draws a fully closed ellipse by `radiusX` / `radiusY` ‚Äî the pen returns to the center. When `radiusX === radiusY` it degenerates into a circle:

<ComponentPreview name="step-ellipse" size="sm" />

### Rectangle

`kind="rectangle"` uses `from` and `to` as opposite corners of an axis-aligned rectangle; optional `cornerRadius` rounds the corners. Unlike `line`, it does not depend on the current cursor, which keeps it close to TikZ `(a) rectangle (b)`. For everyday rectangles prefer the `<Rectangle>` sugar in [Rectangle](/core/components/shapes/rectangle); use this low-level kind when authoring IR-like step sequences or mixing it with other step kinds.

<ComponentPreview name="step-rectangle" size="sm" />

### Edge label

Attach a `label` to a `<Step>` to drop a single line of text near the segment's start / middle / end. `label` is an object:

- `text`: required, single-line label content
- `position`: one of seven keywords `'at-start' | 'very-near-start' | 'near-start' | 'midway' | 'near-end' | 'very-near-end' | 'at-end'`, or a numeric t in `[0, 1]` for any fine position; default `'midway'` (t=0.5)
- `side`: `'above'` / `'below'` / `'left'` / `'right'` / `'sloped'`, default `'above'`; `'sloped'` rotates the label along the tangent so it sits on the line
- `textColor`: label text color; falls back to scope `labelDefault` ‚Üí the owning path's master `color` ‚Üí currentColor (follows the master color, not stroke)
- `opacity`: label-only opacity 0..1; multiplied with the owning path opacity (element-internal axis; does not compound across scopes)
- `font`: label font overrides (family / size / weight / style); per-field fallback to `labelDefault` ‚Üí renderer default

Two equivalent forms ‚Äî the prop form `<Step label={{...}} />`, or the sugar form `<Step><EdgeLabel>...</EdgeLabel></Step>` (needs `import { EdgeLabel } from '@retikz/react'`); **the prop wins when both are set**. `'move'` and `'cycle'` cannot carry a label ‚Äî they don't draw a segment, so any label on them is silently dropped:

<ComponentPreview name="step-label" />

#### Label color and font

An edge label's color follows the owning path's **master `color`** ‚Äî with `<Path color>` the label is auto-colored to match; setting only `stroke` (no `color`) does not, so the label keeps its default color. Use `textColor` to override explicitly, and `font` / `opacity` for smaller / faded secondary labels:

<ComponentPreview name="step-label-color" size="sm" />

A scope `labelDefault` gives a whole group of labels a shared default (see [Scope style inheritance](/core/components/layout/scope)); inheritance order: label explicit > `labelDefault` > host path master color > currentColor.

#### label.position: 7 keywords + arbitrary numeric t

`position` accepts 7 keywords or a numeric t in `[0, 1]`. Keywords are sugar for the corresponding t:

| Keyword | t | Meaning |
| --- | --- | --- |
| `'at-start'` | `0` | Segment start |
| `'very-near-start'` | `0.125` | Very near start |
| `'near-start'` | `0.25` | Near start (1/4 in) |
| `'midway'` | `0.5` | Midpoint (default) |
| `'near-end'` | `0.75` | Near end (3/4 in) |
| `'very-near-end'` | `0.875` | Very near end |
| `'at-end'` | `1` | Segment end |

A numeric form like `position={0.3}` precisely places the label along the segment. **Values outside `[0, 1]` are rejected at schema validation, not clamped at compile time** (`position={-0.1}` / `position={1.2}` fail IR validation).

#### Geometric meaning of t per step kind

t is normalized along the whole segment; each kind has its own parametrization:

| Step kind | Parametrization | t=0.25 | t=0.5 | t=0.75 |
| --- | --- | --- | --- | --- |
| `'line'` | Normalized arc length | 1/4 in | Midpoint | 3/4 in |
| `'step'` (fold N=2) | Each leg gets `1/N` of t ‚Äî the corner is always at `t = j/N`, **independent of leg length** | Leg-1 midpoint | **Corner** | Leg-2 midpoint |
| `'curve'` / `'cubic'` | Bezier parameter t (**not arc length**) | Bezier t=0.25 | Bezier t=0.5 (usually **not** the visual midpoint) | Bezier t=0.75 |
| `'bend'` | Lowered to cubic, then Bezier t | Same as cubic | Same as cubic | Same as cubic |
| `'arc'` | t maps linearly across `startAngle..endAngle` (degrees) | Quarter of the angle sweep | Mid angle | Three-quarters of the sweep |
| `'circlePath'` | Full circle t‚àà[0, 1]: t=0 at angle 0 (+x), CCW growth | 90¬∞ (visually down) | 180¬∞ (‚àíx) | 270¬∞ (visually up) |
| `'ellipsePath'` | Same as circlePath: t=0 at angle 0, CCW growth | Ellipse at 90¬∞ | Ellipse at 180¬∞ (**not** arc-length midpoint) | Ellipse at 270¬∞ |

The eight demos below show one set of t markers per kind ‚Äî `t=0.25` vs `'midway'` side-by-side ‚Äî so the keyword-as-sugar relationship and the per-kind geometry of t are clear at a glance.

##### Line

Straight segments are the most direct case: `t=0.25`, `midway`, and `t=0.75` land at one quarter, the midpoint, and three quarters of the segment length.

Quadratic Beziers use the curve parameter t. `midway` is Bezier t=0.5, which is not necessarily half of the visual arc length.

##### Bend

## API Reference

| Prop | Type | Default | Description |
| --- | --- | --- | --- |
| `kind` | `'move' \| 'line' \| 'step' \| 'cycle' \| 'curve' \| 'cubic' \| 'bend' \| 'arc' \| 'circlePath' \| 'ellipsePath' \| 'rectangle'` | `'line'` | See the "Eleven kinds" table below; IR also has `generator`, without React `<Step>` DSL |
| `via` | `'-\|' \| '\|-'` | ‚Äî | Required for `kind="fold"`: `-\|` horizontal-then-vertical; `\|-` vertical-then-horizontal |
| `from` | `IRTarget` | ‚Äî | Required only for `kind="rectangle"`: one rectangle corner |
| `to` | `IRTarget` | ‚Äî | Not used for `kind="cycle"` / `'arc'` / `'circlePath'` / `'ellipsePath'`; required for `kind="rectangle"` and other cursor-advancing kinds. Target: node id, Cartesian `[x, y]`, polar object, offset object `{ of, offset }`, proportional placement `{ between, t }`, relative-offset object `{ relative }` / `{ relativeAccumulate }`, or sugar string `'+dx,dy'` / `'++dx,dy'` |
| `control` | `[number, number]` | ‚Äî | Required for `kind="curve"`: single control point of the quadratic Bezier |
| `control1` | `[number, number]` | ‚Äî | Required for `kind="cubic"`: first control point (shapes start tangent) |
| `control2` | `[number, number]` | ‚Äî | Required for `kind="cubic"`: second control point (shapes end tangent) |
| `bendDirection` | `'left' \| 'right'` | ‚Äî | Required for `kind="bend"`: visual left / right relative to from‚Üíto direction |
| `bendAngle` | `number` | `30` | Optional for `kind="bend"`: bend angle in degrees |
| `startAngle` | `number` | ‚Äî | Required for `kind="arc"`: start angle in degrees (0¬∞ ‚Üí +x, 90¬∞ ‚Üí visually down) |
| `endAngle` | `number` | ‚Äî | Required for `kind="arc"`: end angle in degrees |
| `radius` | `number` | ‚Äî | Required for `kind="arc"` / `'circlePath'`: radius of the circle / arc |
| `radiusX` | `number` | ‚Äî | Required for `kind="ellipsePath"`: x-axis radius |
| `radiusY` | `number` | ‚Äî | Required for `kind="ellipsePath"`: y-axis radius |
| `cornerRadius` | `number` | ‚Äî | Optional for `kind="rectangle"`: corner radius |
| `label` | `IRStepLabel` | ‚Äî | Edge label (works on every kind except `'move'` / `'cycle'`): `{ text, position?, side?, textColor?, opacity?, font? }`; equivalent sugar form `<Step><EdgeLabel>‚Ä¶</EdgeLabel></Step>`, prop wins on conflict |

### Eleven kinds

| kind | Behavior | Path analogue |
| --- | --- | --- |
| `'move'` | Move cursor, **no draw** | path "M" |
| `'line'` | Straight line from cursor to target | path "L" |
| `'step'` | Right-angle fold through one corner | two path "L" segments |
| `'cycle'` | Close back to most recent `move` (no `to`) | path "Z" |
| `'curve'` | Quadratic Bezier (one control point) | path "Q" |
| `'cubic'` | Cubic Bezier (two control points) | path "C" |
| `'bend'` | Arc shorthand (direction + angle), lowered to cubic | path "C" |
| `'arc'` | Cursor as center; arc by start / end angle + radius | path "A" |
| `'circlePath'` | Cursor as center; full circle, pen returns to center | path "A"√ó2 |
| `'ellipsePath'` | Cursor as center; full ellipse, pen returns to center | path "A"√ó2 |
| `'rectangle'` | Draw a rectangle using `from` and `to` as opposite corners | TikZ `(a) rectangle (b)` |

The IR schema also contains a `generator` step for external path-generator registries. It has no corresponding React `<Step kind="generator">` DSL and is usually supplied through direct IR or a higher-level package.

The first step of a `<Path>` must be `move` ‚Äî if you write `'line'` / `'step'` / `'cycle'`, the compiler coerces it to `move` (cycle has no `to`, so it falls back to origin `[0, 0]`).

### Two via values

Only meaningful when `kind="fold"`. Given cursor `(px, py)` and target `(tx, ty)`:

| via | Corner | Path |
| --- | --- | --- |
| `'-\|'` | `(tx, py)` | Horizontal to `tx`, then vertical to `ty` |
| `'\|-'` | `(px, ty)` | Vertical to `ty`, then horizontal to `tx` |

### Fields of `label`

`label` is a single object, applied uniformly across all kinds that accept it:

| Field | Type | Default | Description |
| --- | --- | --- | --- |
| `text` | `string` | ‚Äî | Required label text (single line) |
| `position` | one of 7 keywords or `number` (0..1) | `'midway'` | Normalized position t along the segment ‚Äî keywords are sugar for t (at-start=0 / very-near-start=0.125 / near-start=0.25 / midway=0.5 / near-end=0.75 / very-near-end=0.875 / at-end=1); values outside `[0, 1]` are rejected by the schema |
| `side` | `'above' \| 'below' \| 'left' \| 'right' \| 'sloped'` | `'above'` | Anchor + offset direction; `above` / `below` shift along y, `left` / `right` along x; `sloped` keeps the anchor and rotates the label along the tangent |
| `textColor` | `string` | ‚Äî | Label text color; falls back to scope `labelDefault` ‚Üí host path master `color` ‚Üí currentColor (follows the master color, not stroke) |
| `opacity` | `number` | ‚Äî | Label-only opacity 0..1; multiplied with the owning path opacity (does not compound across scopes) |
| `font` | `object` | ‚Äî | Label font overrides (family / size / weight / style); per-field fallback to `labelDefault` ‚Üí renderer default |

The geometric meaning of t per step kind is in the "[Geometric meaning of t per step kind](#geometric-meaning-of-t-per-step-kind)" table above. In short: line uses normalized arc length; fold splits the t interval evenly by polyline leg count (the corner is always at t=0.5); curve / cubic / bend use the Bezier parameter t (**not** arc length ‚Äî t=0.5 is usually not the visual midpoint); arc uses angle parametrization; circlePath / ellipsePath traverse the full curve starting at angle 0 (+x), CCW.

### Forms of `to`

| Form | Example | Behavior |
| --- | --- | --- |
| Node id | `'a'` | Auto-attaches to Node border (see [Anchors](/core/concepts/primitive-relations/anchors)) |
| Node + connection surface | `{ id: 'a', boundary: 'circle' }` | Per-edge override of the target node's [connection surface](/core/components/draw/path) (clip shape); never changes the node footprint |
| Cartesian | `[60, 0]` | Literal `[x, y]` |
| Polar | `{ origin: 'a', angle: 30, radius: 50 }` | Polar (see [Positioning](/core/concepts/basic-concepts/position)) |
| Offset | `{ of: 'a', offset: [10, 5] }` | Base + `(dx, dy)`; `of` accepts node id, Cartesian literal, or PolarPosition |
| Proportional placement | `{ between: ['a', 'b'], t: 0.5 }` | Proportional point along A ‚Üí B; endpoints must be independently resolvable absolute targets |
| Relative (relative) | `{ relative: [10, 0] }` or `'+10,0'` | Previous step's endpoint + `[dx, dy]`; does **not** update prevEnd |
| Relative (relativeAccumulate) | `{ relativeAccumulate: [10, 0] }` or `'++10,0'` | Previous step's endpoint + `[dx, dy]`; **updates** prevEnd |
