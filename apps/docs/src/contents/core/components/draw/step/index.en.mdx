---
title: Step
description: Path's child node — one step = one action (move / line / step fold / cycle close / curve / cubic / bend / arc / circlePath / ellipsePath); `to` accepts node id / Cartesian / polar / offset (of+offset) / relative offsets (relative / relativeAccumulate); only valid as a direct child of `<Path>`.
---

`<Step>` describes a single path action: `move` lifts the pen, `line` draws a straight line, `step` folds through a single right-angle corner, `cycle` closes the path, `curve` / `cubic` / `bend` draw curves of various shapes, and `arc` / `circlePath` / `ellipsePath` use the cursor as the center to draw an arc / full circle / full ellipse. It is **only** valid as a direct child of `<Path>`; placed elsewhere it is silently ignored.

Use this table to pick a `kind` first:

| kind | Needs `to` | Draws | Advances cursor | Supports label |
| --- | --- | --- | --- | --- |
| `move` | Yes | No | Yes | No |
| `line` / `step` / `curve` / `cubic` / `bend` | Yes | Yes | Yes | Yes |
| `cycle` | No | Yes | Back to subpath start | No |
| `arc` | No | Yes | To arc endpoint | Yes |
| `circlePath` / `ellipsePath` | No | Yes | Back to circle / ellipse center | Yes |

## Usage

```tsx
import { Path, Step } from '@retikz/react';
```

```tsx
<Path>
  <Step kind="move" to="a" />
  <Step kind="line" to="b" />
</Path>
```

### Closing (cycle)

`kind="cycle"` closes the current sub-path back to the most recent `move` target, matching SVG path `Z`. `cycle` carries **no** `to` field (the destination is implicit). Saves writing one explicit endpoint vs. drawing a `line` back to the start:

### Quadratic curve

`kind="curve"` draws a quadratic Bezier with one control point, matching SVG path `Q`. The control point governs the bend direction and amplitude; control points currently use Cartesian `[x, y]`:

<ComponentPreview name="step-curve" />

### Cubic curve

`kind="cubic"` draws a cubic Bezier with two control points, matching SVG path `C`. `control1` shapes the start tangent; `control2` shapes the end tangent. Common for S-shaped curves and complex connectors:

<ComponentPreview name="step-cubic" />

### Bend (arc shorthand)

`kind="bend"` is shorthand for an arced connection — give a direction and angle, control points are computed at compile time and lowered to a cubic. `bendDirection` is the visual left/right relative to the from→to path; `bendAngle` defaults to 30:

<ComponentPreview name="step-bend" size="lg" />

### Offset positioning

`to` also accepts the **`{ of, offset }`** form — a base point plus an arbitrary `(dx, dy)`. `of` has three shapes — node id, Cartesian literal, or PolarPosition (recursive polar) — symmetric with `<Node position>`'s offset form:

<ComponentPreview name="step-offset-target" size="lg" />

### Relative coordinates (relative / relativeAccumulate)

`to` also accepts relative-offset objects: `{ relative: [dx, dy] }` and `{ relativeAccumulate: [dx, dy] }` both resolve against the **previous step's endpoint** (`prevEnd`, falling back to `[0, 0]` when there is none). The only difference is **whether prevEnd advances**:

- `{ relative }` — chained offsets all anchor on the **same** point; prevEnd does **not** update
- `{ relativeAccumulate }` — each segment accumulates and prevEnd moves with it

You can also write the sugar strings `'+dx,dy'` / `'++dx,dy'` (e.g. `<Step to="+1,0" />`); the React adapter rewrites them in place to the matching IR objects. In the demo below, the two paths have identical step sequences and only the target form differs — the solid line (relative) traces an L, while the dashed line (relativeAccumulate) climbs a staircase:

<ComponentPreview name="step-relative" />

### Arc

`kind="arc"` uses the **cursor as center** and draws a circular arc by start / end angle and radius. Angles follow the retikz polar convention: 0° points to +x, 90° is visually downward. The pen ends at the arc's endpoint:

<ComponentPreview name="step-arc" />

### Circle

`kind="circlePath"` uses the **cursor as center** and draws a fully closed circle by `radius` — the pen returns to the center after closing. Useful for endpoint dots, circular node borders, etc.:

<ComponentPreview name="step-circle" />

### Ellipse

`kind="ellipsePath"` uses the **cursor as center** and draws a fully closed ellipse by `radiusX` / `radiusY` — the pen returns to the center. When `radiusX === radiusY` it degenerates into a circle:

<ComponentPreview name="step-ellipse" />

### Edge label

Attach a `label` to a `<Step>` to drop a single line of text near the segment's start / middle / end. `label` is an object:

- `text`: required, single-line label content
- `position`: one of seven keywords `'at-start' | 'very-near-start' | 'near-start' | 'midway' | 'near-end' | 'very-near-end' | 'at-end'`, or a numeric t in `[0, 1]` for any fine position; default `'midway'` (t=0.5)
- `side`: `'above'` / `'below'` / `'left'` / `'right'` / `'sloped'`, default `'above'`; `'sloped'` rotates the label along the tangent so it sits on the line

Two equivalent forms — the prop form `<Step label={{...}} />`, or the sugar form `<Step><EdgeLabel>...</EdgeLabel></Step>` (needs `import { EdgeLabel } from '@retikz/react'`); **the prop wins when both are set**. `'move'` and `'cycle'` cannot carry a label — they don't draw a segment, so any label on them is silently dropped:

<ComponentPreview name="step-label" />

#### label.position: 7 keywords + arbitrary numeric t

`position` accepts 7 keywords or a numeric t in `[0, 1]`. Keywords are sugar for the corresponding t:

| Keyword | t | Meaning |
| --- | --- | --- |
| `'at-start'` | `0` | Segment start |
| `'very-near-start'` | `0.125` | Very near start |
| `'near-start'` | `0.25` | Near start (1/4 in) |
| `'midway'` | `0.5` | Midpoint (default) |
| `'near-end'` | `0.75` | Near end (3/4 in) |
| `'very-near-end'` | `0.875` | Very near end |
| `'at-end'` | `1` | Segment end |

A numeric form like `position={0.3}` precisely places the label along the segment. **Values outside `[0, 1]` are rejected at schema validation, not clamped at compile time** (`position={-0.1}` / `position={1.2}` fail IR validation).

#### Geometric meaning of t per step kind

t is normalized along the whole segment; each kind has its own parametrization:

| Step kind | Parametrization | t=0.25 | t=0.5 | t=0.75 |
| --- | --- | --- | --- | --- |
| `'line'` | Normalized arc length | 1/4 in | Midpoint | 3/4 in |
| `'step'` (fold N=2) | Each leg gets `1/N` of t — the corner is always at `t = j/N`, **independent of leg length** | Leg-1 midpoint | **Corner** | Leg-2 midpoint |
| `'curve'` / `'cubic'` | Bezier parameter t (**not arc length**) | Bezier t=0.25 | Bezier t=0.5 (usually **not** the visual midpoint) | Bezier t=0.75 |
| `'bend'` | Lowered to cubic, then Bezier t | Same as cubic | Same as cubic | Same as cubic |
| `'arc'` | t maps linearly across `startAngle..endAngle` (degrees) | Quarter of the angle sweep | Mid angle | Three-quarters of the sweep |
| `'circlePath'` | Full circle t∈[0, 1]: t=0 at angle 0 (+x), CCW growth | 90° (visually down) | 180° (−x) | 270° (visually up) |
| `'ellipsePath'` | Same as circlePath: t=0 at angle 0, CCW growth | Ellipse at 90° | Ellipse at 180° (**not** arc-length midpoint) | Ellipse at 270° |

## API Reference

| Prop | Type | Default | Description |
| --- | --- | --- | --- |
| `kind` | `'move' \| 'line' \| 'step' \| 'cycle' \| 'curve' \| 'cubic' \| 'bend' \| 'arc' \| 'circlePath' \| 'ellipsePath'` | `'line'` | See the "Ten kinds" table below |
| `via` | `'-\|' \| '\|-'` | — | Required for `kind="step"`: `-\|` horizontal-then-vertical; `\|-` vertical-then-horizontal |
| `to` | `IRTarget` | — | Not used for `kind="cycle"` / `'arc'` / `'circlePath'` / `'ellipsePath'`; required for the rest. Target: node id, Cartesian `[x, y]`, polar object, offset object `{ of, offset }`, relative-offset object `{ relative }` / `{ relativeAccumulate }`, or sugar string `'+dx,dy'` / `'++dx,dy'` |
| `control` | `[number, number]` | — | Required for `kind="curve"`: single control point of the quadratic Bezier |
| `control1` | `[number, number]` | — | Required for `kind="cubic"`: first control point (shapes start tangent) |
| `control2` | `[number, number]` | — | Required for `kind="cubic"`: second control point (shapes end tangent) |
| `bendDirection` | `'left' \| 'right'` | — | Required for `kind="bend"`: visual left / right relative to from→to direction |
| `bendAngle` | `number` | `30` | Optional for `kind="bend"`: bend angle in degrees |
| `startAngle` | `number` | — | Required for `kind="arc"`: start angle in degrees (0° → +x, 90° → visually down) |
| `endAngle` | `number` | — | Required for `kind="arc"`: end angle in degrees |
| `radius` | `number` | — | Required for `kind="arc"` / `'circlePath'`: radius of the circle / arc |
| `radiusX` | `number` | — | Required for `kind="ellipsePath"`: x-axis radius |
| `radiusY` | `number` | — | Required for `kind="ellipsePath"`: y-axis radius |
| `label` | `IRStepLabel` | — | Edge label (works on every kind except `'move'` / `'cycle'`): `{ text, position?, side? }`; equivalent sugar form `<Step><EdgeLabel>…</EdgeLabel></Step>`, prop wins on conflict |

### Ten kinds

| kind | Behavior | Path analogue |
| --- | --- | --- |
| `'move'` | Move cursor, **no draw** | path "M" |
| `'line'` | Straight line from cursor to target | path "L" |
| `'step'` | Right-angle fold through one corner | two path "L" segments |
| `'cycle'` | Close back to most recent `move` (no `to`) | path "Z" |
| `'curve'` | Quadratic Bezier (one control point) | path "Q" |
| `'cubic'` | Cubic Bezier (two control points) | path "C" |
| `'bend'` | Arc shorthand (direction + angle), lowered to cubic | path "C" |
| `'arc'` | Cursor as center; arc by start / end angle + radius | path "A" |
| `'circlePath'` | Cursor as center; full circle, pen returns to center | path "A"×2 |
| `'ellipsePath'` | Cursor as center; full ellipse, pen returns to center | path "A"×2 |

The first step of a `<Path>` must be `move` — if you write `'line'` / `'step'` / `'cycle'`, the compiler coerces it to `move` (cycle has no `to`, so it falls back to origin `[0, 0]`).

### Two via values

Only meaningful when `kind="step"`. Given cursor `(px, py)` and target `(tx, ty)`:

| via | Corner | Path |
| --- | --- | --- |
| `'-\|'` | `(tx, py)` | Horizontal to `tx`, then vertical to `ty` |
| `'\|-'` | `(px, ty)` | Vertical to `ty`, then horizontal to `tx` |

### Fields of `label`

`label` is a single object, applied uniformly across all kinds that accept it:

| Field | Type | Default | Description |
| --- | --- | --- | --- |
| `text` | `string` | — | Required label text (single line) |
| `position` | one of 7 keywords or `number` (0..1) | `'midway'` | Normalized position t along the segment — keywords are sugar for t (at-start=0 / very-near-start=0.125 / near-start=0.25 / midway=0.5 / near-end=0.75 / very-near-end=0.875 / at-end=1); values outside `[0, 1]` are rejected by the schema |
| `side` | `'above' \| 'below' \| 'left' \| 'right' \| 'sloped'` | `'above'` | Anchor + offset direction; `above` / `below` shift along y, `left` / `right` along x; `sloped` keeps the anchor and rotates the label along the tangent |

The geometric meaning of t per step kind is in the "[Geometric meaning of t per step kind](#geometric-meaning-of-t-per-step-kind)" table above. In short: line / fold use normalized arc length (fold's corner is always at t=0.5), curve / cubic / bend use the Bezier parameter t (**not** arc length — t=0.5 is usually not the visual midpoint), arc uses angle parametrization, and circlePath / ellipsePath traverse the full curve starting at angle 0 (+x), CCW.

### Six forms of `to`

| Form | Example | Behavior |
| --- | --- | --- |
| Node id | `'a'` | Auto-attaches to Node border (see [Anchors](/core/concepts/anchors)) |
| Cartesian | `[60, 0]` | Literal `[x, y]` |
| Polar | `{ origin: 'a', angle: 30, radius: 50 }` | Polar (see [Positioning](/core/concepts/positioning)) |
| Offset | `{ of: 'a', offset: [10, 5] }` | Base + `(dx, dy)`; `of` accepts node id, Cartesian literal, or PolarPosition |
| Relative (relative) | `{ relative: [10, 0] }` or `'+10,0'` | Previous step's endpoint + `[dx, dy]`; does **not** update prevEnd |
| Relative (relativeAccumulate) | `{ relativeAccumulate: [10, 0] }` or `'++10,0'` | Previous step's endpoint + `[dx, dy]`; **updates** prevEnd |
