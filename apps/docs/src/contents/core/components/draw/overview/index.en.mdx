description: Sugar component — a way array threads through nodes, coordinates, folds, curves, close operations, edge labels, and styling, then expands to equivalent Path + Step.

`<Draw>` is the default entry point for most connections. It accepts a `way` array and expands targets, folds, cycles, curves, shapes, and edge labels into Kernel `<Path>` + `<Step>`. Stroke, fill, arrow, and opacity props match `<Path>`.

Choosing rule: use `<Draw>` for ordinary relationship lines, flowchart edges, and arrowed connections; switch to `<Path>` + `<Step>` when you need multiple `move`s, precise per-segment `kind` control, or IR-close debugging; use the **shape sugar** below to draw a single geometric shape (circle / rectangle / sector / grid, etc.); work with IR directly only at AI / persistence boundaries.

## Shape sugar

Beyond the `way` array, the Draw family also offers a set of **shape sugar** — draw a common geometric shape in one line, also expanding to `<Path>` and sharing the same visual props as `<Draw>` / `<Path>` (`fill` / `stroke` / `strokeWidth` / `dashPattern` / `arrow` / `zIndex`, etc.):

| Component | Draws |
| --- | --- |
| `<Circle>` | circle / half circle / segment |
| `<Ellipse>` | ellipse / partial ellipse |
| `<Arc>` | open circular / elliptical arc |
| `<Sector>` | wedge (closed through the center) |
| `<Rectangle>` | rectangle / rounded rectangle |
| `<Grid>` | background grid (expands to many lines) |
| `<RegularPolygon>` | regular polygon |
| `<Star>` | star |

See [Shapes](/core/components/draw/shapes) for each form, the point contract (pass-through Target vs literal Cartesian), and per-component props. Difference from `<Node shape="...">`: shape sugar is a plain text-less path; a Node is an object with text / anchors that edges can connect to.

## Usage

```tsx
import { Draw } from '@retikz/react';
```

```tsx
<Draw way={['a', 'b']} />
```

### Curves

Draw has two curve families: `curve` / `cubic` / `bend` connect the previous target to the next target, while `arc` draws a circular arc around the previous point. In Bezier curves, control points are not necessarily points the curve passes through; they are handles that pull the curve's direction and amount of bend.

| Form | Mental model | Good for |
| --- | --- | --- |
| `{ curve }` | One control point pulls a simple bend | Ordinary single-arc connectors |
| `{ cubic }` | Two control points shape the start and end directions separately | S-curves, complex connectors, precise tangents |
| `{ bend }` | Direction + angle shorthand; control points are computed for you | Quick curved links between two nodes |
| `{ arc }` | Circular arc around the previous point by angles and radius | Angle marks, circle fragments, geometry diagrams |

#### Quadratic Bezier

`{ curve: [cx, cy] }` is equivalent to `<Step kind="curve">`. It draws from the previous target to the next target, using one control point to set the bend direction and strength. The farther the control point is, the more strongly the curve is pulled.

`{ curve: [cx, cy] }` maps to TikZ's `.. controls (B) ..` path control-point syntax.

</Comparison>

<ComponentPreview name="draw-curve" />

#### Cubic Bezier

`{ cubic: [[c1x, c1y], [c2x, c2y]] }` is equivalent to `<Step kind="cubic">`. It draws a more controllable Bezier curve with two control points: the first shapes how the curve leaves the start point; the second shapes how it approaches the end point. This makes it better than a quadratic curve for S-curves or connectors that need precise tuning.

<ComponentPreview name="draw-cubic" />

#### Bend

`{ bend: 'left' | 'right', angle?: number }` is equivalent to `<Step kind="bend">`. It is shorthand for a curved connection: you only choose whether the curve bends left or right as seen from start to end, plus an optional angle; the compiler computes the control points and expands it to a cubic Bezier.

<ComponentPreview name="draw-bend" size="lg" />

#### Arc

`{ arc: { startAngle, endAngle, radius } }` is equivalent to `<Step kind="arc">`. It draws a circular arc around the previous item. It does not connect to the next target and does not consume the next item. Think of it as "sweep from one angle to another at this radius"; it is useful for angle marks, circle fragments, and geometry helpers.

### Edge Labels

#### Basic Form

`{ label }` is an infix label operator in Draw way. It decorates the next produced segment by setting `step.label`; it does not produce a segment itself. The shorthand `{ label: 'ok' }` is equivalent to `{ label: { text: 'ok' } }`.

<ComponentPreview name="draw-label" size="lg" />

#### Position Rules

`label.position` accepts seven keywords or a numeric `0..1` t. The geometry of t depends on step kind: line / fold use normalized path position, curve / cubic / bend use Bezier t, and arc / circle / ellipse use angular parametrization.

Line segments are the most direct: `start` / `midway` / `end` land near the start, middle, and end of the segment.

<ComponentPreview name="draw-label-line" />

Folded segments divide t evenly across the folded legs. `midway` lands on the corner, not at the visual midpoint by actual segment length.

<ComponentPreview name="draw-label-fold" />

Quadratic Beziers use the curve parameter t. `midway` means Bezier t=0.5, which is not always halfway along the visible curve.

<ComponentPreview name="draw-label-curve" />

Cubic Beziers also place labels by Bezier t; changing the control points changes both the position and tangent at that t.

<ComponentPreview name="draw-label-cubic" />

`bend` expands to a cubic Bezier, so its label positioning follows the same rule as cubic.

<ComponentPreview name="draw-label-bend" />

Arcs place labels by angle interpolation; `midway` means the middle angle between `startAngle` and `endAngle`.

<ComponentPreview name="draw-label-arc" />

Full circles start at the 0° position and traverse the circle; numeric t maps to the circular angle.

<ComponentPreview name="draw-label-circle" />

Full ellipses also use angular parametrization, not equal divisions by ellipse arc length.

<ComponentPreview name="draw-label-ellipse" />

### Style

#### Arrows

`arrow` controls endpoint arrows: `'->'` / `'<-'` / `'<->'` / `'none'`. For shape, size, color, and per-end overrides, see [Arrow](/core/components/draw/arrow).

<ComponentPreview name="draw-arrow" />

#### Fill

`fill` colors closed regions, and `fillRule` controls nested sub-path filling. It is commonly used with `DrawWay.Cycle`.

<ComponentPreview name="draw-fill" />

#### Dash

`dashPattern` uses `Array<number>` for dash / gap lengths. `[4, 2]` means draw 4, skip 2, in user units.

<ComponentPreview name="draw-dashed" />

#### Stacking order (zIndex)

`zIndex` overrides the default declaration-order stacking: higher draws on top, omitted = 0, ties keep declaration order; effective only among siblings at the same level. Same meaning as `<Path>`'s `zIndex` (Draw compiles to Path). The two filled squares below are declared blue-then-red (red on top by default); setting `zIndex={1}` on the earlier blue Draw lifts it above the red one:

## API Reference

### Props

| Prop | Type | Default | Description |
| --- | --- | --- | --- |
| `way` | `Array<WayItem>` | — | At least 2 items; first is always `move`, rest become `line` / `step` / `cycle` and related actions based on item form |
| `stroke` | `string` | `currentColor` | Stroke color |
| `strokeWidth` | `number` | `1` | Stroke width |
| `dashPattern` | `Array<number>` | — | Dash / gap length sequence, for example `[4, 2]` |
| `lineCap` | `'butt' \| 'round' \| 'square'` | `'butt'` | Stroke endpoint shape |
| `lineJoin` | `'miter' \| 'round' \| 'bevel'` | `'miter'` | Stroke corner shape |
| `thickness` | `'ultraThin' \| 'veryThin' \| 'thin' \| 'semithick' \| 'thick' \| 'veryThick' \| 'ultraThick'` | — | Semantic stroke preset; explicit `strokeWidth` wins |
| `arrow` | `'none' \| '->' \| '<-' \| '<->'` | — | Path-level arrow direction |
| `arrowDetail` | `{ shape?, scale?, length?, width?, color?, fill?, opacity?, lineWidth?, start?, end? }` | — | Detailed arrow configuration; see [Arrow](/core/components/draw/arrow) |
| `fill` | `string` | `'none'` | Fill color for closed regions |
| `fillRule` | `'nonzero' \| 'evenodd'` | `'nonzero'` | Fill rule |
| `opacity` | `number` | — | Whole-path opacity, 0..1 |
| `fillOpacity` | `number` | — | Fill-only opacity, 0..1 |
| `drawOpacity` | `number` | — | Stroke-only opacity, 0..1 |
| `zIndex` | `number` (integer) | `0` | Explicit stacking order: higher draws on top; ties keep declaration order; effective only among siblings at the same level (forwarded to the underlying `<Path>`) |

<Comparison target="tikz" title="Style Naming">

`lineCap` / `lineJoin` map to TikZ `line cap` / `line join`; `drawOpacity` maps to `draw opacity`; `thickness` presets reuse semantic stroke-width names from `ultra thin` through `ultra thick`.

</Comparison>

### Curve / Shape / Label Way Items

| Way item | Parameters | Consumes next item | Equivalent Step | Description |
| --- | --- | --- | --- | --- |
| `{ curve }` | `[cx, cy]` | Yes | `kind="curve"` | Quadratic Bezier control point |
| `{ cubic }` | `[[c1x, c1y], [c2x, c2y]]` | Yes | `kind="cubic"` | Two cubic Bezier control points |
| `{ bend }` | `'left' \| 'right'` | Yes | `kind="bend"` | Bend direction, default angle 30 |
| `{ bend, angle }` | `number` | Yes | `kind="bend"` | Override the bend angle |
| `{ arc }` | `{ startAngle, endAngle, radius }` | No | `kind="arc"` | Draw an arc around the current point |
| `{ circle }` | `{ radius }` | No | `kind="circlePath"` | Draw a full circle around the current point |
| `{ ellipse }` | `{ radiusX, radiusY }` | No | `kind="ellipsePath"` | Draw a full ellipse centered on the current point |
| `{ label }` | `string \| StepLabel` | Yes | `step.label` | Decorates the next segment; string shorthand equals `{ text }` |

### StepLabel

| Field | Type | Default | Description |
| --- | --- | --- | --- |
| `text` | `string` | — | Label text |
| `position` | `StepLabelPosition \| number` | `'midway'` | Keyword or numeric `0..1` t value |
| `side` | `'above' \| 'below'` | `'above'` | Which side of the line to place the text on |
