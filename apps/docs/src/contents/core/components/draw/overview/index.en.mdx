---

`<Draw>` is the go-to for 90% of connections. `way` is an array of targets (with infix operators) joined as a path; at compile time it expands to an equivalent Kernel `<Path>` + `<Step>` tree, so `<Draw>` adds no capability the Kernel can't already express.

## Usage

```tsx
import { Draw } from '@retikz/react';
```

```tsx
<Draw way={['a', 'b']} />
```

### Fourteen forms of way item

Each item in `way` accepts fourteen forms ‚Äî the first five are direct targets (matching `<Step>.to`), the rest are infix operators / keywords. **Note**: curve operators (curve / cubic / bend) merge with the next item; shape operators (arc / circle / ellipse) center on the previous item and **do not** consume the next; the edge-label operator (label) decorates the **next** segment without producing one itself.

| Form | Example | Meaning |
| --- | --- | --- |
| Node id | `'a'` | Reference a Node in the same Tikz; endpoint auto-attaches to border |
| Cartesian | `[120, 0]` | Literal `[x, y]` |
| Polar | `{ origin: 'a', angle: 60, radius: 80 }` | Polar (see [Positioning](/core/concepts/positioning)) |
| Relative (Relative) | `{ position: [10, 0], type: DrawWay.Relative }` or `'+10,0'` | Previous step's endpoint + `[dx, dy]`; does **not** update prevEnd (TikZ `(+x, +y)`) |
| Relative (Accumulate) | `{ position: [10, 0], type: DrawWay.Accumulate }` or `'++10,0'` | Previous step's endpoint + `[dx, dy]`; **updates** prevEnd (TikZ `(++x, ++y)`) |
| Fold operator | `'-\|'` / `'\|-'` (or `DrawWay.Hv` / `DrawWay.Vh`) | Turn the "previous ‚Üí next" segment into a fold (TikZ `-\|` / `\|-` infix) |
| Cycle | `DrawWay.Cycle` | Close back to way's first item (equivalent to TikZ `cycle` / SVG `Z`) |
| Quadratic curve | `{ curve: [cx, cy] }` | Merge with next item into a curve step; one control point (TikZ `..controls(B)..`) |
| Cubic curve | `{ cubic: [[c1x, c1y], [c2x, c2y]] }` | Merge with next item into a cubic step; two control points (TikZ `..controls(B) and (C)..`) |
| Bend | `{ bend: 'left' \| 'right', angle?: number }` | Merge with next item into a bend step; direction + optional angle (default 30¬∞) |
| Arc operator | `{ arc: { startAngle, endAngle, radius } }` | Center on the previous item; draw an arc ‚Äî **does not** consume the next item |
| Circle operator | `{ circle: { radius } }` | Center on the previous item; draw a full circle ‚Äî **does not** consume the next item |
| Ellipse operator | `{ ellipse: { radiusX, radiusY } }` | Center on the previous item; draw a full ellipse ‚Äî **does not** consume the next item |
| Edge label operator | `{ label: 'accept' }` or `{ label: { text, position?, side? } }` | Decorate the **next** segment ‚Äî sets IR `step.label` on the next step-producing item (ADR-0004) |

### DrawWay keyword constants

`DrawWay` exposes named constants for special way-array items:

```tsx
import { Draw, DrawWay } from '@retikz/react';

<Draw way={['a', 'b', 'c', DrawWay.Cycle]} />            {/* close */}
<Draw way={['a', '-|', 'b']} />                          {/* fold via raw literal */}
<Draw way={['a', DrawWay.Hv, 'b']} />                    {/* fold via constant */}
<Draw way={['a', { position: [80, 0], type: DrawWay.Relative }]} />  {/* relative offset, TS-checked */}
```

| Constant | Raw literal | Meaning |
| --- | --- | --- |
| `DrawWay.Cycle` | (**const-only**) | Close keyword. The underlying string is intentionally ugly (`'retikz-keyword_cycle'`) so it can never collide with a node id. |
| `DrawWay.Hv` | `'-\|'` | Fold: horizontal-then-vertical (infix operator) |
| `DrawWay.Vh` | `'\|-'` | Fold: vertical-then-horizontal (infix operator) |
| `DrawWay.Relative` | (**const-only**) | `type` discriminator for the `WayRelItem` object ‚Äî equivalent to TikZ `(+x, +y)` (does **not** advance prevEnd) |
| `DrawWay.Accumulate` | (**const-only**) | `type` discriminator for the `WayRelItem` object ‚Äî equivalent to TikZ `(++x, ++y)` (advances prevEnd each segment) |

**Why cycle / relative bans the raw literal**: `'cycle'` is a common English word, very likely to collide with a real node id. Cycle therefore goes **only** through `DrawWay.Cycle`; a bare string `'cycle'` in `way` is just a node id reference. Same reason for `Relative` / `Accumulate` ‚Äî the `type` field never accepts a free string, so the object form is fully TS-checkable.

The fold values `'-\|'` / `'\|-'` contain reserved punctuation, so they cannot collide with node ids ‚Äî **both raw literal and constant work**, pick whichever fits.

### Folded segments

Drop `'-|'` or `'|-'` directly into `way` (infix operator) and the "previous ‚Üí next" segment becomes a fold ‚Äî matches TikZ's `(A) -| (B)` form, equivalent to `<Step kind="step" via to>`. Essential for flow charts and UML relationships:

### Quadratic curve

Drop `{ curve: [cx, cy] }` (infix operator) into `way` and the "previous ‚Üí next" segment becomes a quadratic Bezier ‚Äî matching TikZ's `..controls(B)..`, equivalent to `<Step kind="curve">`:

### Cubic curve

`{ cubic: [[c1x, c1y], [c2x, c2y]] }` draws a cubic Bezier ‚Äî two control points shape the start and end tangents independently, matching TikZ's `..controls(B) and (C)..`, equivalent to `<Step kind="cubic">`. Common for S-shaped or complex curves:

<ComponentPreview name="draw-cubic" />

### Bend (arc shorthand)

`{ bend: 'left' | 'right', angle?: number }` is the easier curve form ‚Äî give a direction and angle (default 30¬∞), control points are computed at compile time and lowered to a cubic. Equivalent to `<Step kind="bend">`:

<ComponentPreview name="draw-bend" />

### Arc

`{ arc: { startAngle, endAngle, radius } }` centers on the **previous item** and draws a circular arc ‚Äî equivalent to `<Step kind="arc">`. Unlike curve / cubic / bend, the arc operator **does not** consume the next item ‚Äî the next way item is parsed normally:

<ComponentPreview name="draw-arc" />

### Circle

`{ circle: { radius } }` centers on the **previous item** and draws a full circle ‚Äî equivalent to `<Step kind="circlePath">`. Also **does not** consume the next item; the pen returns to the center after closing. Useful for drawing several independent circles in one go:

<ComponentPreview name="draw-circle" />

### Ellipse

`{ ellipse: { radiusX, radiusY } }` centers on the **previous item** and draws a full ellipse ‚Äî equivalent to `<Step kind="ellipsePath">`. Also **does not** consume the next item:

### Edge label

Drop a `{ label: ... }` operator (infix) into `way` to set the IR `step.label` on the **next** segment ‚Äî equivalent to the Kernel form `<Step><EdgeLabel>...</EdgeLabel></Step>` (see [Step's edge label](/core/components/draw/step#edge-label)). Shorthand: pass a string directly: `{ label: 'accept' }` ‚â° `{ label: { text: 'accept' } }`.

The label operator can sit before any segment-producing way item ‚Äî line target, `-|` / `|-` fold, `{ curve }` / `{ cubic }` / `{ bend }` / `{ arc }` / `{ circle }` / `{ ellipse }`. It **cannot** decorate `DrawWay.Cycle` (IR forbids labels on cycle); a trailing label or two consecutive label operators also throw:

### Relative coordinates (Relative / Accumulate)

Two equivalent ways to express relative offsets in `way`:

- **Object (recommended)**: `{ position: [dx, dy], type: DrawWay.Relative | DrawWay.Accumulate }` ‚Äî fully TS-checked; `position` is `[number, number]` and `type` is a union of two literal constants, so the IDE catches typos and gives autocompletion.
- **Sugar string**: `'+dx,dy'` / `'++dx,dy'` ‚Äî terser, TikZ-idiomatic, but parsed at runtime and unchecked by TS.

Both resolve against the **previous step's endpoint** (`prevEnd`, falling back to `[0, 0]` when there is none). The only semantic difference is whether prevEnd advances: `Relative` (= `'+x,y'`) does **not** advance (chained offsets share the same anchor ‚Äî TikZ `(+x, +y)`); `Accumulate` (= `'++x,y'`) **does** (each segment accumulates ‚Äî TikZ `(++x, ++y)`). The two Draws below use the object form and differ only in `type` ‚Äî the solid line traces an L, the dashed line climbs a staircase:

### Arrows

The `arrow` prop controls path-end arrows: `'->'` / `'<-'` / `'<->'` / `'none'` (default). For multi-node paths, the arrow attaches only to the overall start / end ‚Äî never on intermediate nodes:

### Endpoints / corners + thickness preset + opacity

Draw forwards the rest of Path's stroke-appearance fields: `lineCap` / `lineJoin` (SVG `stroke-linecap` / `stroke-linejoin`); `thickness` is a TikZ-style semantic preset (`'ultraThin'` ‚Ä¶ `'ultraThick'`, with `thin = 1` matching the default `strokeWidth`; explicit `strokeWidth` always wins); three independent opacity channels ‚Äî `opacity` overall, `fillOpacity` for fill only, `drawOpacity` for stroke only (TikZ `draw opacity`). See [Path](/core/components/draw/path) for examples.

## API Reference

| Prop | Type | Default | Description |
| --- | --- | --- | --- |
| `way` | `Array<WayItem>` | ‚Äî | At least 2 items; first is always `move`, rest are `line` / `step` (fold) / `cycle` (close) depending on item form |
| `stroke` | `string` | `currentColor` | Stroke color |
| `strokeWidth` | `number` | `1` | Stroke width |
| `strokeDasharray` | `string` | ‚Äî | SVG `stroke-dasharray` pattern (e.g. `"4 2"`) |
| `lineCap` | `'butt' \| 'round' \| 'square'` | `'butt'` | Stroke endpoint shape (SVG `stroke-linecap`) |
| `lineJoin` | `'miter' \| 'round' \| 'bevel'` | `'miter'` | Stroke corner shape (SVG `stroke-linejoin`) |
| `thickness` | `'ultraThin' \| 'veryThin' \| 'thin' \| 'semithick' \| 'thick' \| 'veryThick' \| 'ultraThick'` | ‚Äî | Semantic stroke preset (`thin = 1`); explicit `strokeWidth` always wins |
| `arrow` | `'none' \| '->' \| '<-' \| '<->'` | ‚Äî | Path-level arrow direction |
| `arrowShape` | `'normal' \| 'open' \| 'stealth' \| 'diamond' \| 'openDiamond' \| 'circle' \| 'openCircle'` | `'normal'` | Arrow tip shape |
| `fill` | `string` | `'none'` | Fill color of the closed region |
| `fillRule` | `'nonzero' \| 'evenodd'` | `'nonzero'` | Fill rule |
| `opacity` | `number` | ‚Äî | Whole-path opacity 0..1 |
| `fillOpacity` | `number` | ‚Äî | Fill-only opacity 0..1 |
| `drawOpacity` | `number` | ‚Äî | Stroke-only opacity 0..1 (TikZ `draw opacity`) |
