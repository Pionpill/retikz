`<Path>` is the low-level path container: children must be `<Step>` nodes (at least 2) defining a sequence of move / line / step (folded) / cycle (close) / curve / cubic / bend (curves) / arc / circlePath / ellipsePath (circle / ellipse) / rectangle actions; plus an IR-level generator step ([path generators](/core/reference/extending/path-generator)). For everyday connections `<Draw>` is shorter; reach for `<Path>` when you need folds, closes, curves, custom stroke, or different step kinds.

It is strongest in three cases: one path contains multiple subpaths (multiple `move`s), individual segments need different `kind` / label / control-point behavior, or you want the JSX structure to mirror IR `path.children` closely for debugging and generation.

## Usage

```tsx
import { Path, Step } from '@retikz/react';
```

```tsx
<Path stroke="currentColor">
  <Step kind="move" to="a" />
  <Step kind="line" to="b" />
</Path>
```

## Composition

`<Path>` requires `<Step>` children (at least 2); if the first isn't `move`, the compiler coerces it to `move`:

```tsx
<Path>
  <Step kind="move" to={...} />     {/* First: must be move */}
  <Step kind="line" to={...} />     {/* Then any number of line / move / step */}
  ...
</Path>
```

`<Step>` supports eleven kinds — `move` / `line` for straight segments, `step` for the right-angle fold, `cycle` to close back, `curve` / `cubic` for Bezier curves, `bend` as the arc shorthand (with out/in angles and self-loops), `arc` / `circlePath` / `ellipsePath` for arcs / full circles / full ellipses centered on the cursor, and `rectangle` for a diagonal rectangle. See [Step](/core/components/draw/step) for details. There is also an IR-level `generator` step (external curve generators, no JSX DSL, fed via IR — see [path generators](/core/reference/extending/path-generator)).

### Arrows

`arrow` picks direction (`'->'` / `'<-'` / `'<->'` / `'none'`); `arrowDetail` is a single object covering every visual spec (shape / scale / length / width / color / fill / opacity / lineWidth) plus optional `start` / `end` sub-objects for per-end overrides. The full reference — seven built-in shapes, per-end merge semantics, color / sizing / opacity, and the complete API table — lives on the **[Arrow](/core/components/draw/arrow)** page.

### Endpoint connection surface (boundary)

When an endpoint references a node (the `{ id }` object form, or an id inside a way), you can add a `boundary` field to override the target node's [`connectAs`](/core/components/node/overview) (connection surface) **for that endpoint only**. It takes the same forms as `connectAs`: `'shape'` (the visual shape itself) / `'circle'` (a true circle) / any registered shape name / `{ type, params }`.

```tsx
<Path arrow="->">
  <Step kind="move" to="a" />
  <Step kind="line" to={{ id: 'star', boundary: 'shape' }} />
</Path>
```

`boundary` only applies where a connection surface is meaningful: endpoint auto-clip (no explicit `anchor`) and that endpoint's compass / angle anchor. In direction-less reference contexts — `between` endpoints, an `offset` `of`, a node center — it is a no-op. A node's default connection surface lives on [`<Node connectAs>`](/core/components/node/overview); `boundary` is the per-edge override.

### Curves and transforms

#### Out / in angles and self-loops

Besides the symmetric `bendDirection` + `bendAngle`, `<Step kind="bend">` accepts `outAngle` / `inAngle` for the start out-angle and end in-angle to draw asymmetric curves (TikZ `out=` / `in=`); `looseness` tunes the control-point distance (default ≈1). **out/in take precedence over `bendDirection`.** When `from == to` (same node / same coordinate) it degenerates into a **self-loop** — the prime use case for out/in (a state-machine self transition), which a symmetric bend cannot draw.

<ComponentPreview name="path-outin-loop" size="sm" />

#### Whole-path transform (rotate / scale)

`rotate` (degrees) / `scale` (a number for uniform, or `{ x, y }` for non-uniform) transform the whole path around its **bounding-box center**, sparing a wrapping `<Scope transforms>` for a single path. The order is a hard contract: path endpoints resolve first, arrow shrink is computed on the **untransformed** geometry, then everything is wrapped in a group — so arrow direction follows the transform correctly without misalignment.

<ComponentPreview name="path-transform" size="sm" />

#### Mid-path marks

`marks` places directional marks along the path: each `{ pos, mark }` has `pos ∈ [0, 1]` (out-of-range rejected by the schema), and `mark` is initially `{ kind: 'arrow', shape, ... }` (a registered arrow name + visual subset, see [Custom Arrows](/core/reference/extending/custom-arrow)), **oriented by the tangent at that point** (not a fixed direction). Lines, Béziers, and arcs are all tangent-oriented.

## API Reference

| Prop | Type | Default | Description |
| --- | --- | --- | --- |
| `id` | `string` | — | Optional stable id: lets other elements anchor to the path and serves as the attach point for [hydration](/core/reference/hydration) event binding (a path needs an `id` before you can bind `onClick` and friends) |
| `color` | `string` | — | Master color (TikZ `color=`); `stroke` / arrow / step label default to it unless individually set (follows the master color, not `stroke`) |
| `stroke` | `string` | `currentColor` | Stroke color |
| `strokeWidth` | `number` | `1` | Stroke width |
| `dashPattern` | `Array<number>` | ? | Dash/gap lengths in user units (e.g. `[4, 2]`) |
| `lineCap` | `'butt' \| 'round' \| 'square'` | `'butt'` | Stroke endpoint shape |
| `lineJoin` | `'miter' \| 'round' \| 'bevel'` | `'miter'` | Stroke corner shape |
| `thickness` | `'ultraThin' \| 'veryThin' \| 'thin' \| 'semithick' \| 'thick' \| 'veryThick' \| 'ultraThick'` | — | Semantic stroke preset (`thin = 1`); explicit `strokeWidth` always wins |
| `arrow` | `'none' \| '->' \| '<-' \| '<->'` | — | Path-level arrow direction (see [Arrow](/core/components/draw/arrow)) |
| `arrowDetail` | `ArrowDetail` object | — | Arrow visual config; fields documented on [Arrow](/core/components/draw/arrow) |
| `fill` | `string \| PaintSpec` | `'none'` | Fill of the closed region: any CSS color, or PaintSpec (gradient / pattern / image) |
| `fillRule` | `'nonzero' \| 'evenodd'` | `'nonzero'` | Fill rule; use `'evenodd'` for rings / holes |
| `opacity` | `number` | — | Whole-path opacity 0..1 (multiplies onto stroke and fill) |
| `fillOpacity` | `number` | — | Fill-only opacity 0..1 |
| `drawOpacity` | `number` | — | Stroke-only opacity 0..1 |
| `zIndex` | `number` (integer) | `0` | Explicit stacking order: higher draws on top; ties keep declaration order; effective only among siblings at the same level |
| `rotate` | `number` | — | Whole-path rotation in degrees (around the bounding-box center) |
| `scale` | `number \| { x, y }` | — | Whole-path scale (number = uniform / `{ x, y }` = non-uniform, around the bounding-box center) |
| `marks` | `Array<{ pos, mark }>` | — | Marks along the path: `pos∈[0,1]`, `mark.kind:'arrow'` + a registered arrow name, oriented by the tangent at that point |
| `children` | `Array<<Step>>` | — | At least 2; if the first isn't `move`, the compiler coerces it to `move`. `<Step>` supports `move` / `line` / `step` (folded) / `cycle` (close) / `curve` / `cubic` (Bezier) / `bend` (arc shorthand, with out/in angles and self-loops) / `arc` / `circlePath` / `ellipsePath` (circle / ellipse) / `rectangle` (diagonal rectangle); the IR also has `generator` (path generators, no JSX DSL) |

## Related

### Path vs Draw

| | `<Draw>` | `<Path>` + `<Step>` |
| --- | --- | --- |
| Form | Single-line `way` array | Container + child nodes |
| Expressiveness | Equivalent to Path (compile-expanded) | All Kernel capability |
| Use when | 90% of connection cases | Explicit Step sequence, future extensions |

`<Draw>` compiles down to `<Path>`; tutorials should lead with `<Draw>` — `<Path>` is for users who want the underlying construct.

- [Draw](/core/components/draw/overview): Sugar preferred for everyday connections
- [Step](/core/components/draw/step): Complete field and kind reference for `<Path>` children
