title: Scope
description: Local transform container — wraps a group of children and translates / rotates / scales them as a unit; mirrors TikZ `\begin{scope}[...]`.
---

`<Scope>` is retikz Kernel's fourth element (alongside `<Node>` / `<Path>` / `<Coordinate>`). It groups a set of IR children at the IR layer and applies a local transform chain to them; maps to TikZ `\begin{scope}[shift=..., rotate=..., scale=...] ... \end{scope}`.

## Usage

```tsx
import { Scope } from '@retikz/react';
```

```tsx
<Scope transforms={[{ kind: 'translate', x: 50, y: 0 }]}>
  {/* Node / Path / Coordinate / nested Scope */}
</Scope>
```

A bare `<Scope></Scope>` (no children, no transforms, no id) → compile emits no Scene primitive; any non-empty field → a `GroupPrim` is emitted, no empty-container overhead.

### When to use Scope

- **Move a cluster as a unit**: a group of nodes + paths form a cluster — moving the cluster only changes the scope's transform, not each child's `position`
- **Rotate a subdiagram**: rotate a subgraph around the scope's local origin; connections and labels follow automatically
- **Nested coordinate frames**: outer scope positions the cluster, inner scope positions children within the cluster — clearer than computing absolute coordinates by hand

### The 6 transform variants

The 4 translate variants mirror the 4 `<Node position>` shapes (Cartesian / polar / direction-relative / offset) one-to-one. At compile time they are all flattened to Cartesian translate before being lowered to Scene `GroupPrim`; `rotate` / `scale` pass through directly.

| kind | Fields | Semantics |
|---|---|---|
| `translate` | `{ x, y }` | Cartesian translation in user units |
| `polar-translate` | `{ origin?, angle, radius }` | Polar translation; `origin` accepts `id` / `[x, y]` / `PolarPosition`; defaults to origin point |
| `at-translate` | `{ direction, of, distance? }` | Place scope at `distance` from referent node along `direction`; `distance` defaults to `<TikZ nodeDistance>` |
| `offset-translate` | `{ of, offset? }` | Scope origin = referent global position + `[dx, dy]`; `offset` defaults to `[0, 0]` |
| `rotate` | `{ degrees, cx?, cy? }` | Rotate by degrees; `cx/cy` default to scope local origin |
| `scale` | `{ x, y? }` | Scale; `y` defaults to `x` (uniform) |

Array order is application order — the 0-th entry takes effect first (innermost), matching SVG transform list / Scene `GroupPrim`. `transforms: [translate, rotate]` means translate then rotate — same semantics as SVG `transform="translate(...) rotate(...)"`.

<Comparison target="tikz" title="Scope mapping">

`<Scope transforms={[{ kind: 'translate', x: 50, y: 0 }]}>...</Scope>` maps to TikZ `\begin{scope}[shift={(50,0)}] ... \end{scope}`: a local transform is pushed at the start of the subdiagram and popped at the end. TikZ stacks CM in the order options appear; retikz applies `transforms` in array index order, reusing the SVG / Scene convention.

</Comparison>

### Reference the scope as a whole

When `<Scope id="cluster">` is set, compile collects the axis-aligned **global bounding box** of all child layouts (nodes / coordinates / nested scope.id bboxes) at the end of Pass 1 and registers a synthetic `rectangle` `NodeLayout` named `cluster` into the **parent** namespace frame. From the outside, `cluster` behaves exactly like a normal rectangle node — paths can target `cluster.north` / `cluster.east` / `cluster.30`, and other nodes can use `cluster` as the referent for polar / direction / offset positioning.

<Comparison target="tikz" title="scope.id mapping">

`<Scope id="cluster">` maps to TikZ `\begin{scope}[local bounding box=cluster] ... \end{scope}`: pgf collects the axis-aligned bounding box of everything drawn inside and registers a rectangle node named `cluster`. retikz reuses the existing scope `id` field as the single entry point — no extra `boundingBox` flag — so the field stays a one-job thing (set id ⇒ get a referenceable bbox handle).

</Comparison>

External paths use `'cluster.<anchor>'` to hit the **whole** cluster's visual border instead of any single inner node:

<ComponentPreview name="scope-id-reference" size="md" />

- bbox endpoints (`.north` / `.east` / `.south-west` / `.30` / etc.) work exactly like a rectangle node — all 9 named anchors plus numeric angles are supported via the shared rect anchor path
- `'cluster'` with no suffix uses `boundaryPoint` clip toward the source (same behavior as a normal node id with no anchor)
- Path endpoints (move/line `to`) are **not** included in the bbox — only nodes / coordinates / nested scope.id bboxes contribute, matching TikZ `local bounding box`'s default

When nested, each scope.id keeps its own bbox; an outer scope.id includes everything inside, while an inner scope.id is independent — the outer bbox covers `A` plus everything in the `inner` subtree; `inner.bbox` only covers `B` and `C`:

<ComponentPreview name="scope-id-referent" size="md" />

Forward references fail to resolve, since the bbox is computed only after the scope subtree finishes.

### Relative positioning inside a scope

When a `<Node>` / `<Coordinate>` / `<Path>` `Step.to` inside a scope uses `PolarPosition` / `AtPosition` / `OffsetPosition`, retikz resolves it under the following rules:

1. **Referent is global**: the `polar.origin` / `at.of` / `offset.of` referent (node id / scope.id / coordinate literal) is resolved to its **global** coordinate
2. **Relative part is measured in the current scope's local frame**: polar's `(angle, radius)`, at's `(direction, distance)`, offset's `[dx, dy]` are all expressed in the **current node's enclosing scope's local coordinate system**
3. **The end result is projected back to global via the current scope's chain**: the referent's global point is first inversely projected to the current scope's local frame, the relative part is added, then the sum is forward-projected back to global

Math: `final = scopeTransform(inverseScopeTransform(referent_global) + relative_part)`

<Comparison target="tikz" title="Relative positioning inside a scope">

Maps to TikZ `\begin{scope}[rotate=90] \node at ($(hub) + (50:0)$) {orbit}; \end{scope}` — `(hub)` resolves to a global coordinate (unaffected by scope rotate), and `+ (50:0)` is measured in the current scope's local frame. retikz reuses this convention: referents are globally visible, while relative parts rotate / scale together with the scope.

</Comparison>

`hub` lives outside the scope (global origin); inside a `rotate(45°)` scope, `orbit` uses polar `{ origin: 'hub', angle: 0, radius: 80 }` — the relative part is local right-80, then projected by the scope rotate to a 45° down-right visual direction. The same polar written outside the scope (see `ref`) directly lands at hub's global right by 80:

<ComponentPreview name="scope-position-polar-in-rotate" size="md" />

`direction: 'right'` is the **current scope's local** right — after the scope rotates 90°, "right" visually becomes "down", and B lands below A by 60. The `B-ref` outside the scope sits at A's global right by 60:

<ComponentPreview name="scope-position-at-cross-scope" size="md" />

`offset: [40, 0]` is measured in scope local; after `scale(2)`, the visual offset becomes (80, 0) — B is to anchor's right by 80. `B-ref` outside the scope keeps the unscaled offset:

<ComponentPreview name="scope-position-offset-with-scale" size="md" />

Detailed rules:

- **Nested polar / nested offset**: the whole nested chain is measured in the **current node's enclosing scope's local frame** — the basis scope does not switch as you recurse
- **Cartesian `Position` literal**: inside a scope, `position={[10, 0]}` keeps v0.1 behavior — `[10, 0]` is measured in the current scope's local frame and projected to global via the scope chain
- **Scope translate does not change relative vectors**: translate is neither rotation nor scaling — `relative=(10, 0)` inside a `translate(50, 0)` scope keeps the same visual direction and length
- **Scope `scale=0`**: the inverse projection is undefined; retikz falls back to local origin (0, 0) to avoid NaN. Avoid zero scale in practice
- **Relative `Step.to` in a path**: same rules as `Node.position` — when path step.to is a polar / offset / Cartesian / `{ relative: [dx, dy] }` literal, the relative part is measured in the path's enclosing scope's local frame

## Namespace and isolation

By default a scope does **not** introduce a namespace boundary — child node / coordinate / nested scope.id ids all register into the current top frame; ids from multiple stacked scopes flow up to the `<TikZ>` root frame and outer paths can reference them directly.

<Comparison target="tikz" title="Default namespace behavior">

TikZ pgf's `\begin{scope}` does not introduce a local namespace — a `\node (foo) {...}` registered inside is globally visible, and outer code can `\draw (foo) -- ...` directly. retikz keeps this convention so v0.1 IR continues to work without changing id references after being wrapped in a scope; opt in to `localNamespace` only when subdiagram reuse or template-style isolation is needed.

</Comparison>

When subdiagram reuse or template-style isolation is needed, opt in with `localNamespace: true` so the scope pushes its own frame and child ids are isolated from the outside:

<ComponentPreview name="scope-local-namespace-basic" size="md" />

- **inside-out lookup**: when an inner path references an id string, the lookup walks the frame stack from top to bottom — inner code **can** see outer ids (shadowing), but outer code **cannot** see inner ids
- **Shadowing**: registering the same id in an inner frame is not a conflict and emits no warning; lookups hit the inner copy first, and the outer id becomes visible again once the frame is popped
- **scope.id always registers in the parent frame**: scope.id is the external handle and is **not** affected by its own `localNamespace`; outer paths can always reference `<Scope id="X">` as `X`, regardless of whether the scope uses local namespacing internally

Nested localNamespace forms a stack of namespace frames:

<ComponentPreview name="scope-local-namespace-nested" size="md" />

The innermost path references `L2-node` / `L1-node` / `root-anchor` — each resolves at the innermost / middle / root frame respectively; the stack search returns at the first match.

When every `localNamespace` layer registers the same id, inside-out lookup automatically shadows outer copies; same id across frames is not a duplicate and no warning is emitted:

<ComponentPreview name="scope-local-namespace-shadowing" size="md" />

## Duplicate id handling

When the same id is registered more than once within the same frame (any combination of node × node, node × coordinate, node × scope.id) compile emits a `DUPLICATE_NODE_ID` warning (collected via `CompileOptions.onWarn`) and **the later definition overrides the earlier one** (last-wins):

<ComponentPreview name="scope-duplicate-id-warn" size="md" />

- Compile does **not** throw — useful for LLM-generated / templated code where duplicates can be benign and you still want a usable output
- Every duplicate emits its own warning (not merged); each carries two IR locators (earlier and current) plus the current frame depth
- Same id across different frames (e.g. an outer id and a same-named id inside `<Scope localNamespace>`) is shadowing rather than a duplicate — **no warning** is emitted

> v0.1 behavior: duplicate ids threw. Starting from alpha.1, the policy is warn + last-wins, so v0.1 code keeps running unchanged; if you want strict behavior, detect `code === 'DUPLICATE_NODE_ID'` inside `onWarn` and re-throw yourself.

## API Reference

| Prop | Type | Default | Description |
| --- | --- | --- | --- |
| `transforms` | `Array<Transform>` | — | Local transform list, 6 variants; applied in array order (0-th first) |
| `children` | `ReactNode` | — | Scope children: `<Node>` / `<Path>` / `<Coordinate>` / nested `<Scope>` |
| `id` | `string` | — | Optional scope reference id; when set, the **axis-aligned global bounding box** of all child layouts is registered as a synthetic rectangle layout into the **parent frame**, so outer paths can reference it as `'cluster'` (bbox center after clip), `'cluster.<anchor>'` (9 named anchors), or `'cluster.<deg>'` (numeric angle). Other nodes can use scope.id as a `polar.origin` / `at.of` / `offset.of` referent (takes the bbox center) |
| `localNamespace` | `boolean` | `false` | Whether to create a local namespace: `true` pushes a child frame and isolates child ids from the outside (external code cannot see them; inner code can still inside-out reach outer ids); default false = global flat |
