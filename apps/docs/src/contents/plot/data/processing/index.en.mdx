[Data Model](/plot/data/model) declares what the fields **are**; this page is about getting real-world data **in** — swapping sources, parsing, and validating. All three assume a model is declared.

## Swapping data sources

Data stays out of the config (the IR), so one `<Plot>` config can run against different sources — as long as a model is declared (the model is the "portable" switch). There are three ways to bind a new source:

### Same names, same types

When field names / types match, just swap `data` and it re-renders straight away — not a line of config changes.

### Different names: `fieldMap`

When the data's field names don't line up with the config's logical names, supply a `fieldMap` (logical name → the actual path in the data); the config is untouched:

```tsx
// the data calls them period / amount, the config still says month / revenue
<Plot data={rows} model={[{ name: 'month', type: 'temporal' }, { name: 'revenue', type: 'continuous' }]}
      fieldMap={{ month: 'period', revenue: 'amount' }}>
  <LineMark x="month" y="revenue" order="month" />
</Plot>
```

`fieldMap` only renames (it doesn't transform values — computation belongs to transforms); it must be used with a model — without one there are no "logical fields" to map.

### Different storage formats, same type

A single `temporal` field reads correctly whether the data stores it as a `Date`, an ISO string, or an epoch number; a `continuous` field accepts a number or a well-formed numeric string (`"120"`). No extra config. Non-default storage formats (`'2024/01/01'`, epoch seconds, `'50%'`) go through the parsing below.

## Parsing non-default formats

Built-in parsing only accepts fixed shapes: `temporal` takes `Date` / epoch-ms / strict ISO, `continuous` takes numbers / strict numeric strings. Formats outside that table need you to spell out how to parse them — use a declarative `format` for common shapes, and `resolveField` for truly arbitrary logic.

### Declarative formats: `format`

For common non-default formats like `'2024/01/01'`, epoch seconds, or `'50%'`, you don't need a function — give the field a declarative `format`, which **enters the IR, is serializable, and an LLM can generate it**:

```tsx
<Plot data={rows} model={[
  { name: 'createdAt', type: 'temporal',  format: 'slashDate' },    // '2024/01/01' → time
  { name: 'ts',        type: 'temporal',  format: 'epochSeconds' }, // 1700000000 → ms
  { name: 'ratio',     type: 'continuous', format: 'percent' },     // '50%' → 0.5
]}>{/* … */}</Plot>
```

Available `format` values (each binds to one type):

| `format` | Type | Parsing |
| --- | --- | --- |
| `iso` | temporal | strict ISO (same as omitting) |
| `epochSeconds` | temporal | epoch seconds → ms |
| `epochMillis` | temporal | epoch milliseconds |
| `slashDate` | temporal | strict `YYYY/MM/DD` (at UTC midnight) |
| `numberString` | continuous | lenient numeric string (allows whitespace / thousands commas) |
| `percent` | continuous | percent string `'50%'` → 0.5 |

Rules:

- **`format` implies `type`**: with a `format` you can omit `type` — `{ name: 'ratio', format: 'percent' }` is continuous automatically, never inferred as categorical.
- **Must be compatible with `type`**: if the format's bound type conflicts with the `type` you wrote (e.g. `{ type: 'continuous', format: 'slashDate' }`), retikz **throws a clear error** rather than going silent.
- `slashDate` accepts only strict `YYYY/MM/DD`; ambiguous layouts like `DD/MM/YYYY` / `MM/DD/YYYY` are not accepted (so the same string can't parse to different dates in different environments).
- Parsing priority: `resolveField`'s `parse` > declarative `format` > built-in default.

### Custom parsing: `resolveField`

For **arbitrary formats** outside the table (a `'2024Q1'` quarter string, a custom separator) or to **force a field to a type** (a numeric enum as `categorical`), pass `resolveField` a function — it **does not enter the IR** (a runtime option, alongside `fieldMap`) and returns, per field name, a "type override + optional custom parser":

```tsx
<Plot data={rows} model={[{ name: 'quarter', type: 'temporal' }]}
  resolveField={(field) => {
    // a format outside the table: give type and parser together ('2024Q1' → first day of the quarter)
    if (field === 'quarter') return { type: 'temporal', parse: raw => {
      const [year, q] = String(raw).split('Q');
      return Date.UTC(Number(year), (Number(q) - 1) * 3, 1);
    } };
    if (field === 'statusCode') return { type: 'categorical' }; // numeric enum as a category (value kept)
    return undefined;                                            // everything else: model/inference + built-in parsing
  }}>
  {/* … */}
</Plot>
```

Rules:

- Returning `{ type }` overrides the field's type (higher priority than the model and inference); returning `{ parse }` uses a custom parser to turn the raw value into a canonical one (`temporal`/`continuous` return a number, `categorical` a string/number), and returning `undefined` skips that value.
- With **only `parse` and no `type`**, the type comes from the model declaration — so a bare `parse` **requires a model** (without one the type source is unclear, and it errors).
- `resolveField` **does not bypass strict mode**: with a model declared, referencing an unlisted field still errors — it only overrides type/parsing, it cannot turn a misspelled field into a valid one.
- Being a function, it is not serializable, so `resolveField` is the escape hatch for hand-written code; for a fully portable / LLM-friendly spec, prefer the model's declarative `type` / `format`.

## Validating dirty data

Document / heterogeneous data often has missing fields or mixed types across rows. retikz's policy is **skip, never fabricate, never silently draw wrong**:

| Situation | Behavior |
| --- | --- |
| A row is missing the field | that datum is skipped (normal for heterogeneous data, no error) |
| The leaf is an object / array (non-scalar) | that datum is skipped (objects never enter the chart) |
| Same type, different storage format | converted automatically (see "Swapping data sources" above) |
| The field is a `bigint` (e.g. a database int64) | converted to a number automatically (safe integers only; out-of-range is treated as invalid) |
| The path passes through an array (needs unwinding) | not auto-unwound yet — flatten in the data layer first |

The default "skip" can leave you staring at a blank chart. Turn on `validateData` to have retikz **sample-check** the binding and throw on problems, pointing at **which field and how many invalid / missing** (e.g. `field "x": 3/100 invalid`), turning "why is it empty" into a clear message:

```tsx
<Plot data={rows} model={[...]} validateData>
  {/* missing field / nothing convertible → compile-time error instead of a blank chart */}
</Plot>
```

For a stricter **full** check, pass `invalid="error"` — any invalid value (a missing field, an unconvertible value, an out-of-range bigint) in a field the chart actually uses throws immediately and names the field, rather than sampling:

```tsx
<Plot data={rows} model={[...]} invalid="error">
  {/* any participating field with an invalid value → error that names the field */}
</Plot>
```

`invalid` defaults to `'skip'` (the table's "skip" behavior): invalid points are dropped but the **whole row is kept**, still participating in stacking / sorting transforms; `'error'` only checks the fields the chart references, leaving other dirty columns alone.
