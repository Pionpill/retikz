title: Introduction
description: retikz's charting layer — a declarative Plot IR describes what to draw, data is supplied separately, and lowerPlots lowers both into core primitives.
---

`@retikz/plot` is a charting layer built on top of core. You declare a **Plot IR** that says what a chart is — its coordinate system, scales, and which field becomes a point or a line — while the data is supplied separately, and one function combines the two into something renderable.

It deliberately splits a chart into three parts, each with one job:

| Part | What it is | Responsibility |
| --- | --- | --- |
| **Plot IR** | Config object (JSON-serializable) | Decides how the chart looks: marks, scales, coordinate, field bindings |
| **Data** | External dataset, arbitrary JS | Provides the actual values; **never enters the IR**, so the IR does not grow with the data |
| **lowerPlots** | Fixed lowering function | Projects IR + data into core points / lines for the existing renderer |

`@retikz/plot-react` wraps these three into a single `<Plot>` component. The chart below draws five months of revenue as both a line and scatter points — same data, same scales, two stacked marks:

The composition DSL is the most direct way to write it: you only declare what to draw (which field becomes a line, which becomes points); scales and the coordinate system are inferred by `<Plot>`, and the data is a plain array of rows passed in separately:

```tsx
import { LineMark, Plot, PointMark } from '@retikz/plot-react';

const sales = [
  { month: 0, revenue: 10 },
  { month: 1, revenue: 14 },
  // ...
];

<Plot data={sales} width={360} height={200}>
  <LineMark x="month" y="revenue" order="month" />
  <PointMark x="month" y="revenue" />
</Plot>;
```

The JSX is just an authoring surface: `<Plot>` reads its mark children synchronously and assembles a normalized **Plot IR**. You can also hand-write or generate that IR (e.g. from a program / LLM) and render it through the `<Plot spec>` entry:

```tsx
const spec = {
  namespace: 'plot',
  type: 'plot',
  data: { ref: 'sales' }, // a reference only, no data
  scales: [
    { type: 'linear', name: '__x' },
    { type: 'linear', name: '__y' },
  ],
  coordinate: { type: 'cartesian2D', x: '__x', y: '__y' },
  marks: [
    { type: 'line', order: 'month', encoding: { x: { field: 'month' }, y: { field: 'revenue' } } },
    { type: 'point', encoding: { x: { field: 'month' }, y: { field: 'revenue' } } },
  ],
};

// when you already hold a full IR: use the spec entry + a named dataset
<Plot spec={spec} data={{ sales }} width={360} height={200} />;
```

The IR only carries `data: { ref: 'sales' }` (a name); the real data is passed in separately at render time — the same `spec` works with any dataset whose fields match. That is the point of decoupling IR from data: config describes the shape, data only fills in the content.

For server-side / build-time use, `@retikz/plot-vanilla`'s `renderPlot` returns an SVG string (zero DOM):

```ts
import { renderPlot } from '@retikz/plot-vanilla';

const svg = renderPlot(spec, { sales }, { width: 360, height: 200 }); // SSR: SVG string
```
