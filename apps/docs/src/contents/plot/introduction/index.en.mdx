`@retikz/plot` is a charting layer built on top of core. You declare a **Plot IR** that says what a chart is — its coordinate system, scales, and which field becomes a point or a line — while the data is supplied separately, and one function combines the two into something renderable.

It deliberately splits a chart into three parts, each with one job:

| Part | What it is | Responsibility |
| --- | --- | --- |
| **Plot IR** | Config object (JSON-serializable) | Decides how the chart looks: marks, scales, coordinate, field bindings |
| **Data** | External dataset, arbitrary JS | Provides the actual values; **never enters the IR**, so the IR does not grow with the data |
| **lowerPlots** | Fixed lowering function | Projects IR + data into core points / lines for the existing renderer |

`@retikz/plot-react` wraps these three into a single `<Plot>` component. The chart below draws five months of revenue as both a line and scatter points, with axes and gridlines added automatically — same data, same scales, two stacked marks:

The **composition DSL** is the most direct way to write it: you only declare what to draw (which field becomes a line, which becomes points); scales and the coordinate system are inferred by `<Plot>`, and the data is a plain array of rows pulled out into `line-scatter.data.ts`. The full code lives in the **Code** panel below the chart — the React source, the assembled **Plot IR**, the equivalent **Vanilla** builder code, and the data file are all there.

The JSX is just an authoring surface: `<Plot>` reads its mark children synchronously and assembles a normalized **Plot IR** (the **IR** tab in the panel). You can also go the other way — hand-write or generate that IR (e.g. from a program / LLM) and render it through the `<Plot spec={spec} data={{ sales }}>` entry. Either way, the IR carries only a reference name for the data (`data: { ref: 'sales' }`): the real data is passed in separately at render time, so the same spec works with any dataset whose fields match — that is the point of decoupling IR from data, config describes the shape and data only fills in the content.

For server-side / build-time use, `@retikz/plot-vanilla`'s `renderPlot(spec, data, options)` returns an SVG string (zero DOM) — the **Vanilla** tab in the panel.

## Axes and gridlines are drawn by default

The ticks, axis lines, and grid in the chart above are all automatic — you only wrote marks, and `<Plot>` adds x / y axes plus a horizontal grid by default. To customize, add `<Axis>` children; once you write any `<Axis>`, you get **exactly what you list** and no defaults are filled in:

```tsx
<Plot data={sales} width={360} height={200}>
  <LineMark x="month" y="revenue" order="month" />
  <Axis dimension="x" tickCount={5} />
  <Axis dimension="y" grid />
</Plot>
```

`<Axis>` switches: `tickCount` (target number of ticks), `grid` (whether this axis draws gridlines aligned to its ticks), and `tickLabels={false}` (keep the axis line and ticks, drop the numbers). Grid is a **property of an axis**, not a separate component — an x-axis grid is vertical lines, a y-axis grid is horizontal lines, and the two overlaid is what reads as a "grid".

When you want a bare plot with no axes at all, pass `bare` to `<Plot>`:

```tsx
<Plot data={sales} width={360} height={200} bare>
  <LineMark x="month" y="revenue" order="month" />
</Plot>
```

`width × height` is the **outer box** including axes and tick labels: once the axes and labels claim the edges, the remaining rectangle is the actual plotting area. Tune the label font size and surrounding whitespace with `fontSize` and `margin`.

## Data can come from anywhere

The data above is hard-coded, but in the real world it usually comes from elsewhere — a backend endpoint, a public API, a CSV. Because the data **never enters the IR** and is only passed in as plain rows at render time, fetching a large dataset from an external source and charting it works exactly like local data: fetch it first, then hand it to `<Plot data>` the same way.

The fetching logic (`fetch` + state) is pulled out into a sibling `.data` file just like hard-coded data, so the demo file stays simple and only consumes it — on the React side it is wrapped as a hook (`useCarPerformance()` in `line-scatter-fetch.data.ts`, shown as the data file in the Code panel). The chart below pulls the classic "cars" dataset (~400 rows) from [vega-datasets](https://github.com/vega/vega-datasets) live over a CDN and draws horsepower vs. fuel economy as a scatter; the charting lines are identical to the local-data case — where the data came from is transparent to `<Plot>`:
