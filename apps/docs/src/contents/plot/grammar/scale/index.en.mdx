---

A scale decides how a data value becomes a visual quantity. Position channels (`x` / `y`) turn into canvas coordinates, the `size` channel turns into a radius — each goes through a scale. `<Plot>` auto-picks a scale from the field type, so common charts need none by hand; reach for `scaleX` / `scaleY` when you want a log or square-root axis.

## Scale types

| Type | For | Notes |
| --- | --- | --- |
| `linear` | Continuous numbers | Evenly-spaced linear mapping (default) |
| `log` | Positive values across magnitudes | Logarithmic; point / line only |
| `sqrt` | Area-perceptual | Square-root; point / line only (the `size` channel's default) |
| `pow` | Custom power | Power mapping (needs `exponent`; via a full spec) |
| `time` | Dates / time | Time axis with human-readable tick boundaries |
| `band` / `point` | Categories | Equal bands (bars) / evenly-spaced points (categorical lines) |
| `ordinal` | Category → color | One discrete color per category (multi-series coloring) |
| `sequential` | Continuous value → color | One-directional color band (low→high), for continuous / temporal fields |
| `diverging` | Quantity with a midpoint → color | Two-sided hues, pale center; profit/loss or deviation-from-mean |
| `quantize` | Continuous value → discrete bins | Equal-width bins, one color each (choropleth) |
| `threshold` | Continuous value → discrete bins | Custom breakpoints (business thresholds / alerts) |
| `quantile` | Continuous value → discrete bins | Data-quantile bins, skew-resistant |

## Linear vs log

When data spans several orders of magnitude, a linear axis crushes the small values against the bottom. A log axis (`scaleY="log"`) gives every magnitude equal height, so small values stay legible. Both charts use the same yearly user counts.

## Square root

`scaleY="sqrt"` maps by square root, taming the visual exaggeration of large values; it is also the default scale for the `size` channel (point radius), so a dot's **area** stays proportional to the data.

## Point / line only

`log` / `pow` / `sqrt` apply only to points and lines. Bars and areas start from a 0 baseline, and 0 is undefined on a log / power axis — pairing a non-linear scale with a bar / area errors out and tells you to use point / line instead.

## Color scales

The scales above map values to **position / length / radius**; color is a different kind of output. Position scales are consumed by the `x` / `y` channels, color scales by the `color` channel — the two never interfere. Explaining a color scale to the reader is the job of the [legend](/plot/components/legend).

| Type | For | Output |
| --- | --- | --- |
| `ordinal` | Categorical field | One discrete color per category (multi-series, categorical coloring) |
| `sequential` | Continuous / temporal field | A one-directional band: low→high along one gradient (the workhorse for continuous color) |
| `diverging` | Quantity with a meaningful midpoint | Two-sided hues, pale center: profit/loss, deviation from mean |

Sequential and diverging color comes from a **named scheme** — a preset set of color band names that is serializable and AI-generatable. Sequential schemes include `viridis` (default), `blues`, `greens`, `magma`; diverging schemes include `rdbu` (default), `brbg`, `spectral`. Use `range` to override a scheme with custom endpoint colors.

Bind `color` to a continuous numeric field and `<Plot>` derives a sequential scale (viridis by default); a categorical field gets an ordinal scale. Continuous color encoding works only on per-datum marks (point / bar / sector); a line or area colors the whole path at once, so its color is not constant along the path and binding a continuous `color` errors.

Diverging and custom schemes are configured through the full spec scales for now." />

## Discretization / binned color

A sequential scale is a **continuous** band (smooth gradient). But many cases want a continuous quantity cut into **a few discrete bins, one color each** — graded maps, threshold alerts, skew-resistant binning. These three needs map to three discretization scales, differing only in **how the bins are cut**:

| Type | Cut by | For |
| --- | --- | --- |
| `quantize` | domain split into `count` equal-width bins | choropleth / evenly-distributed continuous values |
| `threshold` | user-given breakpoints (`breakpoints`) | business thresholds / alerts (`<60` red, `60–80` yellow, `≥80` green) |
| `quantile` | data quantiles, roughly equal sample count per bin | skewed data / outlier resistance |

Each bin's color reuses the named schemes above — N discrete colors are sampled at evenly spaced points along one continuous band (e.g. `scheme: 'blues'` sampled into 5 bins); or give a `range` array to override the scheme, where the array length is the bin count.

<p className="text-sm text-muted-foreground">quantize, 5 bins: continuous density cut into 5 equal-width bins, each taking one sampled color off blues.</p>

A few constraints to keep in mind:

- **threshold breakpoints must be strictly ascending**; bin count = breakpoints + 1, so a `range` must have length `breakpoints.length + 1` (e.g. `[60, 80]` pairs with 3 colors).
- **`count` defaults to 5** for quantize / quantile; when a `range` is given the bin count is the range length and `count` is ignored.
- **quantile takes no explicit numeric domain** — the bin boundaries come from the data, so passing a domain errors.
- Like continuous color, discretized color works **only on per-datum marks** (point / bar / sector); a line or area colors the whole path at once, so binding a discretization color errors.

## Setting it explicitly

`scaleX` / `scaleY` on `<Plot>` are shorthands that override the auto-pick. For finer control — log base, power exponent — write the full spec `scales` (see Reference).
