---

一张图分两半：**配置**（画什么——mark / 坐标系 / 比例尺）和**数据**（画哪些值）。retikz 把两者分开——数据**不进**图的配置（IR），编译时单独传入。这让同一份配置能复用到不同数据上。

最简单的用法：把数据行数组交给 `<Plot data>`，mark 用字段名引用列。

```tsx
<Plot data={rows}>
  <LineMark x="month" y="revenue" order="month" />
</Plot>
```

字段名是**路径**，所以嵌套数据（如 MongoDB 文档 / 接口返回的 JSON）也能直接用：`x="user.address.city"` 会逐级取到叶子值。叶子必须是标量（字符串 / 数字 / 布尔 / 时间），不能是对象或数组。

## 数据模型：声明字段类型

光有数据，retikz 不知道 `month` 是时间还是普通字符串、`revenue` 是连续数值还是分类。给一份 **model**（字段名 + 类型），它就全知道了——比例尺怎么选、坐标轴怎么格式化、校验怎么做，都由类型驱动。

```tsx
<Plot data={sales} model={[
  { name: 'month',   type: 'temporal' },
  { name: 'revenue', type: 'continuous' },
]}>
  <LineMark x="month" y="revenue" order="month" />
</Plot>
```

字段类型（三类测量种类）：

| 类型 | 含义 | 例子 |
| --- | --- | --- |
| `continuous` | 连续可度量数值 | 销量、价格、温度、占比 |
| `categorical` | 离散类别 | 国家、颜色名、评级 |
| `temporal` | 日期 / 时间 | 日期、时间戳 |

**声明 model = 严格模式**：mark / transform 引用的每个数据字段都必须在 model 里列出，拼错字段名会**立刻报错**（而不是画出一张空图）。**不声明 model**：retikz 从数据抽样推断类型（数值→continuous、ISO 日期串 / `Date`→temporal、其余字符串→categorical），适合快速试画。

**`type` 可选——部分声明**：字段类型可省略，只写 `{ name }`。这样该字段照样进严格模式 + 可移植契约（靠名字），但**类型留给数据推断**。适合「大部分字段一目了然、只有少数要点名」——只给数值枚举、奇葩日期格式这类需纠偏的字段写 `type`，其余留空：

```tsx
<Plot data={sales} model={[
  { name: 'createdAt',  type: 'temporal' },    // 特殊格式，点名
  { name: 'statusCode', type: 'categorical' }, // 数值枚举，点名
  { name: 'revenue' },                          // 仅 name → 推断 continuous
]}>
  {/* … */}
</Plot>
```

省略 `type` 的字段需到 lowering 时看数据才能定类型——要让 spec 不看数据就完全自描述（LLM 友好 / 完全可移植），还是把 `type` 填全。

## 类型驱动的比例尺

声明了 model，位置通道（x / y）**不必再手写比例尺**——retikz 按字段类型自动选：

| 字段类型 | 自动选用的位置比例尺 |
| --- | --- |
| `continuous` | linear（线性） |
| `temporal` | time（时间轴，自动日期刻度） |
| `categorical` | band（分类带） |

需要覆盖时，显式声明的比例尺**永远优先**于自动派生。若显式比例尺与字段类型明显冲突（如把 `categorical` 字段配到线性轴、把 `temporal` 配到分类带），retikz 会**报清晰错误**而不是画出无意义的图。

## 分类顺序：`order`

`categorical` 字段默认按**数据出现顺序**排类别（轴、图例都跟着这个序）。想控制顺序——比如尺码该按 `S → M → L → XL`，而不是数据里碰巧出现的次序——给字段一个 `order`：

```tsx
<Plot data={rows} model={[
  { name: 'size',  type: 'categorical', order: ['S', 'M', 'L', 'XL'] }, // 显式顺序
  { name: 'grade', type: 'categorical', order: 'ascending' },           // 升序排序
  { name: 'city',  type: 'categorical' },                               // 默认：按出现序
]}>{/* … */}</Plot>
```

| `order` | 效果 |
| --- | --- |
| 不写 / `'data'` | 按数据出现序（默认） |
| `'ascending'` / `'descending'` | 对类别排序：全数值按数值比，否则按字符串 |
| `Array`（如 `['S','M','L']`） | 以数组为顺序；数据里有、数组没有的类别追加到末尾 |

规则：

- **挂在字段上，位置与颜色同序**：同一字段既作 x 轴（band）又作颜色（ordinal）时，两处类别顺序一致。
- **只对 categorical 有效**：给 `continuous` / `temporal` 字段写 `order` 会**报错**（顺序对连续值 / 时间没有意义）。
- **显式比例尺 domain 优先**：若比例尺自己写了 `domain`，以它为准，`order` 被忽略。
- 写了**非默认** `order`（排序或显式数组）就表示「这个分类有序」——retikz 三类字段里没有「有序类别」专门类型，有序性靠这个参数表达。

## 换数据源：同一张图，三种对接方式

因为数据不进配置，同一份 `<Plot>` 配置能换着数据源用。前提是声明了 model（model 就是「可移植」的开关）：

1. **同名同类型** —— 直接换 `data`，字段名 / 类型一致即可重新出图。
2. **字段名不同** —— 给 `fieldMap`（逻辑名 → 数据里的实际路径），配置不动：

   ```tsx
   // 数据里叫 period / amount，配置里仍用 month / revenue
   <Plot data={rows} model={[{ name: 'month', type: 'temporal' }, { name: 'revenue', type: 'continuous' }]}
         fieldMap={{ month: 'period', revenue: 'amount' }}>
     <LineMark x="month" y="revenue" order="month" />
   </Plot>
   ```

3. **存储格式不同但类型相同** —— 自动转换。同一个 `temporal` 字段，数据里是 `Date` 对象、ISO 字符串还是毫秒时间戳，都能正确读；`continuous` 字段是数字还是规范的数字串（`"120"`）也都认。无需任何额外配置。

`fieldMap` 只做改名（值不变换，计算交给 transform）；它必须配合 model 使用——没有 model 就没有「逻辑字段」可映射。

## 声明式格式：`format`

内置解析只认固定形态（`temporal` 收 `Date` / 毫秒 / 严格 ISO，`continuous` 收数字 / 严格数字串）。碰上 `'2024/01/01'`、epoch 秒、`'50%'` 这类常见的非默认格式，不必动用函数——给字段一个声明式 `format`，它**进 IR、可序列化、LLM 也能生成**：

```tsx
<Plot data={rows} model={[
  { name: 'createdAt', type: 'temporal',  format: 'slashDate' },    // '2024/01/01' → 时间
  { name: 'ts',        type: 'temporal',  format: 'epochSeconds' }, // 1700000000 → 毫秒
  { name: 'ratio',     type: 'continuous', format: 'percent' },     // '50%' → 0.5
]}>{/* … */}</Plot>
```

可选的 `format`（每个绑定一种类型）：

| `format` | 适用类型 | 解析 |
| --- | --- | --- |
| `iso` | temporal | 严格 ISO（等价不写） |
| `epochSeconds` | temporal | epoch 秒 → 毫秒 |
| `epochMillis` | temporal | epoch 毫秒 |
| `slashDate` | temporal | 严格 `YYYY/MM/DD`（按 UTC 零点） |
| `numberString` | continuous | 宽松数字串（容空白 / 千分位逗号） |
| `percent` | continuous | 百分比串 `'50%'` → 0.5 |

规则：

- **format 蕴含 type**：写了 `format` 就可省略 `type`——`{ name: 'ratio', format: 'percent' }` 自动当 continuous，不会被推断成分类。
- **必须与 type 兼容**：format 绑定的类型与你写的 `type` 冲突（如 `{ type: 'continuous', format: 'slashDate' }`）会**报清晰错误**，不静默。
- `slashDate` 只认严格 `YYYY/MM/DD`；`DD/MM/YYYY`、`MM/DD/YYYY` 这类有地区歧义的布局不收（避免同一串在不同环境解析成不同日期）。
- 解析优先级：`resolveField` 的 `parse` > 声明式 `format` > 内置默认。常见格式用 `format`，真正任意的逻辑才落到下面的 `resolveField`。

## 自定义解析：`resolveField`

声明式 `format` 覆盖了常见格式；碰上词表外的**任意格式**（如 `'2024Q1'` 季度串、自定义分隔符）或想**强制某字段当某类型**（数值枚举当 `categorical`）时，给 `resolveField` 一个函数——它**不进 IR**（是运行时选项，和 `fieldMap` 同层），按字段名返回「类型覆盖 + 可选自定义解析器」：

```tsx
<Plot data={rows} model={[{ name: 'quarter', type: 'temporal' }]}
  resolveField={(field) => {
    // 词表外的格式：类型 + 解析一起给（'2024Q1' → 季度首日）
    if (field === 'quarter') return { type: 'temporal', parse: raw => {
      const [year, q] = String(raw).split('Q');
      return Date.UTC(Number(year), (Number(q) - 1) * 3, 1);
    } };
    if (field === 'statusCode') return { type: 'categorical' }; // 数值枚举当类别（值原样）
    return undefined;                                            // 其余走 model/推断 + 内置解析
  }}>
  {/* … */}
</Plot>
```

规则：

- 返回 `{ type }` 覆盖该字段类型（优先级高于 model 与推断）；返回 `{ parse }` 用自定义解析器把原始值转成规范值（`temporal`/`continuous` 返数字、`categorical` 返字符串/数字），返 `undefined` 跳过该值。
- **只给 `parse` 不给 `type`** 时，类型沿用 model 声明——所以 `parse` 单独出现**必须有 model**（无 model 时类型来源不清，会报错）。
- `resolveField` **不绕过 strict**：声明了 model，引用未列出的字段照样报错——它只覆盖类型/解析，不能把拼错字段洗成合法字段。
- 因为是函数、不可序列化，`resolveField` 适合手写代码的逃生舱；要 spec 完全可移植 / LLM 友好，优先用 model 的声明式 `type` / `format`。

## 数据有问题时

文档型 / 异构数据常有缺字段、类型不一的行。retikz 的处理是**跳过、不伪造、不静默画错**：

| 情形 | 行为 |
| --- | --- |
| 某行缺这个字段 | 跳过该数据点（异构数据常态，不报错） |
| 叶子是对象 / 数组（非标量） | 跳过该数据点（绝不把对象塞进图） |
| 同类型不同存储格式 | 自动转换（见上「换数据源」第 3 条） |
| 字段是 `bigint`（如数据库 int64） | 自动转 number（仅安全整数；超出范围按非法处理） |
| 路径中途是数组（需展开） | 暂不支持自动展开，请先在数据层 flatten |

默认「跳过」可能让你对着一张空图摸不着头脑。打开 `validateData` 让 retikz **抽样**校验、有问题直接报错，并指出**哪个字段、多少非法 / 缺失**（如 `field "x": 3/100 invalid`），把「为什么图是空的」变成明确提示：

```tsx
<Plot data={rows} model={[...]} validateData>
  {/* 字段缺失 / 全部无法转换 → 编译期报错，而非空图 */}
</Plot>
```

要更严的**全量**校验，给 `invalid="error"`——图里实际用到的字段里任一非法值（含缺字段、无法转换、超范围 bigint）都立刻报错并指明字段，而不是抽样：

```tsx
<Plot data={rows} model={[...]} invalid="error">
  {/* 任一参与字段有非法值 → 报错并指明字段 */}
</Plot>
```

`invalid` 默认是 `'skip'`（上表的「跳过」行为）：非法点跳过、但**整行仍保留**，照常参与 stack / 排序等数据变换；`'error'` 只检查图里实际引用的字段，不波及其它脏列。
