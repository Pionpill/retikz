For readers who want to understand how retikz works, contribute code, or write an adapter for a new framework.

The official repo currently ships five core packages:

| Package | Role | Status |
| --- | --- | --- |
| [`@retikz/math`](https://github.com/Pionpill/retikz/tree/main/packages/core/math) | Zero-dependency pure computational geometry (vectors / affine / arc / intersections / circles / hull); the compute base core depends on | Officially maintained |
| [`@retikz/core`](https://github.com/Pionpill/retikz/tree/main/packages/core/core) | IR schemas, geometry, parsers, and the Scene compiler; zero React, zero DOM | Officially maintained |
| [`@retikz/render`](https://github.com/Pionpill/retikz/tree/main/packages/core/render) | Renderer-neutral Scene output adapters: SVG descriptors/strings, Canvas 2D, hydration helpers, and animation runtime utilities | Officially maintained |
| [`@retikz/react`](https://github.com/Pionpill/retikz/tree/main/packages/core/react) | React adapter: JSX components that compile to IR, then render through SVG or Canvas | Officially maintained |
| [`@retikz/vanilla`](https://github.com/Pionpill/retikz/tree/main/packages/core/vanilla) | Framework-free runtime: builder functions, SVG/Canvas mounting, hydration, animation controls, and SVG string output | Officially maintained |

Vue / Svelte / Solid / Native adapters are still future or community surfaces, but the IR remains universal: a new framework adapter can reuse `@retikz/core` and most renderer work from `@retikz/render`, then supply its own component bridge and text measurer where needed.

## @retikz/core

Zero-React, zero-DOM. `zod` is the only runtime dependency.

### Package Layout

| Module | Responsibility |
| --- | --- |
| [`ir/`](https://github.com/Pionpill/retikz/tree/main/packages/core/core/src/ir) | IR schema (zod) — position / node / coordinate / path / step / scope / paint / clip / scene, all defined with zod |
| [`geometry/`](https://github.com/Pionpill/retikz/tree/main/packages/core/core/src/geometry) | Shape / boundary geometry — rect / circle / ellipse / diamond / polar / contour / bend / segment; pure compute primitives (vectors / affine / arc / intersections / circles / hull) sank to [`@retikz/math`](https://github.com/Pionpill/retikz/tree/main/packages/core/math), re-exported by core |
| [`primitive/`](https://github.com/Pionpill/retikz/tree/main/packages/core/core/src/primitive) | Scene data model — render-target-agnostic primitives, resources, markers, and clipping |
| [`compile/`](https://github.com/Pionpill/retikz/tree/main/packages/core/core/src/compile) | IR → Scene compiler — node layout, path resolution, style cascade, resource collection, layout |
| [`parsers/`](https://github.com/Pionpill/retikz/tree/main/packages/core/core/src/parsers) | DSL → IR parsers — currently the Sugar `way` array and `target` strings |
| [`shapes/`](https://github.com/Pionpill/retikz/tree/main/packages/core/core/src/shapes) / [`arrows/`](https://github.com/Pionpill/retikz/tree/main/packages/core/core/src/arrows) / [`patterns/`](https://github.com/Pionpill/retikz/tree/main/packages/core/core/src/patterns) / [`path-generators/`](https://github.com/Pionpill/retikz/tree/main/packages/core/core/src/path-generators) | v0.2 extension surfaces — custom shapes, arrows, pattern motifs, and path generators |
| [`index.ts`](https://github.com/Pionpill/retikz/blob/main/packages/core/core/src/index.ts) | Public API surface — every adapter must import from this file only |

### Recommended Reading Order

Walk inward from both ends: input contract (IR) first, then output contract (Scene), then the `compile` step that bridges them.

1. **[`src/index.ts`](https://github.com/Pionpill/retikz/blob/main/packages/core/core/src/index.ts) — Public API**<br />
   Skim the named re-exports. What `@retikz/core` exposes (and doesn't) lives here; every adapter must import from this single file.

2. **[`ir/scene.ts`](https://github.com/Pionpill/retikz/blob/main/packages/core/core/src/ir/scene.ts) — IR root**<br />
   `SceneSchema = { version, type: 'scene', children: ChildSchema[] }`; `ChildSchema` is a discriminated union of node / path / coordinate / scope.

3. **[`ir/node.ts`](https://github.com/Pionpill/retikz/blob/main/packages/core/core/src/ir/node.ts) + [`ir/path/`](https://github.com/Pionpill/retikz/tree/main/packages/core/core/src/ir/path) + [`ir/position/`](https://github.com/Pionpill/retikz/tree/main/packages/core/core/src/ir/position) — IR fields**<br />
   Read the zod schemas with their `.describe(...)` text. Each `.describe` is the LLM tool-use contract — schema and field semantics in one place.

4. **[`primitive/scene.ts`](https://github.com/Pionpill/retikz/blob/main/packages/core/core/src/primitive/scene.ts) + the sibling prim files — Scene output**<br />
   `Scene = { primitives, layout, resources? }`. Primitives and resources are the render-target-agnostic common subset every adapter consumes; SVG-only / Canvas-only features are forbidden.

5. **Geometric foundation**<br />
   Pure functions + plain data, no classes. The pure compute primitives (vectors / affine / `arc` / intersections / circles / hull) live in the standalone [`@retikz/math`](https://github.com/Pionpill/retikz/tree/main/packages/core/math) package, re-exported by core; core's own [`geometry/`](https://github.com/Pionpill/retikz/tree/main/packages/core/core/src/geometry) keeps the business-flavored shape / boundary geometry ([`rect`](https://github.com/Pionpill/retikz/blob/main/packages/core/core/src/geometry/rect.ts) / [`polar`](https://github.com/Pionpill/retikz/blob/main/packages/core/core/src/geometry/polar.ts) / [`contour`](https://github.com/Pionpill/retikz/blob/main/packages/core/core/src/geometry/contour.ts), …) used during path compilation.

6. **[`compile/compile.ts`](https://github.com/Pionpill/retikz/blob/main/packages/core/core/src/compile/compile.ts) — Compiler entry**<br />
   `compileToScene(ir, options): Scene`. The main flow recursively processes children, lays out nodes and scopes, resolves paths, and collects paint / clip / marker resources. Reading it shows how an IR becomes a Scene.

7. **[`compile/node.ts`](https://github.com/Pionpill/retikz/blob/main/packages/core/core/src/compile/node.ts) / [`path/`](https://github.com/Pionpill/retikz/tree/main/packages/core/core/src/compile/path) / [`position.ts`](https://github.com/Pionpill/retikz/blob/main/packages/core/core/src/compile/position.ts) — Sub-compilers**<br />
   Drill in as needed: node layout / path stepping / position resolution. [`text-metrics.ts`](https://github.com/Pionpill/retikz/blob/main/packages/core/core/src/compile/text-metrics.ts) ships a fallback measurer; adapters can inject a more accurate one.

8. **[`parsers/`](https://github.com/Pionpill/retikz/tree/main/packages/core/core/src/parsers) — Sugar entry (optional)**<br />
   [`parseWay`](https://github.com/Pionpill/retikz/blob/main/packages/core/core/src/parsers/way.ts) / [`parseTargetSugar`](https://github.com/Pionpill/retikz/blob/main/packages/core/core/src/parsers/target-sugar.ts) translate `<Draw way={[...]}>` and similar Sugar forms into IR. `@retikz/react` calls them inside its Sugar components.

9. **Registry extension surfaces (optional)**<br />
   [`shapes/`](https://github.com/Pionpill/retikz/tree/main/packages/core/core/src/shapes) / [`arrows/`](https://github.com/Pionpill/retikz/tree/main/packages/core/core/src/arrows) / [`patterns/`](https://github.com/Pionpill/retikz/tree/main/packages/core/core/src/patterns) / [`path-generators/`](https://github.com/Pionpill/retikz/tree/main/packages/core/core/src/path-generators): third-party packages should extend here instead of modifying core enums.

### Constraints to Keep in Mind

Keep these constraints in mind when reading core; they explain why some implementation paths go through an extra layer instead of depending directly on the browser or React:

- **Zero React, zero DOM** — core must not `import 'react'` / `import 'react-dom'`; browser-specific capabilities (text measurement, events) are injected by the adapter.
- **IR data is 100% JSON-serializable** — no functions, class instances, refs, or Symbols in the IR schema.
- **Geometry uses pure functions + plain data only** — no classes; function bundles are exposed as `const ns = { method1, method2 }` namespaces.
- **`compileToScene` is a pure function** — same IR + same options ⇒ identical Scene; no `Math.random()`, `Date.now()`, or module-level mutable state.
- **Runtime dependency allowlist** — `zod` only; new runtime deps must be justified in the PR description.

## @retikz/react

`peerDependencies: react / react-dom`. The whole package does three things: turn JSX into an IR, call `compileToScene` to obtain a Scene, and render the Scene into SVG elements.

### Package Layout

| Module | Responsibility |
| --- | --- |
| [`kernel/`](https://github.com/Pionpill/retikz/tree/main/packages/core/react/src/kernel) | Kernel JSX components — `<Layout>` / `<Node>` / `<Path>` / `<Step>` / `<Text>` / `<Coordinate>` / `<Scope>`, plus `builder` (JSX → IR) and `unbuilder` (IR → JSX) bridges |
| [`sugar/`](https://github.com/Pionpill/retikz/tree/main/packages/core/react/src/sugar) | Sugar JSX components — `<Draw way={[...]}>`, `<EdgeLabel>`, and shape sugar; expand into Kernel at compile time, introduce no new capabilities |
| [`render/`](https://github.com/Pionpill/retikz/tree/main/packages/core/react/src/render) | Scene → SVG elements — `renderPrim`, `paintDefs`, `clipDefs`, `arrowMarkers`, `browser-measurer`, SVG viewBox formatting |
| [`index.ts`](https://github.com/Pionpill/retikz/blob/main/packages/core/react/src/index.ts) | Public API surface — Kernel + Sugar components and the `convertReactNodeToIR` / `convertIRToReactNode` bridges |

### Recommended Reading Order

1. **[`src/index.ts`](https://github.com/Pionpill/retikz/blob/main/packages/core/react/src/index.ts) — Public API**<br />
   See which Kernel / Sugar components, types, and bridges are exposed. `WayDSL` and related constants from core are re-exported here so React users can import from a single package.

2. **[`kernel/Layout.tsx`](https://github.com/Pionpill/retikz/blob/main/packages/core/react/src/kernel/Layout.tsx) — Top-level orchestrator**<br />
   `<Layout>` is the entry point of the whole pipeline: children → `buildIR` → `compileToScene` → `renderPrim` → `<svg>`. One file gives you the adapter's overall shape.

3. **[`kernel/builder.ts`](https://github.com/Pionpill/retikz/blob/main/packages/core/react/src/kernel/builder.ts) — JSX → IR**<br />
   Dispatches by child `displayName` to construct IRNode / IRPath / IRStep / IRScope, etc. The companion [`unbuilder.ts`](https://github.com/Pionpill/retikz/blob/main/packages/core/react/src/kernel/unbuilder.ts) goes the other way (IR → React element tree), useful for editor / AI workflows that "feed IR, render components".

4. **[`kernel/Node.tsx`](https://github.com/Pionpill/retikz/blob/main/packages/core/react/src/kernel/Node.tsx) / [`Path.tsx`](https://github.com/Pionpill/retikz/blob/main/packages/core/react/src/kernel/Path.tsx) / [`Step.tsx`](https://github.com/Pionpill/retikz/blob/main/packages/core/react/src/kernel/Step.tsx) etc. — Kernel components**<br />
   These components don't render anything themselves — they only declare a `displayName` and hand props to `builder`. Read one and you've read them all.

5. **[`render/renderPrim.tsx`](https://github.com/Pionpill/retikz/blob/main/packages/core/react/src/render/renderPrim.tsx) — Scene → SVG element**<br />
   Dispatches by primitive `type` to `<rect>` / `<ellipse>` / `<text>` / `<path>` / `<g>`; paint / clip / marker resources are materialized by sibling defs components. This layer is the main rewrite surface when porting to another framework.

6. **[`render/browser-measurer.ts`](https://github.com/Pionpill/retikz/blob/main/packages/core/react/src/render/browser-measurer.ts) — Text measurer injection**<br />
   A canvas-`measureText`-based `TextMeasurer`, injected into `compileToScene` via `CompileOptions.measureText`. When there's no document (SSR), it falls back to core's `fallbackMeasurer` automatically.

7. **[`sugar/Draw.tsx`](https://github.com/Pionpill/retikz/blob/main/packages/core/react/src/sugar/Draw.tsx) — Sugar expansion**<br />
   `<Draw way={[...]}>` internally calls [`parseWay`](https://github.com/Pionpill/retikz/blob/main/packages/core/core/src/parsers/way.ts) (from core) to obtain a step array, then renders Kernel `<Path><Step/></Path>`. Sugar introduces no new capabilities; everything expands to Kernel.

8. **Shape components in [`sugar/`](https://github.com/Pionpill/retikz/tree/main/packages/core/react/src/sugar) (optional)**<br />
   `<Circle>` / `<Ellipse>` / `<Arc>` / `<Sector>` / `<Rectangle>` / `<Grid>` / `<RegularPolygon>` / `<Star>` are Sugar too: they expand common geometry idioms into equivalent `<Path>` declarations.

### Mapping when writing a new adapter

If you're building `@retikz/vue` / `@retikz/svelte` / etc., the React adapter maps over like this:

| In React | Your new adapter needs |
| --- | --- |
| `kernel/*.tsx` components (declare `displayName`) | Equivalent component types in your framework (with a recognizable marker) |
| `kernel/builder.ts` (JSX → IR) | A children / vnode → IR bridge for your framework |
| `render/renderPrim.tsx` + defs components | Mapping Scene primitives and resources to your framework's native elements |
| `render/browser-measurer.ts` | A measurer for your environment (browser / SSR / Native) — implement `TextMeasurer` |

`@retikz/core`'s IR, `compileToScene`, geometry, and Scene primitive types are reused unchanged — no need to rewrite them.
