---
title: Versioning
description: retikz follows Semantic Versioning with pre-release channels. 0.x is loose, 1.x onwards is strict.
---

retikz follows Semantic Versioning 2.0.0: versions read as `MAJOR.MINOR.PATCH`, optionally tagged `-alpha.x` / `-beta.x` / `-rc.x`.

The current repository version is `0.2.0-rc.1`. That means v0.2 is in rc validation: public APIs are frozen, and only compatible bug fixes, docs completion, installation acceptance, and release-note cleanup should land. The previous stable line is `0.1.0`.

## Segment meanings

| Segment | Bumped when | Upgrade guidance |
| --- | --- | --- |
| MAJOR | Backward-incompatible API change | Read the migration guide |
| MINOR | Backward-compatible new capability | Upgrade freely |
| PATCH | Backward-compatible bug fix | Upgrade freely |

`@retikz/core` and `@retikz/react` normally share the same version and **ship together** — unless release notes say otherwise, users should not need to upgrade one without the other.

## 0.x vs 1.x

| Phase | API stability | Breaking change frequency |
| --- | --- | --- |
| `0.x` | Unstable, alpha / beta polish | May land in a MINOR |
| `1.x` onwards | Strict SemVer | Only in MAJOR |

Breaking changes before `1.0.0` **do not force a MAJOR bump**, but the changelog and release notes will call them out explicitly.

## Pre-release channels

| Channel | Format | Main content | Main driver | Suitable for |
| --- | --- | --- | --- | --- |
| alpha | `0.2.0-alpha.x` | New features / new IR fields / API reshaping (schema in flux, IR unstable) | **Human-designed + AI-implemented** | Exploration / feedback |
| beta | `0.2.0-beta.x` | Performance / readability / refactoring / test hardening / docs completion; final breaking renames are still allowed | **Human-led optimization and refactoring** | Integration testing |
| rc | `0.2.0-rc.x` | Public API frozen; compatible bug fixes, docs, installation acceptance, and release notes only | Human-led | Pre-production rehearsal |
| stable | `0.2.0` | Under SemVer; normal PATCH / MINOR / MAJOR cadence | — | Production |

Channels move forward only: alpha → beta → rc → stable. There's no rollback.

retikz uses a **"human-designed + AI-implemented"** split during alpha: humans produce ADRs (schema fields, file scopes, test quadrants, option decisions), AI relays through spec / implementation / tests / docs, and humans reconcile and review at the end — keeping architectural decisions under human control while scaling out implementation work. Full workflow details: [AI-assisted development](/about/developer/ai-assisted-development).

beta inverts this — schema is frozen, the API stops growing, and AI's freedom narrows; this phase emphasizes human review of alpha-era implementation, performance tuning, refactoring, test hardening, and documentation completion, laying the foundation for rc / stable.

rc narrows the work again: public APIs no longer break, and npm uses the `next` dist-tag. Installation acceptance uses `pnpm add @retikz/react@next react react-dom`; any issue that requires a breaking compatibility fix should stop the rc and return to a human version decision.

## peerDependencies

| Package | peer | Version |
| --- | --- | --- |
| `@retikz/react` | `react` / `react-dom` | `>=18` |
| `@retikz/core` | (none) | — |

Bumping React's major version usually ships as a MINOR; we only bump MAJOR when the new React version is incompatible with our old API.

## Deprecation flow

| Step | When | Marker |
| --- | --- | --- |
| Mark deprecated | Immediately | `@deprecated` type, `console.warn` at runtime |
| Grace period | At least 2 MINORs | Changelog points to the replacement |
| Removal | Next MAJOR | Listed under Removed |

The deprecation flow is not enforced during `0.x`; it kicks in once `1.0` ships.
