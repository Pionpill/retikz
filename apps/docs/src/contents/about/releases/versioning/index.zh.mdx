---
title: 版本策略
description: retikz 用语义化版本（SemVer），三段加预发布通道。0.x 期间宽松、1.x 起严格。
---

retikz 遵循 Semantic Versioning 2.0.0：版本号写作 `MAJOR.MINOR.PATCH`，必要时附预发布标签 `-alpha.x` / `-beta.x` / `-rc.x`。

当前仓库版本是 `0.2.0-rc.1`。这表示 v0.2 已进入 rc 验收窗口：公开 API 冻结，只接受兼容 bug fix、文档补齐、安装验收与发布说明收敛。上一条稳定线是 `0.1.0`。

## 三段含义

| 段 | 何时 +1 | 升级建议 |
| --- | --- | --- |
| MAJOR | 不兼容的 API 变更 | 阅读迁移指南 |
| MINOR | 向下兼容的新能力 | 直接升 |
| PATCH | 向下兼容的 bug fix | 直接升 |

`@retikz/core` 与 `@retikz/react` 默认共享同一版号、**同步发布**——除非发布说明特别写明，否则用户不需要单独升级其中一个包。

## 0.x 与 1.x 的差别

| 阶段 | API 稳定性 | 破坏性变更频率 |
| --- | --- | --- |
| `0.x` | 不稳，alpha / beta 期间持续打磨 | 可能在 MINOR 引入 |
| `1.x` 起 | 严格 SemVer | 仅在 MAJOR 引入 |

进入 `1.0.0` 之前的破坏性变更**不强行算 MAJOR + 1**，但会在 changelog 与发布说明里显式标注。

## 预发布通道

| 通道 | 形如 | 主要内容 | 主要执行 | 适合 |
| --- | --- | --- | --- | --- |
| alpha | `0.2.0-alpha.x` | 新功能 / 新 IR 字段 / API 形态重塑（schema 仍在重塑、IR 不稳） | **人工设计 + AI 实现** | 探索 / 反馈 |
| beta | `0.2.0-beta.x` | 性能 / 可读性 / 重构 / 测试加固 / 文档完善；允许最后一轮破坏性命名调整 | **人工主导优化与重构** | 集成验证 |
| rc | `0.2.0-rc.x` | 公开 API 冻结；只做兼容 bug 修复、文档、安装验收与发布说明 | 人工主导 | 生产前演练 |
| 稳定 | `0.2.0` | 进入 SemVer 守护，按 PATCH / MINOR / MAJOR 节奏迭代 | — | 生产 |

通道升级是单向的：alpha → beta → rc → stable，不会回退。

retikz 在 alpha 阶段采用 **"人工设计 + AI 实现"** 分工：人工写 ADR（schema 字段 / 文件 scope / 测试象限 / 选项决策），AI 接力写 spec / 实现 / 测试 / 文档，最后人工对账审稿——架构决策由人控、实现劳动可放大。具体流程见 [AI 辅助开发](/about/developer/ai-assisted-development)。

beta 阶段反过来——schema 已冻、API 不再扩张，AI 自由发挥的空间收窄；这一阶段着重对 alpha 期的实现做人工 review、性能调优、重构、补加固测试、补完整文档，为 rc / 正式版打底。

rc 阶段再收窄一层——公开 API 不再破坏，npm dist-tag 使用 `next`。安装验证以 `pnpm add @retikz/react@next react react-dom` 为准；如果发现必须破坏兼容才能修的问题，应停止 rc，重新回到人工版本决策。

## peerDependencies

| 包 | peer | 版本 |
| --- | --- | --- |
| `@retikz/react` | `react` / `react-dom` | `>=18` |
| `@retikz/core` | （无） | — |

升 React 主版本通常发 MINOR；只有当新版本不兼容老 API 时才升 MAJOR。

## 弃用流程

| 步骤 | 何时 | 标志 |
| --- | --- | --- |
| 标记弃用 | 即时 | 类型上 `@deprecated`、运行时 `console.warn` |
| 保留期 | 至少 2 个 MINOR | changelog 提示替代项 |
| 移除 | 下一个 MAJOR | changelog 列入 Removed |

`0.x` 期间弃用流程不强制；进入 `1.0` 起严格执行。
